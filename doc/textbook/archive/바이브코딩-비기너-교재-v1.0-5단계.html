VIBE CODING FOR BEGINNERS · 아이디어에서 배포까지

코드를 몰라도,
말로 만드는 나의 첫 프로그램

바이브 코딩(Vibe Coding)은 AI에게 자연어로 "이런 걸 만들어줘"라고 지시하면 코드를 대신 작성해 주는 개발 방식입니다. 이 교재는 비개발자·입문자가 ① 환경 구축 → ② 개발 → ③ Git 관리 → ④ 웹 배포 → ⑤ 중급 도약까지, 실제 하나의 프로그램을 완성하는 전체 여정을 5단계로 안내합니다.

# 바이브 코딩은 이렇게 시작됩니다
~/my-first-app $ claude
> "할 일을 기록하는 웹앱을 만들어줘. 완료하면 취소선이 그어지게."
⏺ 3개 파일 생성 · index.html · style.css · app.js

1단계

바이브 코딩을 위한 컴퓨터 환경 구축

학습 목표 — 맥(Mac)과 윈도우 PC에 바이브 코딩 필수 프로그램을 설치하고, 나에게 맞는 LLM·도구를 선택하고, CLI 방식과 웹 LLM 방식의 차이를 이해한다.

바이브 코딩 환경은 크게 4개의 층으로 이루어집니다. 아래 구성도처럼 "내가 지시 → AI 도구가 코드 작성 → 런타임이 실행 → Git이 기록"하는 구조입니다.

👨‍💻 전문가 의견 — 도구는 어떻게 고르나 실사용 비교 리뷰들의 공통 결론은 이렇습니다. "코드를 직접 보면서 편집하는 게 좋으면 Cursor, '이 작업 해줘'라고 시키고 결과만 받고 싶으면 Claude Code가 맞다. 둘 다 써본 사람들은 보통 Cursor에서 코드를 보면서 Claude Code로 복잡한 작업을 처리하는 식으로 병행한다"는 것. 또 Cursor가 코드의 세부 사항을 이해하며 AI의 도움을 받는 '책임감 있는 AI 지원 개발'이라면, Claude Code는 AI에게 작업을 완전히 위임하는 방식에 가깝다고 평가됩니다. 비개발자라면 에디터 설치·모델 선택·터미널 실행이라는 진입 장벽 때문에, 프리뷰 기능이 있는 LLM이나 Lovable처럼 'PRD를 주면 동작하는 프로토타입이 나오는' 서비스가 더 좋은 출발점이라는 조언도 있습니다. 한편 2026년 도구 순위를 매긴 리뷰는 Claude Code(앤스로픽)·Cursor·GitHub Copilot·Antigravity(구글)·Kiro(아마존)를 TOP5로 꼽으며, "AI 시대를 맞아 처음 코딩을 시작한 사용자 상당수는 Cursor와 Antigravity에서 편안함을 느낄 수 있다"고 평가했고, Antigravity는 빠르게 MVP를 만들고 싶은 경우 UI 중심으로 자동 개발을 진행하기에 적합하다는 분석도 있습니다. 이 교재의 권장 경로: 웹 빌더 또는 Antigravity로 감 잡기 → Cursor 또는 Claude Code로 본격 입문.

LLM(모델) 자체는 어떻게 다른가?

Claude (Anthropic)

긴 문맥 이해와 심층 추론에 강점. 복잡한 코드베이스 분석, 설계·아키텍처 단계에서 높은 평가. Claude Code·Cursor에서 사용.

GPT 계열 (OpenAI)

범용성과 생태계가 넓음. Codex는 클라우드에서 백그라운드 작업을 돌리는 자율 에이전트 방식이 특징.

Gemini (Google)

무료 할당량이 후하고 멀티모달(이미지·PDF 이해)에 강함. Gemini CLI·Antigravity로 사용.

1-3. CLI로 할까, 웹 LLM으로 직접 할까?

많은 입문자가 헷갈리는 지점입니다. 정답은 "둘 다 쓰되, 역할을 나눈다"입니다. 실전 워크플로우 가이드들은 "생각은 웹(Claude Web 등 채팅형 LLM), 실행은 코드(Claude Code 등 CLI)"라고 요약합니다. 웹 LLM은 아이디어 확장·전략 논의·문서 작성 같은 기획 단계에, CLI는 실제 코드 작성·파일 생성·디버깅·Git 커밋 같은 개발 단계에 사용하는 것입니다.

2단계

바이브 코딩의 기본 과정 — PRD에서 로컬 테스트까지

학습 목표 — 별도의 웹 LLM에서 PRD(제품 요구사항 문서)를 만들고, AI 코딩 도구로 구현하고, 내 컴퓨터에서 테스트하는 표준 사이클을 익힌다. 일반 프로그램과 웹 프로그램 두 가지 실전 사례로 연습하고, 외부 API(유튜브·공공데이터)와 LLM API를 연결하는 실전 예제까지 완성한다.

2-1. 별도의 LLM에서 PRD 만들기

PRD(Product Requirements Document, 제품 요구사항 문서)는 "어떤 제품을 만들 것인가"를 정의하는 문서로, 구체적인 구현 방법이 아니라 제품이 해야 할 것과 하지 말아야 할 것을 정리합니다. 왜 코딩보다 문서가 먼저일까요? 개발 중 "이 기능도 추가하면 어때?"라는 유혹이 올 때 "PRD에 없으니까 다음 버전에 하자"라고 판단하는 기준이 되기 때문입니다. AI에게도 명확한 지도를 주는 셈이라 결과물의 품질이 크게 올라갑니다.

👨‍💻 전문가 의견 — PRD 작성의 실전 팁 ① 바이브 코딩을 할 때는 '기획서'보다 PRD라는 단어를 쓰면 AI가 더 잘 알아듣고, LLM이 영어 데이터를 압도적으로 많이 학습했기 때문에 PRD를 영어로 만들면 유리하다는 조언이 있습니다(한국어로 대화하고 문서만 영어로 뽑아달라고 하면 됩니다). ② 구조화된 워크플로우를 정리한 개발자는 PRD를 만든 뒤 LLM에게 이를 바탕으로 단계별 실행 계획을 세우게 하고, 기능을 작고 완결된 단위(수직 분할)로 DB→서버→UI 전체 계층을 한 번에 구현하되 점진적으로 복잡도를 높이라고 권합니다. ③ 한 번에 하나의 기능을 맡아 끝까지 완성하는 것이 초보자의 혼란을 가장 줄여 줍니다.

웹 LLM(claude.ai 등)에 넣는 PRD 생성 프롬프트 예시

# claude.ai / ChatGPT / Gemini 채팅창에 붙여넣기
당신은 시니어 프로덕트 매니저입니다. 아래 아이디어로 초보 개발자와
AI 코딩 도구가 함께 쓸 PRD를 마크다운으로 작성해 주세요.

[아이디어] 매일 마신 물의 양을 기록하고 목표 달성률을 보여주는 웹앱

PRD에 포함할 것:
1. 목적과 해결하려는 문제   2. 대상 사용자
3. 핵심 기능 (MVP 3개 이내로 제한)   4. 화면 구성 설명
5. 기술 스택 (초보자 기준: HTML/CSS/JS 단일 파일 권장)
6. 이번 버전에서 하지 않을 것 (Out of Scope)

결과로 받은 문서를 PRD.md라는 파일로 프로젝트 폴더에 저장합니다. 이것이 2단계 전체의 "설계도"가 됩니다.

2-2. AI 도구에서 코딩하기 — 두 가지 실전 사례

사례 A · 일반 프로그램 (Python 데스크톱 도구)

웹이 아닌 "내 컴퓨터에서 도는 프로그램"의 예로, 폴더 안의 파일 이름을 규칙에 따라 일괄 변경하는 도구를 만들어 봅니다. 방송 촬영본 정리처럼 반복 작업 자동화에 딱 맞는 유형입니다.

  1. 프로젝트 폴더 생성: mkdir file-renamer && cd file-renamer 후 PRD.md 저장
  2. Claude Code 실행(claude) 후 지시: "PRD.md를 읽고 Python으로 구현해줘. 실행 전 어떤 파일이 어떻게 바뀔지 미리보기를 먼저 보여주는 안전장치를 꼭 넣어줘."
  3. AI가 renamer.py를 생성하면 코드를 훑어보고(이해 안 되는 줄은 "이 줄은 뭐 하는 거야?"라고 질문) 실행: python3 renamer.py
  4. 테스트용 폴더에서 먼저 실행해 보고, 수정 사항을 자연어로 요청: "날짜를 파일 맨 앞에 YYYYMMDD_ 형식으로 붙여줘"

사례 B · 웹 프로그램 (할 일 관리 웹앱)

웹 프로그램은 브라우저에서 열리는 프로그램입니다. 결과가 즉시 눈에 보여 입문용으로 가장 좋습니다.

  1. 폴더 생성(todo-app) 후 PRD.md 저장 — 핵심 기능: 할 일 추가 / 완료 체크 / 삭제, 새로고침해도 유지(localStorage)
  2. AI 도구에 지시: "PRD.md대로 index.html, style.css, app.js 3개 파일로 만들어줘. 프레임워크 없이 순수 HTML/CSS/JS로."
  3. 생성된 index.html을 브라우저로 열어 즉시 확인
  4. 디자인·기능 수정을 대화로 반복: "완료된 항목은 회색 취소선으로", "다크 모드 버튼 추가해줘"
💡 좋은 지시(프롬프트)의 공식 "이메일과 비밀번호 기반 로그인 기능을 만들어줘. JWT 토큰 방식을 사용하고, 비밀번호는 bcrypt로 해싱해야 해. 실패 시 'Invalid credentials' 에러를 반환하고, 성공 시 user 객체와 token을 함께 반환해"처럼 구체적이고 명확할수록 AI는 정확한 결과를 제공합니다. 또한 AI 모델은 학습 데이터에 많이 노출된 기술을 훨씬 잘 다루므로, 실험적인 라이브러리 대신 React 같은 인기 기술 스택을 선택하면 더 정확하고 안정적인 코드를 얻을 수 있습니다.

2-3. 로컬에서 테스트하기

"로컬(local)"은 배포 전 내 컴퓨터 안을 뜻합니다. 유형별 테스트 방법은 다음과 같습니다.

프로그램 유형실행 방법확인 위치
Python 일반 프로그램python3 파일명.py터미널 출력
정적 웹 (HTML/CSS/JS)index.html 더블클릭, 또는 VS Code 'Live Server' 확장브라우저
Node 기반 웹 (React/Next.js 등)npm install 후 npm run dev (→ 2-4절에서 상세 설명)브라우저에서 http://localhost:3000 접속

localhost:3000은 "내 컴퓨터의 3000번 문"이라는 뜻의 주소로, 나만 볼 수 있는 개발용 미리보기입니다. 전 세계에 공개하는 방법은 4단계(배포)에서 다룹니다.

  • PRD의 핵심 기능이 모두 동작하는가? (기능 하나씩 직접 눌러보기)
  • 이상한 값을 넣어도 안 깨지는가? (빈 값, 아주 긴 글자, 한글/이모지)
  • 에러가 나면 → 에러 메시지를 통째로 복사해 AI에게 붙여넣고 "고쳐줘"
  • 브라우저 개발자 도구(F12) Console 탭에 빨간 에러가 없는가?
👨‍💻 전문가 의견 — 테스트도 AI에게 Claude Code 가이드에 따르면 "관련 파일을 읽고, 계획을 세우고, 코드를 짜고, 커밋"하는 자동화 워크플로우가 가능하며, 기능 단위 테스트뿐 아니라 "이 기능 구현하다 저 기능이 깨지진 않았나"를 보는 통합 테스트, "로그인부터 결제까지 모든 단계가 동작하나"를 보는 E2E 테스트도 AI가 만들어 실행해 줄 수 있습니다. 이는 스펙대로 구현됐는지를 바이브 코더와 AI 양측이 확인하는 전략입니다.

2-4. 실전 예제 — 외부 데이터(API) 가져와서 LLM으로 분석하기

지금까지는 데이터가 내 앱 안에만 있었습니다. 이번에는 API를 통해 바깥세상의 데이터를 가져옵니다. 실습 주제는 유튜브 채널을 등록하면 그 채널의 최신 영상 리스트를 받아오고, LLM(Claude)에게 콘텐츠 전략을 분석시키는 프로그램입니다. 이 과정에서 초보자가 가장 많이 실수하는 API 키 발급·보관·배포 보안을 함께 배웁니다.

STEP 0 · API 키란? — 그리고 공공 데이터의 세계

API 키는 "이 데이터를 요청한 사람이 등록된 사용자 본인"임을 증명하는 출입증 문자열입니다. 유튜브뿐 아니라 대한민국 공공데이터포털(data.go.kr)도 같은 방식입니다 — 회원가입 후 원하는 데이터(예: 기상청 날씨, 버스 도착 정보, 한국거래소 시세)의 "활용 신청"을 하면 인증키(serviceKey)가 발급되고, 그 키를 주소에 붙여 요청하면 JSON 형식의 데이터를 받습니다. 발급 방식만 다를 뿐 "키를 받고 → 요청에 실어 보내고 → JSON을 받는" 구조는 전 세계 모든 API가 동일하므로, 아래 유튜브 예제를 한 번 익히면 공공 데이터에도 그대로 적용할 수 있습니다.

STEP 1 · 유튜브 API 키 발급 (키①)

  1. Google Cloud Console(console.cloud.google.com)에 구글 계정으로 로그인 → 상단에서 새 프로젝트 생성 (이름 예: youtube-analyzer)
  2. 좌측 메뉴 API 및 서비스 → 라이브러리 → "YouTube Data API v3" 검색 → 사용(Enable) 클릭
  3. API 및 서비스 → 사용자 인증 정보(Credentials) → + 사용자 인증 정보 만들기 → API 키
  4. 발급된 키(AIza...로 시작)를 복사. 키 제한 설정에서 "YouTube Data API v3만 허용"으로 제한해 두면 유출 시 피해를 줄일 수 있음

※ YouTube Data API는 무료이지만 하루 10,000 유닛의 할당량(quota)이 있습니다. 영상 목록 조회는 요청당 1~2유닛이라 학습용으로는 충분합니다.

STEP 2 · LLM(Claude) API 키 발급 (키②)

  1. Anthropic Console(console.anthropic.com) 접속 → 계정 생성 후 로그인
  2. 결제 수단 등록 후 소액 크레딧 충전 (API는 채팅 구독과 별도로, 쓴 만큼 과금되는 종량제)
  3. 좌측 API Keys → + Create Key → 이름 입력(예: youtube-project) → 생성
  4. 키는 생성 직후 딱 한 번만 전체가 표시됩니다. 즉시 복사해 안전한 곳(.env 파일)에 보관하세요. 잃어버리면 새로 발급해야 합니다

※ OpenAI(platform.openai.com), Google AI Studio(aistudio.google.com — 무료 할당량 제공) 등 다른 LLM도 발급 절차는 거의 동일합니다: 콘솔 접속 → API Keys 메뉴 → 생성 → 1회 표시 → 복사.

STEP 3 · 프로젝트 만들기 — 그리고 npm이 하는 일

# 프로젝트 폴더 생성 후 이동
mkdir youtube-analyzer && cd youtube-analyzer

# ① 이 폴더를 'Node.js 프로젝트'로 선언 → package.json 파일이 생김
npm init -y

# ② 필요한 부품(라이브러리) 설치 → node_modules 폴더에 다운로드됨
npm install dotenv
🔍 자세히 보기 — npm install과 npm run dev는 대체 무슨 뜻일까?

npm(Node Package Manager)은 Node.js를 설치하면 함께 딸려오는 "프로그램 부품(라이브러리) 자동 장보기 도구"입니다. 전 세계 개발자들이 만들어 둔 200만 개 이상의 부품 창고(npm 레지스트리)에서 필요한 것을 내려받습니다.

① package.json — 프로젝트의 "설명서 겸 부품 목록"입니다. "이 프로젝트는 어떤 부품(dependencies)이 필요하고, 어떤 실행 버튼(scripts)이 있는지"가 적혀 있습니다. GitHub에서 남의 프로젝트를 받았을 때 가장 먼저 열어보는 파일입니다.

② npm install — package.json의 부품 목록을 읽고, 필요한 라이브러리를 전부 node_modules 폴더에 다운로드합니다. 이케아 가구의 부품 봉투를 채우는 과정과 같습니다. GitHub에서 clone한 프로젝트가 실행이 안 될 때 90%는 이걸 안 해서입니다 — node_modules는 용량이 커서 Git에 올리지 않기 때문에(.gitignore 처리), 새 컴퓨터에서는 항상 npm install부터 해야 합니다.

③ npm run dev — package.json의 scripts 항목에 적힌 "dev"라는 이름의 명령을 실행하라는 뜻입니다. 즉 "미리 등록해 둔 실행 버튼을 누르는 것"입니다. Next.js 프로젝트라면 dev = "개발용 서버 켜기"로 등록돼 있어 localhost:3000에서 미리보기가 열리고, 코드를 고치면 화면이 자동 갱신됩니다. 끄려면 터미널에서 Ctrl + C. 이름은 약속일 뿐이라 npm run 아무이름도 만들 수 있습니다 — 아래 예제에서는 "dev": "node analyze.js"로 등록해 우리 분석 프로그램을 실행 버튼화합니다.

이제 package.json을 열어 실행 버튼을 등록합니다 (AI에게 "package.json에 dev 스크립트 추가해줘"라고 시켜도 됩니다):

// package.json
{
  "name": "youtube-analyzer",
  "type": "module",                  // 최신 import 문법 사용 선언
  "scripts": {
    "dev": "node analyze.js"         // npm run dev = node analyze.js 실행
  },
  "dependencies": {
    "dotenv": "^16.4.5"              // npm install로 설치된 부품 목록
  }
}

STEP 4 · 키를 안전하게 보관하기 — .env와 .gitignore

발급받은 두 키를 코드가 아닌 .env 파일에 넣습니다. .env는 "이 컴퓨터에만 존재하는 비밀 금고"입니다.

# .env  (프로젝트 폴더에 새 파일로 생성 — 따옴표 없이)
YOUTUBE_API_KEY=AIzaSyD...여기에_유튜브_키
ANTHROPIC_API_KEY=sk-ant-api03-...여기에_클로드_키

# .gitignore  (Git이 절대 올리면 안 되는 것들의 목록)
.env
node_modules/
⚠️ 키 보관·배포의 4대 철칙 — 실제 사고가 가장 많은 지점

① .env는 반드시 .gitignore에. 이걸 빠뜨리고 push하면 키가 GitHub에 영구 기록됩니다. 봇들이 공개 저장소를 24시간 스캔하며 키를 수집하므로, 노출된 키는 수 분 안에 도용될 수 있습니다. 실수로 올렸다면 지우는 걸로 끝이 아니라 즉시 해당 콘솔에서 키를 폐기(revoke)하고 재발급해야 합니다 — Git 히스토리에 남아 있기 때문입니다.

② 키는 절대 브라우저(프론트엔드) 코드에 넣지 않기. HTML/JS에 키를 쓰면 방문자 누구나 "소스 보기"로 훔칠 수 있습니다. 키가 필요한 요청은 반드시 서버 쪽(Node 스크립트, 서버리스 함수)에서 실행합니다. 이 예제가 브라우저가 아닌 Node.js로 도는 이유입니다.

③ 배포할 때는 플랫폼의 환경 변수 메뉴에. Vercel이라면 프로젝트 Settings → Environment Variables에 같은 이름(YOUTUBE_API_KEY 등)으로 등록하면, 서버가 .env 대신 그 값을 읽습니다. .env 파일 자체를 업로드하는 것이 아닙니다.

④ 키에 한도와 제한을 걸기. Google 콘솔에서는 API 키를 특정 API만 쓰도록 제한하고, Anthropic/OpenAI 콘솔에서는 월 사용액 한도를 설정해 두세요. 유출돼도 피해가 제한됩니다.

STEP 5 · 코드 작성 — 채널 등록 → 영상 리스트 수집

AI 도구에게 아래 PRD식 지시를 내려도 되고, 코드를 그대로 만들어 원리를 살펴봐도 됩니다. 채널은 유튜브 주소의 @핸들(예: youtube.com/@KBSnews)로 등록합니다.

// youtube.js — 유튜브 채널의 최신 영상 목록 가져오기
import 'dotenv/config';                        // .env의 비밀 값을 불러오는 부품
const KEY = process.env.YOUTUBE_API_KEY;       // 금고에서 키① 꺼내기
const HANDLE = '@KBSnews';                     // ★ 분석할 채널을 여기에 등록

// ① 채널 핸들 → 채널의 '업로드 재생목록 ID' 알아내기
const chRes = await fetch(
  `https://www.googleapis.com/youtube/v3/channels?part=contentDetails,snippet&forHandle=${HANDLE}&key=${KEY}`
);
const chData = await chRes.json();             // 응답을 JSON으로 해석
const uploadsId = chData.items[0].contentDetails.relatedPlaylists.uploads;
console.log('채널명:', chData.items[0].snippet.title);

// ② 그 재생목록에서 최신 영상 10개 꺼내기
const listRes = await fetch(
  `https://www.googleapis.com/youtube/v3/playlistItems?part=snippet&playlistId=${uploadsId}&maxResults=10&key=${KEY}`
);
const listData = await listRes.json();

// ③ 필요한 정보(제목·게시일)만 골라 정리
export const videos = listData.items.map(v => ({
  title: v.snippet.title,
  publishedAt: v.snippet.publishedAt.slice(0, 10)
}));
console.table(videos);                          // 표 형태로 출력

읽는 법: fetch(주소)는 "이 주소로 데이터 주세요"라는 요청이고, 주소 끝에 &key=로 출입증을 첨부합니다. 돌아온 답장은 JSON이라는 문자 형식이라 .json()으로 풀어서 씁니다.

STEP 6 · LLM에게 분석 시키기 — 키②로 Claude API 호출

// analyze.js — 수집한 영상 목록을 Claude에게 보내 분석 받기
import 'dotenv/config';
import { videos } from './youtube.js';          // STEP 5의 결과를 가져옴

const res = await fetch('https://api.anthropic.com/v1/messages', {
  method: 'POST',                               // '보내는' 요청
  headers: {
    'content-type': 'application/json',
    'x-api-key': process.env.ANTHROPIC_API_KEY, // 금고에서 키② 꺼내 첨부
    'anthropic-version': '2023-06-01'
  },
  body: JSON.stringify({                        // LLM에게 보낼 편지 내용
    model: 'claude-sonnet-4-6',
    max_tokens: 1500,
    messages: [{
      role: 'user',
      content: `다음은 한 유튜브 채널의 최신 영상 10개 목록입니다.
${JSON.stringify(videos, null, 2)}

이 채널에 대해 분석해 주세요:
1. 주로 다루는 주제 (상위 3가지)
2. 제목 작성 패턴의 특징
3. 업로드 빈도와 콘텐츠 전략에 대한 인사이트`
    }]
  })
});

const data = await res.json();
console.log('\n===== 🤖 Claude의 분석 =====\n');
console.log(data.content[0].text);              // 분석 결과 출력

  • 에러 403 quotaExceeded → 유튜브 일일 할당량 초과. 내일 다시 또는 요청 수 줄이기
  • 에러 401 authentication_error → 키가 틀렸거나 .env 파일 이름·위치 확인 (프로젝트 최상위 폴더)
  • 에러 Cannot find module 'dotenv' → npm install을 안 한 것. 부품부터 설치
  • 응용 과제: 결과를 화면에 보여주는 웹앱으로 확장한다면? → 키 보호를 위해 API 호출은 서버리스 함수(Vercel의 /api 폴더)로 옮기고, 브라우저는 그 함수만 호출하게 설계 (4단계 배포와 연결되는 지점)
  • 응용 과제: HANDLE을 공공데이터포털 주소로 바꾸면 그대로 "우리 동네 버스 도착 분석기"가 됩니다
3단계

Git에 올리기 — 버전 관리와 협업의 시작

학습 목표 — Git의 핵심 개념을 이해하고, GitHub에 코드를 올리고, 집·사무실의 서로 다른 PC에서 같은 저장소로 이어서 작업하며, 다른 사람과 프로젝트를 공유한다.

3-1. Git 전반적인 이해 — 게임의 '세이브 포인트'

Git은 코드의 변경 역사를 기록하는 버전 관리 시스템입니다. 게임의 세이브 포인트처럼, 잘 되던 시점을 저장해 두면 AI가 코드를 망가뜨려도 언제든 그 시점으로 되돌아갈 수 있습니다. GitHub는 그 기록을 인터넷에 보관·공유하는 서비스(원격 저장소)입니다. 전문가들은 바이브 코딩에서 Git이 선택이 아닌 필수라고 강조합니다 — "Git을 마스터하지 않고 바이브 코딩을 시도하는 것은 안전벨트 없이 고속도로를 달리는 것과 같다"는 표현이 대표적입니다.

딱 6개만 외우면 되는 Git 명령어

# ── 프로젝트 시작 시 딱 1번 ──
git init                                  # 이 폴더의 버전 관리 시작
git remote add origin [GitHub 저장소 주소]  # 클라우드 보관소 연결

# ── 작업할 때마다 반복하는 3종 세트 ──
git add .                                 # 바뀐 파일 전부 담기
git commit -m "할 일 추가 기능 완성"          # 메모와 함께 세이브
git push                                  # GitHub에 업로드

# ── 다른 PC에서 이어서 할 때 ──
git pull                                  # 최신 버전 내려받기
💡 GitHub 첫 저장소 만들기 (5분) ① github.com 가입 → 우측 상단 + → New repository ② 이름 입력(예: todo-app), Public/Private 선택 → Create ③ 화면에 나오는 "…or push an existing repository" 명령 3줄을 내 터미널에 복사 실행. 끝입니다. 참고로 GitHub는 코드 저장소 자체는 무료로 꽤 넉넉하게 쓸 수 있어서, 개인 개발·포트폴리오·사이드 프로젝트 정도라면 Free 플랜만으로 대부분 충분합니다. 그리고 바이브 코딩이라면 이 모든 과정조차 AI에게 "지금까지 작업 커밋하고 GitHub에 push해줘"라고 시킬 수 있습니다.

3-2. 집과 사무실, 다른 PC에서 같은 저장소로 작업하기

GitHub가 "원본"이 되고 각 PC는 그 사본을 갖는 구조입니다. 규칙은 단 하나 — 앉으면 pull, 일어나면 push.

  1. 새 PC 최초 1회: git clone https://github.com/내계정/todo-app.git — 저장소 전체를 복제
  2. 작업 시작 전(매번): git pull — 다른 PC에서 올린 최신 내용 받기
  3. 작업 종료 시(매번): git add . && git commit -m "메모" && git push
  4. push를 깜빡하고 퇴근했다면? 다음 날 그 PC에서 push 먼저 → 다른 PC에서 pull
⚠️ 충돌(Conflict)이 났어요! push를 잊고 양쪽 PC에서 같은 파일을 고치면 Git이 "어느 쪽이 맞아?"라고 묻는 충돌이 발생합니다. 당황하지 말고 ① VS Code가 보여주는 충돌 표시에서 남길 쪽을 클릭해 선택하거나 ② AI 도구에 "지금 git conflict 났어. 해결해줘"라고 요청하세요. 예방책은 "앉으면 pull, 일어나면 push" 습관 하나면 충분합니다.

3-3. 같은 저장소로 다른 사람들과 프로젝트 공유하기

혼자 쓰던 저장소에 동료를 초대하면 곧바로 협업이 됩니다. 두 가지 수준이 있습니다.

수준 1 · Collaborator 초대

저장소 Settings → Collaborators → Add people로 동료의 GitHub 계정을 초대. 초대받은 사람도 clone/pull/push가 가능해집니다. 소규모 팀·스터디에 적합.

수준 2 · 브랜치 + Pull Request

각자 브랜치(git checkout -b feature/login)에서 작업 후 PR(Pull Request)로 "합쳐 주세요"를 요청 → 동료 검토 후 main에 병합. 실무 표준 방식.

바이브 코딩에서도 브랜치를 활용한 Git 워크플로우는 필수로 꼽힙니다 — AI가 만든 실험적 코드를 브랜치에 격리해 두면, 실패해도 main(안정 버전)은 무사하기 때문입니다. 또한 README.md 파일을 잘 써 두면(프로젝트 소개·설치법·실행법) 저장소 첫 화면에 자동 표시되어 공유 문서 역할을 합니다. 이 역시 AI에게 "이 프로젝트의 README를 작성해줘"라고 맡기면 됩니다.

  • 저장소를 Private으로 만들면 초대한 사람만 볼 수 있다 (회사 프로젝트 권장)
  • 비밀 정보(.env, API 키)는 .gitignore에 등록해 절대 push하지 않는다
  • 커밋 메시지는 "수정" 대신 "무엇을 왜 바꿨는지" 한 줄로 쓴다
  • 큰 기능은 브랜치에서, main은 항상 동작하는 상태로 유지한다
4단계

로컬을 넘어 세상으로 — 웹 배포 (Deployment)

학습 목표 — Vercel·GitHub Pages·Netlify·Render 등 배포 플랫폼의 특징과 가격을 비교해 내 프로젝트에 맞는 곳을 고르고, Git 연동으로 배포·관리·업데이트하는 방법을 익힌다.

배포(Deploy)란 내 컴퓨터(localhost)에만 있던 웹앱을 인터넷 서버에 올려 전 세계 누구나 주소로 접속할 수 있게 만드는 일입니다. 과거에는 서버를 직접 사서 관리해야 했지만, 지금은 GitHub에 push만 하면 자동으로 배포되는 플랫폼들이 있어 초보자도 5분이면 사이트를 공개할 수 있습니다.

4-1. 배포 플랫폼 5종 — 특징과 가격 비교

Vercel — Next.js의 본가

Next.js 개발사가 직접 만든 플랫폼으로 React·프론트엔드 프레임워크에 최적화. push마다 자동 배포 + PR마다 미리보기 URL 제공. 서울에 CDN이 있어 한국 접속 시 TTFB 30~40ms 수준의 최상급 속도를 보여 국내 사용자에게 특히 유리합니다.

Hobby 무료 · Pro $20/월

GitHub Pages — 완전 무료 정적 호스팅

깃허브 저장소의 정적 파일(HTML/CSS/JS)을 무료로 호스팅. 포트폴리오·문서·블로그에 이상적. 공개 저장소라면 대역폭 제한 없이 완전 무료이지만, 서버 사이드 기능은 지원하지 않습니다.

Public 저장소 무료 (Private Pages는 Pro 필요)

Netlify — 정적 사이트 만능형

직관적 UI와 자동 Git 연동, 폼(Form) 처리 등 정적 사이트용 편의 기능 내장. 무료 플랜에서도 상업적 사용이 허용된다는 점이 Vercel과의 가장 큰 실무 차이입니다. 무료 빌드 시간은 월 300분으로 제한됩니다.

Free 무료 · Pro 약 $19/월

Render — 백엔드까지 한 곳에서

정적 사이트뿐 아니라 Flask·Node.js 같은 백엔드 API, PostgreSQL 데이터베이스, 백그라운드 작업, 크론 잡까지 하나의 대시보드에서 배포 가능한 풀스택 클라우드. 무료 웹 서비스는 일정 시간 미사용 시 잠들었다 깨어나는(콜드 스타트) 특성이 있습니다.

정적 사이트 무료 · 웹 서비스 무료 티어 / 유료 $7/월~

Cloudflare Pages — 무료 한도의 최강자

글로벌 CDN 기업 Cloudflare의 호스팅. 무료 플랜의 대역폭이 사실상 무제한급이라 Vercel 무료 100GB 한도에 가까워진 사용자들이 이주하는 대표 목적지로 꼽힙니다. Workers(서버리스)·D1(DB) 등 생태계 확장성도 좋습니다.

Free 무료 · Paid $5/월~

👨‍💻 전문가 의견 — 어떤 기준으로 고르나 플랫폼 비교 분석의 결론은 명확합니다. "Next.js로 만든다면 Vercel — React Server Components, 스트리밍, ISR에 대한 프레임워크 수준 지원은 다른 플랫폼이 따라오지 못한다. 다만 무료 티어의 비상업 제한을 유의하라. 올인원 배포 플랫폼과 Git 기반 CI/CD, 미리보기 배포, 내장 폼을 원하면 Netlify를 유지하라"는 것. 그리고 "단순 정적 사이트라면 GitHub Pages가 비용 면에서 이기기 어렵고, 풀스택 앱에는 Render가 최적"이라는 평가가 일반적입니다. 요약하면 정적 = GitHub Pages, React/Next = Vercel, 폼·상업용 정적 = Netlify, 백엔드+DB = Render입니다.

4-2. Git을 이용해 배포하기 (Vercel 실습)

  1. 가입: vercel.com 접속 → "Continue with GitHub"로 GitHub 계정 연동 가입
  2. Import: 대시보드에서 Add New → Project → 3단계에서 올린 내 저장소(todo-app) 선택
  3. 설정: 프레임워크는 자동 감지되므로 대부분 그대로 Deploy 클릭
  4. 완성: 1~2분 후 https://todo-app-내계정.vercel.app 주소 발급 — 이제 전 세계에서 접속 가능!
  5. 커스텀 도메인(선택): Settings → Domains에서 보유한 도메인 연결(무료 SSL 자동 적용)

배포 후 관리·업데이트 — 운영의 3대 루틴

루틴방법포인트
업데이트코드 수정 → git push → 끝push만 하면 자동 재배포. 별도 업로드 작업이 전혀 없음
미리보기 배포브랜치에 push하면 브랜치 전용 Preview URL 발급main(실서비스)에 합치기 전에 안전하게 확인
롤백(되돌리기)대시보드 Deployments 목록에서 과거 배포 선택 → "Promote to Production"업데이트가 잘못돼도 클릭 한 번에 이전 버전으로 복구
환경 변수Settings → Environment Variables에 API 키 등록비밀 키는 코드가 아닌 플랫폼 설정에 보관 (보안 필수)
모니터링대시보드의 Analytics·Logs 탭 확인방문자 수·에러·무료 한도 사용량을 주기적으로 점검

GitHub Pages로 배포하기 (정적 사이트라면 이쪽이 더 간단)

  1. GitHub 저장소 → Settings → Pages 메뉴 이동
  2. Source를 Deploy from a branch, 브랜치를 main / root로 선택 후 Save
  3. 1~2분 뒤 https://내계정.github.io/저장소명/ 주소로 공개 완료

※ Render 배포도 흐름은 동일합니다: render.com 가입 → New → Static Site(또는 Web Service) → GitHub 저장소 연결 → 자동 배포. 백엔드(Flask/Node) 프로젝트라면 시작 명령(예: gunicorn app:app)만 추가로 지정합니다.

5단계

중급자로 가는 길 — 기술 로드맵과 학습 자원

학습 목표 — 바이브 코딩을 넘어 "AI가 만든 코드를 이해하고 통제하는" 중급자가 되기 위해 배워야 할 웹개발 기술과 신뢰할 수 있는 학습 사이트를 파악한다.

바이브 코딩의 역설: AI가 코드를 다 짜 주는데 왜 공부해야 할까요? 전문가들은 "코딩(문법을 치는 행위)은 이미 LLM이 숙련자보다 빠르지만, 문제를 정의하고 최적의 해결책을 설계하며 유지보수 가능한 시스템을 만드는 '프로그래밍'은 여전히 사람의 몫이며, 개발자는 제품 관리자처럼 생각해야 한다"고 말합니다. 즉 중급자의 목표는 코드를 외우는 것이 아니라 AI의 결과물을 읽고, 평가하고, 방향을 제시하는 능력입니다.

5-1. 배워야 할 웹개발 기술 — 4단계 로드맵

이 순서는 실제 시장 데이터에 근거합니다. Stack Overflow의 2025년 전 세계 49,000명 이상 개발자 조사에서 JavaScript는 사용률 1위(66%)를 유지했고, TypeScript는 44%까지 올라섰으며, React는 프론트엔드 프레임워크 중 독보적인 점유율을 기록하고 있습니다. 특히 2026년 기준 대부분의 기업 프로젝트가 TypeScript로 작성되므로 Lv.2에서 반드시 익혀야 합니다.

레벨핵심 기술왜 배우나 (바이브 코딩 관점)졸업 프로젝트 예시
Lv.1HTML / CSS(Flexbox·Grid) / JavaScript 기본AI가 만든 화면 코드를 "읽을" 수 있게 됨. 이 셋을 건너뛰고 프레임워크부터 배우는 것은 기초 공사 없이 건물을 올리는 것과 같다는 것이 로드맵 전문가들의 공통 의견자기소개 페이지를 AI 없이 직접 → AI로 리팩토링 비교
Lv.2React, TypeScript, Tailwind CSS, npmCursor·v0·Lovable 등 AI 도구 대부분이 기본으로 생성하는 스택. TypeScript는 잘못된 타입을 전달하면 편집기에서 즉시 빨간 줄로 알려주어 런타임 버그를 사전에 차단2단계의 할 일 앱을 React + TS로 재구축
Lv.3Next.js, REST API, 데이터베이스(SQL·Supabase 등), 인증(로그인)"데이터가 저장되는 진짜 서비스"의 필수 요소. Git·배포(3~4단계)와 결합해 실사용 가능한 앱 완성회원가입·DB 저장이 되는 미니 서비스 배포
Lv.4테스트(Jest 등), 웹 보안 기초(XSS·키 관리), CI/CD(GitHub Actions), 성능·모니터링바이브 코딩 사고 사례(키 노출·DB 삭제)의 재발 방지책. AI 결과물을 "검증"하는 눈을 갖추는 단계내 서비스에 자동 테스트 + 배포 파이프라인 구축

5-2. 참고할 만한 웹사이트와 공부 방법

MDN Web Docs developer.mozilla.org

웹 개발의 사실상 표준 문서. HTML·CSS·JavaScript·Web API 등 웹 기술 전반을 기초부터 심화까지 상세히 다루며, 초보자를 위한 학습(Learn) 섹션이 한국어로도 제공됩니다. 모르는 문법은 항상 여기서 확인하는 습관을.

roadmap.sh

프론트엔드를 비롯한 개발 전 분야의 로드맵을 집대성한 사이트. 항목마다 추천 읽을거리를 풍부하게 제공하지만 방대하므로 필수 요소부터 차근차근 접근할 것이 권장됩니다.

web.dev (Google)

Google 개발자들이 운영하는 웹 기술 문서로, 학습 경로와 양질의 기술 아티클을 제공합니다. 성능·접근성 등 "잘 만든 웹"의 기준을 배우기 좋습니다.

freeCodeCamp / W3Schools

브라우저 안에서 바로 코드를 쳐 보며 배우는 무료 실습형 커리큘럼. W3Schools는 HTML·CSS·JavaScript·SQL·Python 등을 풍부한 예제와 함께 정리한 튜토리얼 사이트로 빠른 문법 확인에 유용합니다.

생활코딩 · 코딩애플 · 노마드코더 (한국어)

국내 입문자 대상 대표 무료·유료 강의 채널. 생활코딩의 HTML 강의는 웹을 전혀 몰라도 차근차근 따라가며 처음 코딩을 접하기 좋은 강의로 꼽히며, 코딩애플은 첫날부터 웹을 만들며 배우는 프로젝트 중심 커리큘럼을 표방합니다.

ko.javascript.info

'코어 자바스크립트'로 불리는 무료 한국어 교재. JavaScript의 동작 원리를 깊게 이해하고 싶을 때 정독할 만한 표준 자료입니다.

👨‍💻 전문가 의견 — 중급자로 가는 공부법 ① "완벽하게 배운 후 시작하겠다는 생각은 영원히 시작하지 않겠다는 말과 같다. 오늘 HTML 파일 하나를 만들어 보는 것이 어떤 강의를 고를지 고민하는 시간보다 훨씬 가치 있다" — 로드맵보다 실행이 먼저입니다. ② "실제로 코드를 치고, 에러를 마주하고, 해결하는 과정의 반복이 실력을 만든다"는 점에서, AI에게 코드를 받은 뒤 "이 코드를 한 줄씩 설명해줘", "왜 이 방식을 썼어? 다른 방법은?"이라고 되묻는 습관이 최고의 개인 과외가 됩니다. ③ 마인드셋 전환도 중요합니다 — "AI가 나를 위해 무엇을 해줄 수 있지?"가 아니라 "내가 AI를 위해 무엇을 제공할 수 있지?"를 물어야 하며, AI에게는 명확한 맥락과 구체적인 요구사항이 필요합니다.

이 교재를 마친 뒤 추천하는 다음 3개 프로젝트

  1. 나만의 포트폴리오 사이트 — Lv.1 기술 + GitHub Pages 배포. 이 교재의 1~4단계를 그대로 한 번 더 복습
  2. 업무 자동화 도구 — 반복하는 업무 하나(파일 정리, 보고서 생성 등)를 Python으로. PRD 작성 → CLI 도구 실습
  3. 데이터가 저장되는 미니 서비스 — Next.js + Supabase + Vercel. Lv.3로 넘어가는 관문 프로젝트
