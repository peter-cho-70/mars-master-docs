01바이브 코딩과 다섯 도구의 관계 — 전체 지도

WHY THESE FIVE?

바이브 코딩은 2025년 초 안드레이 카파시(Andrej Karpathy)가 제안한 개념으로, 코드를 한 줄씩 직접 작성하는 대신 자연어로 의도를 설명하고 AI가 구현하게 하는 개발 방식입니다. 처음에는 IDE 안의 코드 자동완성에서 출발했지만, 2026년 현재는 그 범위가 크게 확장되었습니다. AI에게 시키는 대상이 "코드 작성"에서 "업무 전체의 자동화"로 넓어졌고, 그 실행 주체가 바로 AI 에이전트입니다.

이 문서에서 다루는 다섯 가지는 서로 경쟁 제품이 아니라, 바이브 코딩 워크플로우를 구성하는 서로 다른 계층입니다.

💡 용어 정리
흔히 "오픈 크로우"라고 부르기도 하지만 공식 명칭은 OpenClaw(오픈클로)입니다. 2025년 11월 Peter Steinberger가 Clawdbot이라는 이름으로 출시했고, Moltbot을 거쳐 2026년 초 OpenClaw로 최종 개명했습니다. "Ollma"의 정확한 표기는 Ollama(올라마)입니다.

02OpenClaw(오픈클로) — 메시징 게이트웨이형 개인 AI 비서

SELF-HOSTED PERSONAL AI ASSISTANT

2.1 개념과 위상

OpenClaw는 셀프호스팅 게이트웨이(gateway)입니다. 내 컴퓨터나 서버에서 게이트웨이 프로세스 하나를 돌리면, 그것이 텔레그램·WhatsApp·디스코드·슬랙·iMessage 등 29개 이상의 메시징 채널과 AI 코딩 에이전트를 연결하는 다리가 됩니다. 즉 스마트폰 메신저에서 메시지를 보내면, 집이나 서버에 있는 AI 에이전트가 파일을 수정하고, 브라우저를 조작하고, 크론 작업을 등록한 뒤 결과를 다시 메신저로 답해 줍니다.

2026년 6월 기준 GitHub 스타 38만 개 이상으로, 세계에서 가장 널리 채택된 셀프호스팅 AI 에이전트 프레임워크입니다. MIT 라이선스 오픈소스이며 OpenClaw Foundation이 관리합니다.

2.2 아키텍처

핵심 구조는 단순합니다. Gateway가 단일 진실 공급원(single source of truth)이 되어 세션·라우팅·채널 연결을 모두 관리하고, 그 뒤에 실제 작업을 수행하는 에이전트 런타임이 붙습니다.

  • Gateway — 채널 플러그인, WebChat, 모바일 노드를 하나의 프로세스로 묶는 중앙 허브. 기본 포트 18789에서 브라우저 Control UI 제공
  • 채널 플러그인 — Telegram, WhatsApp, Discord, Slack, Signal, iMessage, Matrix, Teams 등. 필요 시 온디맨드 설치
  • 멀티에이전트 라우팅 — 에이전트별·워크스페이스별·발신자별로 격리된 세션. DM은 메인 세션 공유, 그룹채팅은 각각 별도 세션
  • 도구(Tools) — 브라우저 자동화, Canvas(라이브 UI 렌더링), 크론 스케줄링, 노드(모바일 기기 카메라·음성) 등 1급 지원
  • 스킬 생태계(ClawHub) — 커뮤니티가 만든 스킬을 설치해 기능 확장

2.3 설치 실습 (5분 퀵스타트)

요구사항: Node.js 24 권장(호환 최소 Node 22 LTS 22.19+), 사용할 AI 제공자의 API 키 1개, Windows/macOS/Linux 아무거나. Windows에서는 WSL2 환경을 권장합니다.

  1. 전역 설치
    npm install -g openclaw@latest
  2. 온보딩 마법사 실행 + 데몬 등록
    openclaw onboard --install-daemon

    게이트웨이, 워크스페이스, 채널, 스킬 설정을 단계별로 안내합니다. 여기서 모델 제공자(Anthropic, OpenAI, Ollama 등)를 선택하고 API 키를 넣습니다.

  3. 대시보드 접속
    openclaw dashboard
# 브라우저에서 http://127.0.0.1:18789/ 접속
  4. 채널 연결 (텔레그램이 가장 빠름)

    BotFather로 봇 토큰을 발급받아 온보딩에서 붙여넣으면, 이후 폰에서 텔레그램으로 에이전트와 대화할 수 있습니다.

설정 파일은 ~/.openclaw/openclaw.json에 있습니다. 보안을 위해 최소한 발신자 허용 목록은 걸어두는 것이 좋습니다.

{
  "channels": {
    "whatsapp": {
      "allowFrom": ["+8210XXXXXXXX"],
      "groups": { "*": { "requireMention": true } }
    }
  },
  "messages": { "groupChat": { "mentionPatterns": ["@openclaw"] } }
}

2.4 바이브 코딩 관점에서의 OpenClaw

OpenClaw의 본질은 "바이브 코딩의 인터페이스를 IDE에서 메신저로 옮긴 것"입니다. 지하철에서 텔레그램으로 "어제 짜둔 크롤러에 재시도 로직 붙여서 커밋해줘"라고 보내면, 집의 게이트웨이가 코딩 에이전트를 깨워 작업을 수행합니다. 개발자가 키보드 앞에 없어도 자연어 지시만으로 개발·운영이 이어지는, 바이브 코딩의 "비동기화·모바일화" 단계라고 볼 수 있습니다.

⚠ 주의
OpenClaw는 셸 명령 실행 권한을 가진 에이전트를 메신저에 노출하는 구조이므로, 발신자 허용 목록(allowFrom)과 토큰 관리를 소홀히 하면 사실상 원격 셸을 외부에 여는 것과 같습니다. 반드시 공식 문서의 Security 섹션을 따라 잠그고 사용하세요.

03Hermes Agent(헤르메스) — 자가학습하는 자율 에이전트

THE AGENT THAT GROWS WITH YOU

3.1 개념과 위상

Hermes Agent는 오픈소스 모델 연구소 Nous Research가 2026년 2월 공개한 자율 AI 에이전트입니다. 슬로건은 "함께 성장하는 에이전트(The agent that grows with you)". OpenClaw가 "채널 연결"에 무게를 둔다면, Hermes의 정체성은 학습 루프(learning loop)에 있습니다. 오래 돌릴수록 더 유능해지는 것이 설계 목표입니다.

3.2 핵심 기능 — 무엇이 다른가

① 닫힌 학습 루프 (Closed Learning Loop)

  • 스킬 자동 생성 — 어떤 작업을 수행한 경험으로부터 재사용 가능한 "스킬"을 스스로 만들어 저장
  • 사용 중 자가 개선 — 스킬을 쓰면서 스스로 다듬음. 지식을 영속화하라고 스스로에게 넛지(nudge)를 보냄
  • 크로스 세션 메모리 — SQLite FTS5 전문 검색 + LLM 요약으로 과거 대화를 검색·회상. Honcho 기반 사용자 모델링으로 "나"에 대한 이해가 세션을 넘어 누적
  • 오픈 스킬 표준 — agentskills.io 표준과 호환. 스킬을 커뮤니티(Skills Hub)에서 공유·설치 가능

② 실행 환경의 유연성

터미널 백엔드를 6종 지원합니다: local, Docker, SSH, Daytona, Singularity, Modal. Daytona/Modal은 서버리스 영속화를 지원해, 에이전트 환경이 유휴 시 동면(hibernate)하여 비용이 거의 0에 수렴합니다. "노트북에서 시작해 GPU 클러스터·서버리스까지"가 마케팅 문구가 아니라 실제 아키텍처입니다.

③ 도구와 채널

웹 검색, 페이지 추출, 브라우저 자동화(내비게이션·클릭·타이핑·스크린샷), 비전 분석, 이미지 생성, TTS, 멀티모델 추론 등 60개 이상의 내장 도구를 제공하며, MCP 서버 연결로 무한 확장됩니다. 메시징 게이트웨이도 내장되어 CLI 외에 Telegram, Discord, Slack, WhatsApp, Signal, Matrix, Email, SMS 등 20개 이상 플랫폼에서 대화할 수 있습니다. 격리된 서브에이전트를 병렬로 스폰하는 위임(delegation)과 내장 크론 자동화도 지원합니다.

3.3 설치 실습

사전 요구사항이 없습니다. 설치 스크립트가 필요한 것을 전부 자동 설치합니다.

  1. 설치 (Linux / macOS / WSL2 / Android Termux)
    curl -fsSL https://hermes-agent.nousresearch.com/install.sh | bash

    Windows 네이티브는 PowerShell에서:

    iex (irm https://hermes-agent.nousresearch.com/install.ps1)

    또는 공식 사이트에서 Hermes Desktop 설치 파일(Windows/macOS)을 받아 GUI로 설치할 수도 있습니다.

  2. 초기 설정
    hermes setup --portal

    Nous Portal OAuth 한 번으로 모델 + 4종 게이트웨이 도구(웹 검색, 이미지 생성, TTS, 브라우저)가 모두 연결됩니다. 대안으로 OpenRouter, OpenAI, 또는 임의의 OpenAI 호환 엔드포인트(→ Ollama 연동 가능)를 직접 설정할 수 있습니다.

  3. 첫 대화
    hermes
# 터미널 대화형 세션 시작. 이후 텔레그램 등 게이트웨이 연결은 config에서.

프로젝트 컨텍스트 파일과 SOUL.md(에이전트의 기본 성격 정의), 명령 승인·컨테이너 격리 같은 보안 장치도 문서화되어 있습니다.

3.4 바이브 코딩 관점에서의 Hermes

바이브 코딩의 고질적 약점은 "매번 처음부터 다시 설명해야 한다"는 것입니다. Hermes는 이 문제를 정면으로 겨냥합니다. 내 코딩 스타일, 자주 쓰는 배포 절차, 반복 실수를 스킬과 메모리로 축적하므로, 시간이 지날수록 "대충 말해도 알아듣는" 에이전트가 됩니다. 바이브 코딩이 1회성 프롬프트 기술에서 장기 협업 관계로 진화하는 방향을 보여주는 대표 사례입니다. OpenClaw 대비 개발자 커뮤니티 관심(신규 기여자 수 등)에서 빠르게 성장한 배경도 이 학습 루프 덕분입니다.

04Ollama와 로컬 LLM — 내 컴퓨터에서 돌리는 모델 계층

MODEL LAYER: PRIVACY-FIRST

4.1 개념

Ollama는 대규모 언어 모델을 내 컴퓨터에서 직접 실행하게 해주는 오픈소스 런타임입니다. Docker가 컨테이너를 다루듯 모델을 pull → run으로 다루며, 복잡한 Python 환경이나 CUDA 설정 없이 시작할 수 있습니다. 로컬 LLM을 쓰는 이유는 명확합니다.

🔒 프라이버시

입력 데이터가 외부 서버로 나가지 않음. 사내 문서·개인정보 처리에 유리

💰 비용

토큰 과금·구독료 없음. 에이전트를 24시간 돌려도 전기료만 발생

📴 오프라인

모델을 한 번 받으면 인터넷 없이 동작

🔧 커스터마이징

Modelfile로 시스템 프롬프트·파라미터·템플릿 정의, 파인튜닝 모델 사용 가능

4.2 설치와 기본 사용

  1. 설치
    # macOS / Windows: ollama.com에서 설치 파일 다운로드
# Linux:
curl -fsSL https://ollama.com/install.sh | sh
  2. 모델 받아서 실행
    ollama run qwen3:8b          # 없으면 자동 pull 후 대화 시작
ollama pull qwen3-coder:30b  # 다운로드만
ollama list                  # 설치된 모델 목록
ollama ps                    # 메모리에 로드된 모델 확인
  3. API로 사용 (기본 포트 11434)
    curl http://localhost:11434/api/chat -d '{
  "model": "qwen3:8b",
  "messages": [{"role":"user","content":"안녕"}]
}'

    OpenAI 호환 엔드포인트(/v1)도 제공되어 기존 SDK 코드를 그대로 재사용할 수 있습니다. 단, OpenClaw 연동 시에는 예외가 있습니다(4.5절).

4.3 2026년 추천 모델 (Ollama 기준)

모델크기/특징필요 메모리(대략)용도

4.5 에이전트와의 연동 — 스택이 완성되는 지점

Ollama가 진짜 힘을 발휘하는 것은 OpenClaw·Hermes의 두뇌로 꽂힐 때입니다.

OpenClaw + Ollama

openclaw onboard에서 제공자로 Ollama를 선택하면 Local only / Cloud only / Cloud+Local 세 모드 중 고를 수 있고, 로컬 모델을 자동 발견·자동 pull합니다.

🚫 흔한 실수
OpenClaw는 Ollama의 네이티브 API(/api/chat)를 사용합니다. OpenAI 호환 URL(http://host:11434/v1)을 넣으면 도구 호출(tool calling)이 깨져서 모델이 도구 호출 JSON을 그냥 텍스트로 뱉는 문제가 생깁니다. 반드시 baseUrl: "http://host:11434"처럼 /v1 없이 설정하세요.

Hermes + Ollama

Hermes는 임의의 OpenAI 호환 엔드포인트를 제공자로 지정할 수 있으므로, config에서 Ollama의 로컬 엔드포인트를 가리키면 완전 로컬 추론 스택이 됩니다. 프로젝트 성격에 따라 "민감한 작업은 로컬 모델, 어려운 추론은 클라우드 모델"로 갈아타는 멀티모델 운용이 Hermes의 설계 철학과 잘 맞습니다.

4.8 바이브 코딩 관점에서의 Ollama

바이브 코딩의 반복 루프(요청 → 생성 → 확인 → 재요청)는 토큰을 대량 소모합니다. 로컬 LLM은 이 루프를 비용 걱정 없이 무한 반복하게 해주고, 회사 코드가 외부로 나가면 안 되는 환경에서 바이브 코딩을 가능하게 하는 유일한 선택지이기도 합니다. 2026년 현재 qwen3-coder급 로컬 모델은 일상적인 코딩 작업에서 충분히 실용적인 수준에 도달했습니다.

5.1 두 얼굴의 Hostinger

Hostinger는 전통적으로 워드프레스 등 웹사이트 호스팅 업체로 알려져 있지만, 이 문서의 맥락에서 중요한 것은 두 번째 얼굴, 즉 AI 에이전트 호스팅 인프라입니다. 2026년 들어 Hostinger는 OpenClaw 전용 원클릭 배포 상품을 내놓으며 "개인 AI 비서를 24시간 클라우드에 상주시키는" 용도로 빠르게 자리잡았고, OpenClaw 공식 문서에도 Hostinger 설치 가이드가 별도 페이지로 등재되어 있습니다.

5.2 왜 에이전트에 VPS가 필요한가

내 PC에서 에이전트를 돌리면 공짜지만, PC를 끄면 에이전트도 죽습니다. "새벽 3시에 크론으로 리포트 생성", "여행 중에 텔레그램으로 서버 점검 지시" 같은 시나리오는 상시 구동 환경을 요구하고, 그 최소 비용 답안이 월 $6 안팎의 VPS입니다. Hostinger의 OpenClaw 템플릿은 VPS Docker Manager에 사전 구성되어 있어 클릭 몇 번으로 게이트웨이가 올라오며, Nexos AI 통합으로 외부 API 키 없이 크레딧 방식으로 여러 모델(Anthropic, OpenAI, Gemini 등)을 쓸 수도 있습니다.

⚠ VPS에서의 로컬 LLM은 비현실적
KVM 1(4GB RAM)급 VPS에서는 Ollama로 의미 있는 크기의 모델을 돌리기 어렵습니다. 일반적 구성은 VPS에는 에이전트(게이트웨이)만 올리고, 두뇌는 클라우드 LLM API를 사용하는 것입니다. 로컬 LLM 중심 스택을 원하면 집의 GPU 머신에서 돌리고 Tailscale 등으로 원격 접속하는 편이 낫습니다.

5.3 실습: Hostinger VPS에 OpenClaw 올리기 (개요)

  1. Hostinger에서 KVM 플랜 구매 시 템플릿으로 "OpenClaw" 선택 (또는 기존 VPS에서 Docker Manager → OpenClaw 템플릿 적용)
  2. 발급된 URL로 Control UI 접속 → 온보딩에서 모델 제공자·API 키 설정
  3. 텔레그램 봇 토큰 연결 → 폰에서 대화 시작
  4. 보안 설정: 발신자 허용 목록, 게이트웨이 토큰, 방화벽(UFW), SSH 키 인증 적용

5.4 바이브 코딩 관점에서의 Hostinger

바이브 코딩의 마지막 단계는 언제나 "만든 것을 세상에 내놓는 것"입니다. Hostinger는 이 지점에서 이중 역할을 합니다. 바이브 코딩으로 만든 웹사이트·앱의 배포처(전통적 호스팅)이자, 바이브 코딩을 수행하는 에이전트 자체의 거주지(VPS)입니다. "에이전트에게 사이트를 만들게 하고, 그 에이전트가 사는 같은 서버 옆에 사이트를 배포한다"는 자기완결적 루프가 가능해집니다.

조합 C. 하이브리드 스택 실전 추천

구성: 집 GPU 머신(Ollama) + Hostinger VPS(OpenClaw 게이트웨이) + 멀티모델 라우팅

게이트웨이는 VPS에 상주시켜 24/7 접근성을 확보하고, 민감하거나 대량 반복인 작업은 Tailscale로 연결된 집 Ollama로, 고난도 추론은 클라우드 API로 라우팅합니다. OpenClaw의 Ollama provider 설정에서 baseUrl을 집 머신의 테일넷 주소로 지정하면 됩니다(이때도 /v1 금지). 적합: 비용·프라이버시·가용성 세 마리 토끼를 잡으려는 중급 이상 사용자.

08보안·비용 체크리스트

BEFORE YOU SHIP

보안

  • 에이전트는 셸 실행 권한을 가진 소프트웨어입니다. 메시징 채널에는 반드시 발신자 허용 목록을 설정하세요 (OpenClaw allowFrom, 그룹은 멘션 필수화)
  • VPS는 SSH 키 인증 + 방화벽 기본. Control UI를 공인 IP에 그대로 노출하지 말고 Tailscale 등 사설망 경유를 권장
  • Hermes의 명령 승인(command approval)·컨테이너 격리 옵션을 켜고, 신뢰하지 않는 스킬은 코드를 읽고 설치
  • curl | bash 설치는 편리하지만 실행 전 스크립트를 한 번 열어보는 습관을 권장
  • API 키는 환경변수/auth 프로필로 관리하고 config 파일을 공개 저장소에 커밋하지 않기

비용

  • 에이전트 자체(OpenClaw/Hermes)는 무료. 실비용은 ① 모델 토큰 ② 서버 두 곳에서 발생
  • 클라우드 LLM으로 에이전트를 상시 구동하면 토큰 비용이 생각보다 빠르게 누적됩니다. 크론 작업 빈도와 모델 등급(작업별 라우팅)으로 제어하세요
  • Hostinger 프로모션가는 장기 약정 기준이며 갱신가가 다릅니다 (예: Managed OpenClaw $5.99 → 갱신 $11.99). 약정 종료 시점을 캘린더에 등록해 두세요
  • 로컬 LLM은 토큰 무료지만 GPU 전력·초기 하드웨어 비용이 있습니다. 일 사용량이 적으면 API가, 많으면 로컬이 유리합니다

09호스팅 서비스 심층 비교 — Hostinger 대안과 Vercel류 플랫폼

HOSTING LANDSCAPE 2026

Hostinger를 고르기 전에 알아야 할 것이 있습니다. 2026년 호스팅 시장은 크게 두 진영으로 나뉘어 있고, 두 진영은 "서버를 어떻게 다루는가"라는 철학 자체가 다릅니다.

진영 A
서버 임대형
VPS / 전통 호스팅Hostinger · Hetzner · Contabo · DigitalOcean · Vultr
진영 B
플랫폼형
서버리스 / PaaSVercel · Netlify · Cloudflare Pages · Railway · Render

진영 A(서버 임대형)는 "가상 컴퓨터 한 대"를 빌려줍니다. root 권한을 받아 OS부터 직접 관리하며, 프로세스가 24시간 상주합니다. 진영 B(플랫폼형)는 "코드를 올릴 자리"를 빌려줍니다. git push만 하면 빌드·배포·스케일링·HTTPS를 플랫폼이 알아서 처리하지만, 서버 자체는 만질 수 없습니다.

9.1 Hostinger와 같은 진영 — VPS 제공자 비교

2026년 저가 VPS 시장은 세 가지 성격으로 정리됩니다. Hetzner는 "성능·가성비", Contabo는 "달러당 최대 자원", Hostinger는 "관리형 사용자 경험" 진영의 대표주자이며, DigitalOcean과 Vultr는 그 사이에서 개발자 생태계로 승부합니다.

제공자진입 가격(월)강점약점OpenClaw류 에이전트 호스팅
Hostinger $6.49~ (KVM 1: 1vCPU/4GB) 원클릭 OpenClaw 템플릿, hPanel(한국어), 관리 부담 최소 프로모션가↔갱신가 격차, 장기 약정 유도, 성능은 중위권 최적 공식 템플릿 존재

💡 선택 기준 요약
클릭 몇 번으로 끝내고 싶다 → Hostinger (관리형 UX + OpenClaw 템플릿). 성능 대비 가격이 최우선 → Hetzner. RAM·디스크가 많이 필요한데 예산이 없다 → Contabo (단, 피크 성능 저하 감수). 배우면서 운영하고 싶다 → DigitalOcean (문서가 교과서). 한국에서 낮은 지연시간 → Vultr 서울 리전.

9.2 Vercel과 같은 진영 — 플랫폼형(PaaS/서버리스) 비교

Vercel류 플랫폼은 다시 둘로 나뉩니다. 정적/서버리스 특화(Vercel, Netlify, Cloudflare Pages)와 컨테이너 상시 실행형(Railway, Render)입니다. 이 구분이 뒤에서 중요해집니다.

플랫폼무료 티어유료 시작가강점약점
Vercel Hobby — 개인용, 제약 많음(상업적 사용 제한) $20/월/인 Next.js 최적화의 정점, 프리뷰 배포 UX, v0 등 AI 도구 연계 인원당 과금, 대역폭 초과 요금 비쌈, Next.js 외 이점 감소

Render 무료 인스턴스 (유휴 시 슬립) 플랜 정액 + 컴퓨트 (Pro $25/월~) 예측 가능한 요금, 관리형 PostgreSQL·백업, 안정적 가동률, 백그라운드 워커 1급 지원 무료 티어 콜드스타트, Railway보다 유연성 낮음

9.3 근본 차이 — Hostinger(VPS)와 Vercel(서버리스)은 무엇이 다른가

가격표를 비교하기 전에, 두 모델의 실행 방식 차이를 이해해야 합니다. 이것이 "에이전트를 어디에 두는가"를 결정하기 때문입니다.

관점Hostinger류 (VPS)Vercel류 (서버리스/PaaS)
빌리는 것가상 컴퓨터 한 대 (root 권한)코드 실행 플랫폼 (서버 접근 불가)
실행 모델프로세스가 24시간 상주요청이 올 때만 함수 실행 후 소멸 (Railway/Render는 예외적으로 상시 컨테이너)
배포 방식SSH 접속 → 직접 설치·설정 (또는 템플릿)git push → 자동 빌드·배포·프리뷰 URL
스케일링수동 (플랜 업그레이드)자동 (트래픽 폭증에도 무중단)
운영 책임보안 패치·방화벽·백업 모두 내 몫플랫폼이 전담
비용 구조고정 월정액 (트래픽 적어도 동일)무료 시작 → 사용량 비례 (폭증 시 요금 폭탄 가능)
실행 제약없음 — DB, 크론, 웹소켓, 게임서버 뭐든함수 실행 시간 제한, 장기 실행 프로세스 불가
상태(state)디스크에 자유롭게 저장무상태(stateless) — 외부 DB/스토리지 필요

Hostinger류 (VPS) 장단점

👍 장점

· 완전한 제어권 — 무엇이든 설치·실행

· 상시 프로세스 가능 → AI 에이전트 상주의 필수 조건

· 고정 요금 — 트래픽 폭증에도 요금 동일

· 여러 서비스를 한 서버에 합쳐 비용 절감

👎 단점

· 보안·백업·패치 전부 직접 책임

· 스케일링 수동 — 트래픽 몰리면 다운 위험

· 초기 설정 학습 곡선 (SSH, 방화벽, Docker)

· 프리뷰 배포·롤백 같은 현대적 DX 부재

Vercel류 (서버리스/PaaS) 장단점

👍 장점

· git push 한 번이면 배포 끝 — 바이브 코딩과 최고 궁합

· PR마다 프리뷰 URL 자동 생성, 원클릭 롤백

· 글로벌 CDN·HTTPS·자동 스케일 기본 제공

· 개인 프로젝트는 사실상 무료 (특히 Cloudflare)

👎 단점

· 상시 실행 프로세스 불가 → OpenClaw/Hermes 못 올림 (Railway/Render 제외)

· 함수 실행 시간·메모리 제한, 콜드 스타트

· 사용량 과금 — 트래픽 폭증 시 요금 예측 불가

· 플랫폼 종속(lock-in) — 특정 기능에 의존할수록 이전 어려움

🚫 핵심: Vercel에는 OpenClaw를 올릴 수 없다
OpenClaw·Hermes 같은 에이전트는 24시간 살아있는 게이트웨이 프로세스(웹소켓 연결, 크론, 세션 상태)를 요구합니다. Vercel·Netlify·Cloudflare Pages는 "요청 → 함수 실행 → 소멸" 모델이라 구조적으로 불가능합니다. 플랫폼형 중에서는 상시 컨테이너를 지원하는 Railway·Render만 가능하며, 이 경우 VPS보다 관리는 편하지만 상시 구동 비용은 대체로 VPS($6~)보다 높아집니다.

9.4 바이브 코딩 관점의 결론 — 둘은 경쟁이 아니라 분업

바이브 코딩 워크플로우에서 두 진영의 역할은 명확히 갈립니다.

  • 에이전트가 사는 곳(두뇌의 거처) → Hostinger류 VPS. 상시 프로세스·root 권한·고정 요금이 필요조건. 원클릭 템플릿이 있는 Hostinger, 가성비의 Hetzner, 서울 리전의 Vultr 중 선택
  • 결과물이 배포되는 곳(작품의 전시장) → Vercel류. 에이전트에게 "이 앱 배포해줘"라고 하면 에이전트가 vercel deploy나 git push를 실행하는 구조. Next.js면 Vercel, 트래픽 걱정 없애려면 Cloudflare, 다기능 무료면 Netlify
  • 에이전트 + 배포를 한 플랫폼에서 → Railway/Render (관리 편의성 우선) 또는 VPS 한 대에 Docker로 전부 (비용 우선)

정리하면 "Hostinger vs Vercel"은 잘못된 질문입니다. 올바른 질문은 "내 에이전트는 어느 VPS에 두고, 에이전트가 만든 결과물은 어느 플랫폼에 배포하게 할 것인가"입니다. 전형적인 2026년형 바이브 코딩 셋업은 Hostinger(또는 Hetzner) VPS의 OpenClaw + Vercel(또는 Cloudflare)에 배포되는 산출물의 조합입니다.
