🔍 자세히 보기 — 패키지 관리자와 WSL2가 정확히 뭔가요?

패키지 관리자는 프로그램을 "공식 홈페이지에서 파일 다운로드 → 더블클릭 설치" 대신 터미널 명령어 한 줄로 설치·업데이트·삭제하게 해주는 도구입니다. 앱스토어를 터미널 버전으로 옮겨놓았다고 생각하면 됩니다.

① Homebrew(brew) — macOS·Linux용 패키지 관리자입니다. brew install node처럼 쓰며, 설치와 동시에 실행 경로(PATH, 아래 설명 참고)까지 자동으로 등록해 주는 경우가 많아 맥에서는 사실상 표준입니다.

② winget — Microsoft가 만든 Windows 10/11 공식 기본 내장 패키지 관리자입니다. 별도 설치 없이 바로 winget install 프로그램ID로 사용합니다. 이 교재는 winget을 기본으로 안내합니다.

③ Chocolatey(choco) — winget이 나오기 전부터 있던 커뮤니티 기반 윈도우 패키지 관리자입니다. winget에 아직 등록되지 않은 일부 프로그램이나 오래된 개발 도구를 설치할 때 대안으로 씁니다. 둘 중 하나만 있으면 충분하며, 이 교재의 실습은 winget만으로 진행됩니다.

④ WSL2(Windows Subsystem for Linux 2) — 윈도우 안에서 진짜 리눅스 커널을 그대로 돌리는 가상 환경입니다. 왜 권장할까요? Claude Code 같은 CLI 도구와 오픈소스 생태계 대부분이 macOS·Linux 환경을 기준으로 만들어져 있어서, PowerShell에서 직접 실행하면 경로 표기(\ vs /)나 줄바꿈 문자 차이로 가끔 오류가 나는 도구가 있습니다. WSL2 안에서는 이런 문제 없이 맥·리눅스 사용자와 동일한 명령어·튜토리얼을 그대로 따라 할 수 있습니다. 설치는 PowerShell(관리자 모드)에서 wsl --install 한 줄이면 되고, 재부팅 후 리눅스 배포판(기본 Ubuntu) 안에서 사용자 이름·비밀번호만 설정하면 바로 리눅스 터미널을 쓸 수 있습니다. WSL2 안에서는 brew 대신 apt install 같은 리눅스 패키지 관리자를 사용합니다.

맥에서 10분 설치 (터미널에 순서대로 입력)

# 1. Homebrew 설치 (brew.sh 안내문 그대로)
/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

# 2. 필수 도구 한 번에 설치
brew install node git python
brew install --cask visual-studio-code

# 3. Claude Code 설치 후 로그인
npm install -g @anthropic-ai/claude-code
claude   # 첫 실행 시 브라우저 로그인 안내가 뜹니다

# 4. 설치 확인
node -v && git --version && python3 --version   # 버전이 나오면 성공!

⚠️ "command not found"가 뜬다면 — PATH 문제입니다 설치는 끝났는데 brew나 node를 쳤을 때 zsh: command not found가 뜨는 경우가 있습니다. 원인은 PATH(터미널이 명령어의 실제 파일을 찾아보는 폴더 목록)에 새로 설치한 프로그램의 위치가 아직 등록되지 않았기 때문입니다. 특히 Apple Silicon(M1~M4) 맥에서 Homebrew를 처음 설치하면, 설치 마지막에 아래와 같은 문장을 내 셸 설정 파일에 직접 추가하라고 안내합니다 — 이 줄을 빠뜨리면 매번 이 문제가 생깁니다. 
# ~/.zprofile 파일 맨 아래에 한 번만 추가 (설치 스크립트가 알려주는 그대로)
echo 'eval "$(/opt/homebrew/bin/brew shellenv)"' >> ~/.zprofile
eval "$(/opt/homebrew/bin/brew shellenv)"
추가한 뒤 터미널을 완전히 껐다 켜거나 source ~/.zprofile을 입력하면 반영됩니다. 이 과정이 막막하면 AI에게 "brew 설치는 됐는데 command not found가 떠. PATH 설정을 도와줘"라고 그대로 물어보세요.

윈도우에서 10분 설치 (PowerShell 관리자 모드)

# 1. winget으로 필수 도구 설치
winget install OpenJS.NodeJS.LTS
winget install Git.Git
winget install Python.Python.3.12
winget install Microsoft.VisualStudioCode

# 2. (권장) WSL2 설치 — 리눅스 환경, CLI 도구 호환성 최고
wsl --install

# 3. Claude Code 설치
npm install -g @anthropic-ai/claude-code

🔎 윈도우 설치 명령어, 한 줄씩 뜯어보기

  1. PowerShell을 "관리자 권한"으로 열기 — 시작 메뉴에서 "PowerShell" 검색 → 마우스 오른쪽 클릭 → "관리자 권한으로 실행". winget과 wsl 설치는 시스템 전체에 영향을 주는 작업이라 관리자 권한이 필요합니다.
  2. winget install OpenJS.NodeJS.LTS 등 4줄 — winget은 "회사이름.제품이름" 형태의 고유 ID로 프로그램을 찾습니다(예: OpenJS.NodeJS.LTS). 한 줄씩 실행되며, 중간에 "라이선스에 동의하십니까? (Y/n)"가 뜨면 Y를 입력하고 Enter를 누릅니다.
  3. wsl --install — 앞서 설명한 WSL2(리눅스 환경)를 설치합니다. 완료 후 컴퓨터 재부팅이 필요합니다. 재부팅하면 검은 창이 자동으로 뜨며 리눅스용 사용자 이름과 비밀번호를 새로 만들라고 안내합니다 — 윈도우 로그인 계정과는 별개이니 기억하기 쉬운 것으로 정하세요.
  4. npm install -g @anthropic-ai/claude-code — Node.js 설치가 끝난 뒤에 실행해야 합니다(npm은 Node.js에 포함되어 있음). -g는 "이 컴퓨터 어디서든 쓸 수 있게" 전역 설치한다는 뜻입니다.
⚠️ "'python'은(는) 내부 또는 외부 명령... 이 아닙니다"가 뜬다면 — 역시 PATH 문제

윈도우도 맥과 똑같이 PATH(명령어를 찾는 폴더 목록) 문제가 가장 흔합니다. winget으로 설치했다면 보통 자동으로 PATH에 등록되지만, 혹시 python.org에서 직접 설치했다면 설치 화면 맨 아래의 "Add python.exe to PATH" 체크박스를 반드시 체크해야 합니다. 이미 체크 없이 설치했다면:

  1. 시작 메뉴 → "환경 변수 편집" 검색 → "시스템 환경 변수 편집" 실행
  2. "환경 변수" 버튼 → 사용자 변수의 Path 선택 → 편집
  3. "새로 만들기"로 Python 설치 경로(예: C:\Users\내계정\AppData\Local\Programs\Python\Python312\)를 추가
  4. PowerShell 창을 모두 닫았다가 새로 열어야 반영됨

이 과정도 막막하면 AI에게 "윈도우에서 python이 command not found 떠. PATH 설정을 도와줘"라고 그대로 물어보면 내 화면 상황에 맞춰 단계별로 알려줍니다.

💡 초보자 팁 — "터미널이 무서워요" 터미널은 그저 글자로 컴퓨터에게 말을 거는 창입니다. 마우스 클릭 대신 문장을 입력할 뿐이며, 바이브 코딩에서는 명령어조차 AI에게 "이거 어떻게 설치해?"라고 물어보면 알려줍니다. 설치 중 에러가 나면 에러 메시지를 그대로 복사해 Claude나 ChatGPT에 붙여넣으세요. 대부분 30초 안에 해결됩니다.

2단계

바이브 코딩의 기본 과정 — PRD에서 로컬 테스트까지

학습 목표 — 별도의 웹 LLM에서 PRD(제품 요구사항 문서)를 만들고, AI 코딩 도구로 구현하고, 내 컴퓨터에서 테스트하는 표준 사이클을 익힌다. 일반 프로그램과 웹 프로그램 두 가지 실전 사례로 연습하고, 외부 API(유튜브·공공데이터)와 LLM API를 연결하는 실전 예제까지 완성한다.

2-1. 별도의 LLM에서 PRD 만들기

PRD(Product Requirements Document, 제품 요구사항 문서)는 "어떤 제품을 만들 것인가"를 정의하는 문서로, 구체적인 구현 방법이 아니라 제품이 해야 할 것과 하지 말아야 할 것을 정리합니다. 왜 코딩보다 문서가 먼저일까요? 개발 중 "이 기능도 추가하면 어때?"라는 유혹이 올 때 "PRD에 없으니까 다음 버전에 하자"라고 판단하는 기준이 되기 때문입니다. AI에게도 명확한 지도를 주는 셈이라 결과물의 품질이 크게 올라갑니다.

👨‍💻 전문가 의견 — PRD 작성의 실전 팁 ① 바이브 코딩을 할 때는 '기획서'보다 PRD라는 단어를 쓰면 AI가 더 잘 알아듣고, LLM이 영어 데이터를 압도적으로 많이 학습했기 때문에 PRD를 영어로 만들면 유리하다는 조언이 있습니다(한국어로 대화하고 문서만 영어로 뽑아달라고 하면 됩니다). ② 구조화된 워크플로우를 정리한 개발자는 PRD를 만든 뒤 LLM에게 이를 바탕으로 단계별 실행 계획을 세우게 하고, 기능을 작고 완결된 단위(수직 분할)로 DB→서버→UI 전체 계층을 한 번에 구현하되 점진적으로 복잡도를 높이라고 권합니다. ③ 한 번에 하나의 기능을 맡아 끝까지 완성하는 것이 초보자의 혼란을 가장 줄여 줍니다.

웹 LLM(claude.ai 등)에 넣는 PRD 생성 프롬프트 예시

# claude.ai / ChatGPT / Gemini 채팅창에 붙여넣기
당신은 시니어 프로덕트 매니저입니다. 아래 아이디어로 초보 개발자와
AI 코딩 도구가 함께 쓸 PRD를 마크다운으로 작성해 주세요.

[아이디어] 매일 마신 물의 양을 기록하고 목표 달성률을 보여주는 웹앱

PRD에 포함할 것:
1. 목적과 해결하려는 문제   2. 대상 사용자
3. 핵심 기능 (MVP 3개 이내로 제한)   4. 화면 구성 설명
5. 기술 스택 (초보자 기준: HTML/CSS/JS 단일 파일 권장)
6. 이번 버전에서 하지 않을 것 (Out of Scope)

결과로 받은 문서를 PRD.md라는 파일로 프로젝트 폴더에 저장합니다. 이것이 2단계 전체의 "설계도"가 됩니다.

2-2. AI 도구에서 코딩하기 — 두 가지 실전 사례

사례 A · 일반 프로그램 (Python 데스크톱 도구)

웹이 아닌 "내 컴퓨터에서 도는 프로그램"의 예로, 폴더 안의 파일 이름을 규칙에 따라 일괄 변경하는 도구를 만들어 봅니다. 방송 촬영본 정리처럼 반복 작업 자동화에 딱 맞는 유형입니다.

  1. 프로젝트 폴더 생성: mkdir file-renamer && cd file-renamer 후 PRD.md 저장
  2. Claude Code 실행(claude) 후 지시: "PRD.md를 읽고 Python으로 구현해줘. 실행 전 어떤 파일이 어떻게 바뀔지 미리보기를 먼저 보여주는 안전장치를 꼭 넣어줘."
  3. AI가 renamer.py를 생성하면 코드를 훑어보고(이해 안 되는 줄은 "이 줄은 뭐 하는 거야?"라고 질문) 실행: python3 renamer.py
  4. 테스트용 폴더에서 먼저 실행해 보고, 수정 사항을 자연어로 요청: "날짜를 파일 맨 앞에 YYYYMMDD_ 형식으로 붙여줘"

사례 B · 웹 프로그램 (할 일 관리 웹앱)

웹 프로그램은 브라우저에서 열리는 프로그램입니다. 결과가 즉시 눈에 보여 입문용으로 가장 좋습니다.

  1. 폴더 생성(todo-app) 후 PRD.md 저장 — 핵심 기능: 할 일 추가 / 완료 체크 / 삭제, 새로고침해도 유지(localStorage)
  2. AI 도구에 지시: "PRD.md대로 index.html, style.css, app.js 3개 파일로 만들어줘. 프레임워크 없이 순수 HTML/CSS/JS로."
  3. 생성된 index.html을 브라우저로 열어 즉시 확인
  4. 디자인·기능 수정을 대화로 반복: "완료된 항목은 회색 취소선으로", "다크 모드 버튼 추가해줘"
💡 좋은 지시(프롬프트)의 공식 "이메일과 비밀번호 기반 로그인 기능을 만들어줘. JWT 토큰 방식을 사용하고, 비밀번호는 bcrypt로 해싱해야 해. 실패 시 'Invalid credentials' 에러를 반환하고, 성공 시 user 객체와 token을 함께 반환해"처럼 구체적이고 명확할수록 AI는 정확한 결과를 제공합니다. 또한 AI 모델은 학습 데이터에 많이 노출된 기술을 훨씬 잘 다루므로, 실험적인 라이브러리 대신 React 같은 인기 기술 스택을 선택하면 더 정확하고 안정적인 코드를 얻을 수 있습니다.

2-3. 로컬에서 테스트하기

"로컬(local)"은 배포 전 내 컴퓨터 안을 뜻합니다. 유형별 테스트 방법은 다음과 같습니다.

프로그램 유형실행 방법확인 위치
Python 일반 프로그램python3 파일명.py터미널 출력
정적 웹 (HTML/CSS/JS)index.html 더블클릭, 또는 VS Code 'Live Server' 확장브라우저

localhost:3000은 "내 컴퓨터의 3000번 문"이라는 뜻의 주소로, 나만 볼 수 있는 개발용 미리보기입니다. 전 세계에 공개하는 방법은 4단계(배포)에서 다룹니다.

  • PRD의 핵심 기능이 모두 동작하는가? (기능 하나씩 직접 눌러보기)
  • 이상한 값을 넣어도 안 깨지는가? (빈 값, 아주 긴 글자, 한글/이모지)
  • 에러가 나면 → 에러 메시지를 통째로 복사해 AI에게 붙여넣고 "고쳐줘"
  • 브라우저 개발자 도구(F12) Console 탭에 빨간 에러가 없는가?
👨‍💻 전문가 의견 — 테스트도 AI에게 Claude Code 가이드에 따르면 "관련 파일을 읽고, 계획을 세우고, 코드를 짜고, 커밋"하는 자동화 워크플로우가 가능하며, 기능 단위 테스트뿐 아니라 "이 기능 구현하다 저 기능이 깨지진 않았나"를 보는 통합 테스트, "로그인부터 결제까지 모든 단계가 동작하나"를 보는 E2E 테스트도 AI가 만들어 실행해 줄 수 있습니다. 이는 스펙대로 구현됐는지를 바이브 코더와 AI 양측이 확인하는 전략입니다.

2-4. 실전 예제 — 외부 데이터(API) 가져와서 LLM으로 분석하기

지금까지는 데이터가 내 앱 안에만 있었습니다. 이번에는 API를 통해 바깥세상의 데이터를 가져옵니다. 실습 주제는 유튜브 채널을 등록하면 그 채널의 최신 영상 리스트를 받아오고, LLM(Claude)에게 콘텐츠 전략을 분석시키는 프로그램입니다. 이 과정에서 초보자가 가장 많이 실수하는 API 키 발급·보관·배포 보안을 함께 배웁니다.

STEP 0 · API 키란? — 그리고 공공 데이터의 세계

API 키는 "이 데이터를 요청한 사람이 등록된 사용자 본인"임을 증명하는 출입증 문자열입니다. 유튜브뿐 아니라 대한민국 공공데이터포털(data.go.kr)도 같은 방식입니다 — 회원가입 후 원하는 데이터(예: 기상청 날씨, 버스 도착 정보, 한국거래소 시세)의 "활용 신청"을 하면 인증키(serviceKey)가 발급되고, 그 키를 주소에 붙여 요청하면 JSON 형식의 데이터를 받습니다. 발급 방식만 다를 뿐 "키를 받고 → 요청에 실어 보내고 → JSON을 받는" 구조는 전 세계 모든 API가 동일하므로, 아래 유튜브 예제를 한 번 익히면 공공 데이터에도 그대로 적용할 수 있습니다.

STEP 1 · 유튜브 API 키 발급 (키①)

  1. Google Cloud Console(console.cloud.google.com)에 구글 계정으로 로그인 → 상단에서 새 프로젝트 생성 (이름 예: youtube-analyzer)
  2. 좌측 메뉴 API 및 서비스 → 라이브러리 → "YouTube Data API v3" 검색 → 사용(Enable) 클릭
  3. API 및 서비스 → 사용자 인증 정보(Credentials) → + 사용자 인증 정보 만들기 → API 키
  4. 발급된 키(AIza...로 시작)를 복사. 키 제한 설정에서 "YouTube Data API v3만 허용"으로 제한해 두면 유출 시 피해를 줄일 수 있음

※ YouTube Data API는 무료이지만 하루 10,000 유닛의 할당량(quota)이 있습니다. 영상 목록 조회는 요청당 1~2유닛이라 학습용으로는 충분합니다.

STEP 2 · LLM(Claude) API 키 발급 (키②)

  1. Anthropic Console(console.anthropic.com) 접속 → 계정 생성 후 로그인
  2. 결제 수단 등록 후 소액 크레딧 충전 (API는 채팅 구독과 별도로, 쓴 만큼 과금되는 종량제)
  3. 좌측 API Keys → + Create Key → 이름 입력(예: youtube-project) → 생성
  4. 키는 생성 직후 딱 한 번만 전체가 표시됩니다. 즉시 복사해 안전한 곳(.env 파일)에 보관하세요. 잃어버리면 새로 발급해야 합니다

※ OpenAI(platform.openai.com), Google AI Studio(aistudio.google.com — 무료 할당량 제공) 등 다른 LLM도 발급 절차는 거의 동일합니다: 콘솔 접속 → API Keys 메뉴 → 생성 → 1회 표시 → 복사.

# ② 필요한 부품(라이브러리) 설치 → node_modules 폴더에 다운로드됨

🔍 자세히 보기 — npm install과 npm run dev는 대체 무슨 뜻일까?

npm(Node Package Manager)은 Node.js를 설치하면 함께 딸려오는 "프로그램 부품(라이브러리) 자동 장보기 도구"입니다. 전 세계 개발자들이 만들어 둔 200만 개 이상의 부품 창고(npm 레지스트리)에서 필요한 것을 내려받습니다.

① package.json — 프로젝트의 "설명서 겸 부품 목록"입니다. "이 프로젝트는 어떤 부품(dependencies)이 필요하고, 어떤 실행 버튼(scripts)이 있는지"가 적혀 있습니다. GitHub에서 남의 프로젝트를 받았을 때 가장 먼저 열어보는 파일입니다.

② npm install — package.json의 부품 목록을 읽고, 필요한 라이브러리를 전부 node_modules 폴더에 다운로드합니다. 이케아 가구의 부품 봉투를 채우는 과정과 같습니다. GitHub에서 clone한 프로젝트가 실행이 안 될 때 90%는 이걸 안 해서입니다 — node_modules는 용량이 커서 Git에 올리지 않기 때문에(.gitignore 처리), 새 컴퓨터에서는 항상 npm install부터 해야 합니다.

③ npm run dev — package.json의 scripts 항목에 적힌 "dev"라는 이름의 명령을 실행하라는 뜻입니다. 즉 "미리 등록해 둔 실행 버튼을 누르는 것"입니다. Next.js 프로젝트라면 dev = "개발용 서버 켜기"로 등록돼 있어 localhost:3000에서 미리보기가 열리고, 코드를 고치면 화면이 자동 갱신됩니다. 끄려면 터미널에서 Ctrl + C. 이름은 약속일 뿐이라 npm run 아무이름도 만들 수 있습니다 — 아래 예제에서는 "dev": "node analyze.js"로 등록해 우리 분석 프로그램을 실행 버튼화합니다.

이제 package.json을 열어 실행 버튼을 등록합니다 (AI에게 "package.json에 dev 스크립트 추가해줘"라고 시켜도 됩니다):

// package.json
{
  "name": "youtube-analyzer",
  "type": "module",                  // 최신 import 문법 사용 선언
  "scripts": {
    "dev": "node analyze.js"         // npm run dev = node analyze.js 실행
  },
  "dependencies": {
    "dotenv": "^16.4.5"              // npm install로 설치된 부품 목록
  }
}

발급받은 두 키를 코드가 아닌 .env 파일에 넣습니다. .env는 "이 컴퓨터에만 존재하는 비밀 금고"입니다.

# .env  (프로젝트 폴더에 새 파일로 생성 — 따옴표 없이)

# .gitignore  (Git이 절대 올리면 안 되는 것들의 목록)
.env
node_modules/
⚠️ 키 보관·배포의 4대 철칙 — 실제 사고가 가장 많은 지점

① .env는 반드시 .gitignore에. 이걸 빠뜨리고 push하면 키가 GitHub에 영구 기록됩니다. 봇들이 공개 저장소를 24시간 스캔하며 키를 수집하므로, 노출된 키는 수 분 안에 도용될 수 있습니다. 실수로 올렸다면 지우는 걸로 끝이 아니라 즉시 해당 콘솔에서 키를 폐기(revoke)하고 재발급해야 합니다 — Git 히스토리에 남아 있기 때문입니다.

② 키는 절대 브라우저(프론트엔드) 코드에 넣지 않기. HTML/JS에 키를 쓰면 방문자 누구나 "소스 보기"로 훔칠 수 있습니다. 키가 필요한 요청은 반드시 서버 쪽(Node 스크립트, 서버리스 함수)에서 실행합니다. 이 예제가 브라우저가 아닌 Node.js로 도는 이유입니다.

③ 배포할 때는 플랫폼의 환경 변수 메뉴에. Vercel이라면 프로젝트 Settings → Environment Variables에 같은 이름(YOUTUBE_API_KEY 등)으로 등록하면, 서버가 .env 대신 그 값을 읽습니다. .env 파일 자체를 업로드하는 것이 아닙니다.

④ 키에 한도와 제한을 걸기. Google 콘솔에서는 API 키를 특정 API만 쓰도록 제한하고, Anthropic/OpenAI 콘솔에서는 월 사용액 한도를 설정해 두세요. 유출돼도 피해가 제한됩니다.

STEP 5 · 코드 작성 — 채널 등록 → 영상 리스트 수집

AI 도구에게 아래 PRD식 지시를 내려도 되고, 코드를 그대로 만들어 원리를 살펴봐도 됩니다. 채널은 유튜브 주소의 @핸들(예: youtube.com/@KBSnews)로 등록합니다.

// youtube.js — 유튜브 채널의 최신 영상 목록 가져오기
import 'dotenv/config';                        // .env의 비밀 값을 불러오는 부품
const KEY = process.env.YOUTUBE_API_KEY;       // 금고에서 키① 꺼내기
const HANDLE = '@KBSnews';                     // ★ 분석할 채널을 여기에 등록

// ① 채널 핸들 → 채널의 '업로드 재생목록 ID' 알아내기
const chRes = await fetch(
  `https://www.googleapis.com/youtube/v3/channels?part=contentDetails,snippet&forHandle=${HANDLE}&key=${KEY}`
);
const chData = await chRes.json();             // 응답을 JSON으로 해석
const uploadsId = chData.items[0].contentDetails.relatedPlaylists.uploads;
console.log('채널명:', chData.items[0].snippet.title);

// ② 그 재생목록에서 최신 영상 10개 꺼내기
const listRes = await fetch(
  `https://www.googleapis.com/youtube/v3/playlistItems?part=snippet&playlistId=${uploadsId}&maxResults=10&key=${KEY}`
);
const listData = await listRes.json();

// ③ 필요한 정보(제목·게시일)만 골라 정리
export const videos = listData.items.map(v => ({
  title: v.snippet.title,
  publishedAt: v.snippet.publishedAt.slice(0, 10)
}));
console.table(videos);                          // 표 형태로 출력

읽는 법: fetch(주소)는 "이 주소로 데이터 주세요"라는 요청이고, 주소 끝에 &key=로 출입증을 첨부합니다. 돌아온 답장은 JSON이라는 문자 형식이라 .json()으로 풀어서 씁니다.

3단계

Git에 올리기 — 버전 관리와 협업의 시작

학습 목표 — Git의 핵심 개념을 이해하고, GitHub에 코드를 올리고, 집·사무실의 서로 다른 PC에서 같은 저장소로 이어서 작업하며, 다른 사람과 프로젝트를 공유한다.

3-1. Git 전반적인 이해 — 게임의 '세이브 포인트'

Git은 코드의 변경 역사를 기록하는 버전 관리 시스템입니다. 게임의 세이브 포인트처럼, 잘 되던 시점을 저장해 두면 AI가 코드를 망가뜨려도 언제든 그 시점으로 되돌아갈 수 있습니다. GitHub는 그 기록을 인터넷에 보관·공유하는 서비스(원격 저장소)입니다. 전문가들은 바이브 코딩에서 Git이 선택이 아닌 필수라고 강조합니다 — "Git을 마스터하지 않고 바이브 코딩을 시도하는 것은 안전벨트 없이 고속도로를 달리는 것과 같다"는 표현이 대표적입니다.

딱 6개만 외우면 되는 Git 명령어

# ── 프로젝트 시작 시 딱 1번 ──
git init                                  # 이 폴더의 버전 관리 시작
git remote add origin [GitHub 저장소 주소]  # 클라우드 보관소 연결

# ── 작업할 때마다 반복하는 3종 세트 ──
git add .                                 # 바뀐 파일 전부 담기
git commit -m "할 일 추가 기능 완성"          # 메모와 함께 세이브
git push                                  # GitHub에 업로드

# ── 다른 PC에서 이어서 할 때 ──
git pull                                  # 최신 버전 내려받기

3-2. GitHub 저장소(Repository) 만들고 내 프로젝트와 연결하기

지금부터는 실제 화면을 그대로 따라가며 저장소를 하나 만들어 봅니다. GitHub는 코드 저장소 자체는 무료로 꽤 넉넉하게 쓸 수 있어서, 개인 개발·포트폴리오·사이드 프로젝트 정도라면 Free 플랜만으로 대부분 충분합니다.

STEP 1 · 화면 우측 상단의 "+" 메뉴에서 New repository 선택

github.com에 로그인한 뒤(가입 안 했다면 우선 무료 가입) 어느 화면에서든 우측 상단의 + 아이콘을 클릭하면 아래처럼 메뉴가 펼쳐집니다.

STEP 2 · 저장소 이름을 정하고 Create repository 클릭

Owner(소유 계정)를 선택하고, Repository name에 짧고 기억하기 쉬운 이름(예: todo-app)을 입력합니다. 비밀 정보가 없는 개인 프로젝트라면 Public을, 회사·개인정보가 섞인 프로젝트라면 Private을 선택하세요. Add a README file을 체크해 두면 저장소가 완전히 빈 상태로 생성되지 않아 이후 단계가 더 매끄럽습니다.

STEP 3 · 내 프로젝트 폴더와 연결하기 (Quick setup 화면 그대로)

저장소가 만들어지면 GitHub가 "이 저장소를 어떻게 채울까요?"라는 안내 화면(Quick setup)을 보여줍니다. 이미 로컬에 프로젝트 폴더가 있다면 "…or push an existing repository from the command line" 항목의 3줄을 그대로 복사해 내 터미널에 붙여넣으면 됩니다.

  1. 내 프로젝트 폴더에서 터미널을 열고, 아직 git init을 안 했다면 먼저 실행
  2. 위 3줄을 그대로 붙여넣기 — 최초 1회만 하면 이후로는 git push만 반복
  3. 새로고침하면 GitHub 저장소 화면에 내 파일들이 그대로 올라와 있는 것을 확인
💡 바이브 코딩이라면 이 과정조차 AI에게 위 STEP 1~3을 사람이 직접 클릭할 필요 없이, 저장소만 미리 만들어 두면 나머지는 AI 도구에게 "지금까지 작업을 커밋하고, https://github.com/내계정/todo-app 에 push해줘"라고 지시할 수 있습니다. Claude Code 같은 CLI 도구는 git remote add부터 git push까지 대신 실행해 줍니다.

3-3. 집과 사무실, 다른 PC에서 같은 저장소로 작업하기

GitHub가 "원본"이 되고 각 PC는 그 사본을 갖는 구조입니다. 규칙은 단 하나 — 앉으면 pull, 일어나면 push.

  1. 새 PC 최초 1회: git clone https://github.com/내계정/todo-app.git — 저장소 전체를 복제
  2. 작업 시작 전(매번): git pull — 다른 PC에서 올린 최신 내용 받기
  3. 작업 종료 시(매번): git add . && git commit -m "메모" && git push
  4. push를 깜빡하고 퇴근했다면? 다음 날 그 PC에서 push 먼저 → 다른 PC에서 pull
⚠️ 충돌(Conflict)이 났어요! push를 잊고 양쪽 PC에서 같은 파일을 고치면 Git이 "어느 쪽이 맞아?"라고 묻는 충돌이 발생합니다. 당황하지 말고 ① VS Code가 보여주는 충돌 표시에서 남길 쪽을 클릭해 선택하거나 ② AI 도구에 "지금 git conflict 났어. 해결해줘"라고 요청하세요. 예방책은 "앉으면 pull, 일어나면 push" 습관 하나면 충분합니다.

3-4. 같은 저장소로 다른 사람들과 프로젝트 공유하기

혼자 쓰던 저장소에 동료를 초대하면 곧바로 협업이 됩니다. 두 가지 수준이 있습니다.

수준 1 · Collaborator 초대

저장소 Settings → Collaborators → Add people로 동료의 GitHub 계정을 초대. 초대받은 사람도 clone/pull/push가 가능해집니다. 소규모 팀·스터디에 적합.

수준 2 · 브랜치 + Pull Request

각자 브랜치(git checkout -b feature/login)에서 작업 후 PR(Pull Request)로 "합쳐 주세요"를 요청 → 동료 검토 후 main에 병합. 실무 표준 방식.

바이브 코딩에서도 브랜치를 활용한 Git 워크플로우는 필수로 꼽힙니다 — AI가 만든 실험적 코드를 브랜치에 격리해 두면, 실패해도 main(안정 버전)은 무사하기 때문입니다. 또한 README.md 파일을 잘 써 두면(프로젝트 소개·설치법·실행법) 저장소 첫 화면에 자동 표시되어 공유 문서 역할을 합니다. 이 역시 AI에게 "이 프로젝트의 README를 작성해줘"라고 맡기면 됩니다.

  • 저장소를 Private으로 만들면 초대한 사람만 볼 수 있다 (회사 프로젝트 권장)
  • 비밀 정보(.env, API 키)는 .gitignore에 등록해 절대 push하지 않는다
  • 커밋 메시지는 "수정" 대신 "무엇을 왜 바꿨는지" 한 줄로 쓴다
  • 큰 기능은 브랜치에서, main은 항상 동작하는 상태로 유지한다

3-5. Git과 GitHub의 추가 기능 — 알아두면 좋은 것들

지금까지 배운 6개 명령어와 협업 방법만으로도 충분히 시작할 수 있지만, 저장소 상단의 탭들에는 프로젝트 관리에 유용한 기능이 더 숨어 있습니다. 아래는 실제 저장소 화면의 탭 구성입니다.

Issues 할 일·버그 게시판

"로그인 버튼이 모바일에서 깨짐" 같은 할 일·버그를 기록하고 번호를 매겨 추적합니다. AI에게 "이 Issue 내용대로 고쳐줘"라고 링크를 붙여주면 맥락을 그대로 이해합니다.

Pull Request 상세

브랜치의 변경 내용을 파일별 diff(빨강=삭제, 초록=추가)로 보여주고, 코멘트·승인(Approve)·병합(Merge)까지 한 화면에서 처리합니다. "AI가 만든 코드를 사람이 검토하는" 안전장치 역할을 합니다.

Releases / Tags 배포 버전 표시

git tag v1.0.0으로 "이 시점이 정식 버전이다"라는 이정표를 남깁니다. Releases 탭에서 버전별 변경 내역(changelog)과 다운로드 파일을 정리해 공개할 수 있습니다.

GitHub Actions 자동화(CI/CD)

push할 때마다 "테스트 자동 실행", "빌드 후 자동 배포" 같은 작업을 대신 해주는 로봇입니다. 4단계의 "push하면 자동 배포"도 사실 Vercel·GitHub Pages 내부에서 Actions와 유사한 원리로 동작합니다.

.gitignore 템플릿

저장소 생성 시 "Add .gitignore" 옵션에서 Node/Python 등 언어를 고르면 node_modules/, .env, __pycache__/ 등 흔히 올리면 안 되는 파일 목록이 자동으로 채워집니다.

Fork & Star

Fork는 남의 저장소를 "내 계정으로 통째로 복사"해 자유롭게 실험하는 기능, Star는 마음에 든 프로젝트를 북마크하는 기능입니다. 오픈소스를 참고할 때 가장 먼저 배우는 두 버튼입니다.

4단계

로컬을 넘어 세상으로 — 웹 배포 (Deployment)

학습 목표 — Vercel·GitHub Pages·Netlify·Render 등 배포 플랫폼의 특징과 가격을 비교해 내 프로젝트에 맞는 곳을 고르고, Git 연동으로 배포·관리·업데이트하는 방법을 익힌다.

배포(Deploy)란 내 컴퓨터(localhost)에만 있던 웹앱을 인터넷 서버에 올려 전 세계 누구나 주소로 접속할 수 있게 만드는 일입니다. 과거에는 서버를 직접 사서 관리해야 했지만, 지금은 GitHub에 push만 하면 자동으로 배포되는 플랫폼들이 있어 초보자도 5분이면 사이트를 공개할 수 있습니다.

4-1. 배포 플랫폼 5종 — 특징과 가격 비교

Vercel — Next.js의 본가

Next.js 개발사가 직접 만든 플랫폼으로 React·프론트엔드 프레임워크에 최적화. push마다 자동 배포 + PR마다 미리보기 URL 제공. 서울에 CDN이 있어 한국 접속 시 TTFB 30~40ms 수준의 최상급 속도를 보여 국내 사용자에게 특히 유리합니다.

Hobby 무료 · Pro $20/월

GitHub Pages — 완전 무료 정적 호스팅

깃허브 저장소의 정적 파일(HTML/CSS/JS)을 무료로 호스팅. 포트폴리오·문서·블로그에 이상적. 공개 저장소라면 대역폭 제한 없이 완전 무료이지만, 서버 사이드 기능은 지원하지 않습니다.

Public 저장소 무료 (Private Pages는 Pro 필요)

Netlify — 정적 사이트 만능형

직관적 UI와 자동 Git 연동, 폼(Form) 처리 등 정적 사이트용 편의 기능 내장. 무료 플랜에서도 상업적 사용이 허용된다는 점이 Vercel과의 가장 큰 실무 차이입니다. 무료 빌드 시간은 월 300분으로 제한됩니다.

Free 무료 · Pro 약 $19/월

Render — 백엔드까지 한 곳에서

정적 사이트뿐 아니라 Flask·Node.js 같은 백엔드 API, PostgreSQL 데이터베이스, 백그라운드 작업, 크론 잡까지 하나의 대시보드에서 배포 가능한 풀스택 클라우드. 무료 웹 서비스는 일정 시간 미사용 시 잠들었다 깨어나는(콜드 스타트) 특성이 있습니다.

정적 사이트 무료 · 웹 서비스 무료 티어 / 유료 $7/월~

Cloudflare Pages — 무료 한도의 최강자

글로벌 CDN 기업 Cloudflare의 호스팅. 무료 플랜의 대역폭이 사실상 무제한급이라 Vercel 무료 100GB 한도에 가까워진 사용자들이 이주하는 대표 목적지로 꼽힙니다. Workers(서버리스)·D1(DB) 등 생태계 확장성도 좋습니다.

Free 무료 · Paid $5/월~

👨‍💻 전문가 의견 — 어떤 기준으로 고르나 플랫폼 비교 분석의 결론은 명확합니다. "Next.js로 만든다면 Vercel — React Server Components, 스트리밍, ISR에 대한 프레임워크 수준 지원은 다른 플랫폼이 따라오지 못한다. 다만 무료 티어의 비상업 제한을 유의하라. 올인원 배포 플랫폼과 Git 기반 CI/CD, 미리보기 배포, 내장 폼을 원하면 Netlify를 유지하라"는 것. 그리고 "단순 정적 사이트라면 GitHub Pages가 비용 면에서 이기기 어렵고, 풀스택 앱에는 Render가 최적"이라는 평가가 일반적입니다. 요약하면 정적 = GitHub Pages, React/Next = Vercel, 폼·상업용 정적 = Netlify, 백엔드+DB = Render입니다.

4-2. PRD에서 정한 기술 스택이 배포 플랫폼을 결정한다

2단계(2-1)에서 PRD를 작성할 때 "기술 스택" 항목을 채웠던 것을 기억하시나요? 그 결정은 사실 4단계에서 어느 배포 플랫폼을 골라야 하는지를 이미 절반쯤 정해 놓은 것입니다. 기획 단계의 선택과 배포 단계의 선택은 서로 분리된 일이 아니라 하나로 이어진 파이프라인입니다.

즉 PRD를 쓸 때 "이번 버전 기술 스택: 프레임워크 없이 HTML/CSS/JS 단일 파일"이라고 적었다면 4단계에서 별다른 고민 없이 GitHub Pages로 직행하면 됩니다. 반대로 "React 컴포넌트 구조로 만들고, 나중에 로그인 기능도 넣을 예정"이라고 적었다면 처음부터 Vercel(그리고 5단계에서 배울 DB 연동)을 염두에 두고 개발하는 편이 훨씬 매끄럽습니다. PRD를 만든 뒤 기능을 작고 완결된 단위로 점진적으로 구현하라는 2단계의 조언도, 결국 "이 프로젝트가 마지막에 어떤 방식으로 배포될지"를 염두에 두고 스택을 고를 때 가장 잘 작동합니다.

👨‍💻 실전 팁 — PRD의 "기술 스택" 칸에 이렇게 적으면 4단계가 쉬워집니다 ① 정적 페이지·포트폴리오·간단한 도구 → PRD에 "HTML/CSS/JS 단일 파일, 빌드 과정 없음"이라고 명시 → 4-4절 GitHub Pages로 직행. ② 화면 전환이 많고 재사용 컴포넌트가 필요한 서비스 → PRD에 "React 또는 Next.js 사용"이라고 명시 → 4-3절 Vercel로 직행. ③ 회원가입·내 데이터 저장이 필요한 서비스 → PRD에 "Next.js + Supabase(DB) 사용"이라고 명시 → Vercel 배포 + 5단계 DB 연동을 함께 준비. AI 코딩 도구에게 PRD를 전달할 때 이 기술 스택 문장을 그대로 포함시키면, 처음부터 배포하기 쉬운 구조로 코드를 짜 줍니다.

4-3. Vercel로 배포하기 — 실제 화면으로 따라가는 상세 가이드

Vercel은 2026년 기준 서울 리전 CDN을 갖춰 국내 접속 속도가 가장 빠른 축에 속하고, GitHub와의 연동이 가장 매끄러운 플랫폼입니다. 최근에는 Claude Code·Cursor 같은 AI 에이전트가 터미널에서 바로 배포·환경 변수 설정까지 수행할 수 있는 전용 플러그인(vercel-plugin)도 제공합니다. 아래는 웹 대시보드로 처음 배포하는 전체 과정입니다.

STEP 1 · vercel.com 가입 — GitHub 계정으로 로그인

별도 회원가입 양식 없이 "Continue with GitHub" 버튼 하나로 가입이 끝납니다. GitHub 계정과 연동해 가입하면 이후 내 저장소 목록이 자동으로 대시보드에 나타납니다.

STEP 2 · Add New → Project — 내 저장소 Import

대시보드 우측의 Add New → Project를 누르면 GitHub에 연결된 내 저장소 목록이 뜹니다. 3단계에서 push해 둔 저장소(todo-app) 옆의 Import 버튼을 클릭합니다.

STEP 3 · 설정은 그대로 두고 Deploy

Vercel은 프레임워크를 자동 감지합니다 — Next.js면 Next.js용 빌드 설정을, 순수 HTML이면 정적 사이트 설정을 알아서 채웁니다. 특별한 이유가 없다면 Project Name·Framework Preset은 그대로 두고 Deploy 버튼만 누릅니다.

STEP 4 · 완성 — 배포 대시보드와 실주소 확인

1~2분 뒤 빌드가 끝나면 아래처럼 프로젝트 대시보드가 나타나고, https://todo-app-내계정.vercel.app 형태의 주소가 발급됩니다.

STEP 5 · 환경 변수 등록 (API 키가 있다면 필수)

2-4절에서 만든 .env 파일의 키들은 배포 서버에는 자동으로 올라가지 않습니다. Settings → Environment Variables에서 같은 이름으로 직접 등록해야 합니다.

5단계

내 데이터는 어디에 저장할까 — 로컬 저장 vs 실제 DB

학습 목표 — 사용자의 데이터를 다루는 웹앱을 만들 때, 브라우저에만 저장하는 방식(localStorage)과 실제 데이터베이스(DB)에 저장하는 방식의 차이를 이해하고, 대표적인 DB 서비스 Supabase를 실제로 만들어 배포 사이트(Vercel)와 연동하는 전체 과정을 익힌다.

2단계에서 만든 할 일 관리 앱을 다시 떠올려 보세요. "새로고침해도 유지"되는 이유는 localStorage라는 브라우저 안의 저장 공간에 데이터를 넣어 두었기 때문입니다. 그런데 이 앱을 배포해서 여러 사람이 쓰거나, 내가 폰과 노트북을 오가며 같은 데이터를 보고 싶다면 이야기가 달라집니다. 이번 단계에서는 "어디에 저장할 것인가"라는, 데이터를 다루는 모든 서비스가 반드시 결정해야 하는 질문을 다룹니다.

5-1. 브라우저에 로컬로 저장하기 — localStorage와 그 한계

브라우저는 자체적으로 데이터를 저장할 수 있는 3가지 공간을 제공합니다. 서버도, 회원가입도, API 키도 필요 없이 JavaScript 한 줄로 바로 쓸 수 있어 입문 단계의 첫 프로젝트에 가장 적합합니다.

저장 공간용량유지 기간특징
localStorage약 5~10MB브라우저 캐시를 지우기 전까지 영구이름=값(문자열) 형태로 저장. 가장 많이 쓰는 방식. 2단계 할 일 앱이 이 방식
sessionStorage약 5~10MB탭을 닫으면 즉시 삭제"임시 장바구니"처럼 그 방문 동안만 기억하면 되는 데이터에 사용
IndexedDB수백MB 이상브라우저 캐시를 지우기 전까지 영구표 형태의 대용량 데이터·파일을 저장할 때 사용. localStorage보다 복잡하지만 훨씬 큰 데이터를 다룸
 
// localStorage 사용법 — 딱 3개 함수만 알면 됩니다
localStorage.setItem('todos', JSON.stringify(myTodoList));  // 저장하기
const saved = JSON.parse(localStorage.getItem('todos'));    // 불러오기
localStorage.removeItem('todos');                            // 지우기

문제는 이 데이터가 "이 브라우저, 이 컴퓨터"에만 존재한다는 점입니다. 아래 진단표로 내 프로젝트에 로컬 저장만으로 충분한지 확인해 보세요.

  • 나 혼자만 쓰는 도구, 회원가입이 없는 단순 계산기·메모장 → localStorage로 충분
  • 여러 기기(폰·PC)에서 같은 데이터를 보고 싶다 → DB 필요
  • 다른 사람과 데이터를 공유하거나, 회원가입/로그인이 있다 → DB 필요
  • 서버에서 데이터를 검증·집계(예: "가장 인기있는 항목 Top 10")해야 한다 → DB 필요
  • 브라우저 캐시 삭제·기기 변경으로 데이터가 사라지면 곤란하다 → DB 필요

5-2. 실제 데이터베이스(DB)가 필요한 순간 — 왜, 그리고 무엇을 쓰나

데이터베이스(DB)는 인터넷 어딘가의 서버에서 24시간 돌아가는 "데이터 전용 저장 컴퓨터"입니다. 브라우저가 아니라 서버에 있기 때문에, 어떤 기기에서 접속하든 API를 통해 같은 데이터를 읽고 씁니다. 초보자 입장에서 DB 선택지는 크게 아래와 같이 정리됩니다.

서비스DB 종류무료 한도(요약)Vercel과의 연동이런 사람에게
Supabase 이 교재의 선택PostgreSQL(관계형)프로젝트 2개, 500MB DBVercel Marketplace 공식 연동회원가입·로그인까지 한 번에 해결하고 싶은 입문자
FirebaseFirestore(NoSQL, 문서형)넉넉한 무료 한도, 실시간 동기화별도 SDK로 직접 연동실시간 채팅·협업 도구처럼 "즉시 반영"이 중요한 서비스
Vercel Postgres / NeonPostgreSQL소용량 무료Vercel Marketplace 공식 연동Vercel 생태계 안에서 최대한 단순하게 가고 싶을 때

이 교재는 Supabase를 대표로 다룹니다. 이유는 세 가지입니다. ① PostgreSQL 기반이라 "표(테이블)에 행과 열로 데이터를 저장한다"는, DB 중 가장 표준적이고 배우기 쉬운 구조를 씁니다. ② DB뿐 아니라 회원가입·로그인(Auth), 파일 저장(Storage)까지 한 서비스 안에서 해결되어 초보자가 여러 서비스를 따로 배울 필요가 없습니다. ③ 아래에서 보듯 Vercel Marketplace에 "Vercel Native" 등급으로 공식 등록되어 있어, 대시보드 클릭 몇 번으로 배포 사이트와 바로 연결됩니다.

5-3. Supabase 실전 연동 — 프로젝트 생성부터 배포 사이트 연결까지

STEP 1 · Supabase 프로젝트 만들기

supabase.com에서 GitHub 계정으로 가입 후 New Project를 누르고, 프로젝트 이름과 DB 비밀번호(따로 안전하게 보관), 리전(가까운 곳으로 예: Northeast Asia — Seoul)을 선택합니다.

STEP 2 · Table Editor에서 테이블 만들기

왼쪽 메뉴의 Table Editor에서 New table을 눌러 표를 하나 만듭니다. 마우스 클릭만으로 열(컬럼)과 데이터 타입을 정의할 수 있어, SQL 문법을 몰라도 시작할 수 있습니다.

STEP 3 · API 키 확인하기 (Project URL + anon key)

Settings → API에서 두 값을 복사합니다. 이 둘은 2-4절의 API 키와 같은 역할이지만, Supabase는 anon(공개용) 키와 service_role(관리자용) 키를 구분합니다 — 브라우저 코드에는 반드시 anon 키만 사용해야 합니다.

// .env — 2-4절과 동일한 방식으로 보관
NEXT_PUBLIC_SUPABASE_URL=https://todoappdb.supabase.co
NEXT_PUBLIC_SUPABASE_ANON_KEY=eyJhbGciOi...

// 코드에서 사용 (supabase-js 라이브러리)
import { createClient } from '@supabase/supabase-js';
const supabase = createClient(
  process.env.NEXT_PUBLIC_SUPABASE_URL,
  process.env.NEXT_PUBLIC_SUPABASE_ANON_KEY
);

// 저장하기 — localStorage.setItem 대신
await supabase.from('todos').insert({ title: '장보기', done: false });

// 불러오기 — localStorage.getItem 대신
const { data } = await supabase.from('todos').select('*');

STEP 4 · Vercel Marketplace에서 Supabase를 배포 사이트와 연결하기

가장 중요한 단계입니다. Supabase를 Vercel Storage 탭에서 직접 설치하면, 위에서 복사해 두었던 URL·키를 손으로 옮길 필요 없이 환경 변수가 자동으로 배포 프로젝트에 등록됩니다.

  1. 배포한 Vercel 프로젝트 대시보드에서 Storage 탭 → Browse Marketplace → Supabase 선택
  2. Install 클릭 → 기존에 만든 Supabase 프로젝트를 연결하거나, 여기서 새로 생성
  3. 연결이 끝나면 NEXT_PUBLIC_SUPABASE_URL 등의 환경 변수가 Vercel Environment Variables에 자동으로 채워짐 — 4-3절 STEP 5처럼 손으로 입력할 필요가 없음
  4. 이후 Supabase에서 테이블 구조를 바꾸면, 다음 git push 재배포 시 자동 반영

STEP 5 · 보안의 핵심 — Row Level Security(RLS)

Supabase 테이블은 기본적으로 RLS가 켜져 있으면 아무도 접근할 수 없습니다. "이 행은 로그인한 본인만 읽고 쓸 수 있다" 같은 Row Level Security(RLS) 정책을 직접 정의해야 안전하게 열립니다. AI 도구에게 "Supabase todos 테이블에 '로그인한 사용자는 자기 데이터만 읽고 쓸 수 있다'는 RLS 정책을 만들어줘"라고 요청하면 SQL을 대신 작성해 줍니다.

⚠️ DB를 쓸 때 실제 사고가 가장 많이 나는 지점

① service_role 키는 절대 브라우저(프론트엔드) 코드에 넣지 않습니다. 이 키는 RLS를 무시하고 모든 데이터에 접근할 수 있는 관리자 키라, 노출되면 DB 전체가 위험해집니다. 서버 쪽(API Route, 서버리스 함수)에서만 사용하세요.

② RLS를 끄고 개발하다가 그대로 배포하지 않습니다. "일단 테스트니까 RLS 끄고 하자"는 습관이 실제 서비스에서 데이터 유출 사고로 이어진 사례가 많습니다. 배포 전 반드시 RLS 정책을 켜고 테스트하세요.

③ DB 비밀번호·연결 문자열(connection string)도 API 키와 동일하게 .env·환경 변수로만 관리합니다.

중급자로 가는 길 — 기술 로드맵과 학습 자원

학습 목표 — 바이브 코딩을 넘어 "AI가 만든 코드를 이해하고 통제하는" 중급자가 되기 위해 배워야 할 웹개발 기술과 신뢰할 수 있는 학습 사이트를 파악한다.

바이브 코딩의 역설: AI가 코드를 다 짜 주는데 왜 공부해야 할까요? 전문가들은 "코딩(문법을 치는 행위)은 이미 LLM이 숙련자보다 빠르지만, 문제를 정의하고 최적의 해결책을 설계하며 유지보수 가능한 시스템을 만드는 '프로그래밍'은 여전히 사람의 몫이며, 개발자는 제품 관리자처럼 생각해야 한다"고 말합니다. 즉 중급자의 목표는 코드를 외우는 것이 아니라 AI의 결과물을 읽고, 평가하고, 방향을 제시하는 능력입니다.

6-1. 배워야 할 웹개발 기술 — 4단계 로드맵

이 순서는 실제 시장 데이터에 근거합니다. Stack Overflow의 2025년 전 세계 49,000명 이상 개발자 조사에서 JavaScript는 사용률 1위(66%)를 유지했고, TypeScript는 44%까지 올라섰으며, React는 프론트엔드 프레임워크 중 독보적인 점유율을 기록하고 있습니다. 특히 2026년 기준 대부분의 기업 프로젝트가 TypeScript로 작성되므로 Lv.2에서 반드시 익혀야 합니다.

Lv.1HTML / CSS(Flexbox·Grid) / JavaScript 기본AI가 만든 화면 코드를 "읽을" 수 있게 됨. 이 셋을 건너뛰고 프레임워크부터 배우는 것은 기초 공사 없이 건물을 올리는 것과 같다는 것이 로드맵 전문가들의 공통 의견자기소개 페이지를 AI 없이 직접 → AI로 리팩토링 비교

Lv.3Next.js, REST API, 데이터베이스(SQL·Supabase 등), 인증(로그인)"데이터가 저장되는 진짜 서비스"의 필수 요소. Git·배포(3~4단계)와 결합해 실사용 가능한 앱 완성회원가입·DB 저장이 되는 미니 서비스 배포
Lv.4테스트(Jest 등), 웹 보안 기초(XSS·키 관리), CI/CD(GitHub Actions), 성능·모니터링바이브 코딩 사고 사례(키 노출·DB 삭제)의 재발 방지책. AI 결과물을 "검증"하는 눈을 갖추는 단계내 서비스에 자동 테스트 + 배포 파이프라인 구축

6-2. 참고할 만한 웹사이트와 공부 방법

웹 개발의 사실상 표준 문서. HTML·CSS·JavaScript·Web API 등 웹 기술 전반을 기초부터 심화까지 상세히 다루며, 초보자를 위한 학습(Learn) 섹션이 한국어로도 제공됩니다. 모르는 문법은 항상 여기서 확인하는 습관을.

브라우저 안에서 바로 코드를 쳐 보며 배우는 무료 실습형 커리큘럼. W3Schools는 HTML·CSS·JavaScript·SQL·Python 등을 풍부한 예제와 함께 정리한 튜토리얼 사이트로 빠른 문법 확인에 유용합니다.

국내 입문자 대상 대표 무료·유료 강의 채널. 생활코딩의 HTML 강의는 웹을 전혀 몰라도 차근차근 따라가며 처음 코딩을 접하기 좋은 강의로 꼽히며, 코딩애플은 첫날부터 웹을 만들며 배우는 프로젝트 중심 커리큘럼을 표방합니다.

ko.javascript.info

'코어 자바스크립트'로 불리는 무료 한국어 교재. JavaScript의 동작 원리를 깊게 이해하고 싶을 때 정독할 만한 표준 자료입니다.

👨‍💻 전문가 의견 — 중급자로 가는 공부법 ① "완벽하게 배운 후 시작하겠다는 생각은 영원히 시작하지 않겠다는 말과 같다. 오늘 HTML 파일 하나를 만들어 보는 것이 어떤 강의를 고를지 고민하는 시간보다 훨씬 가치 있다" — 로드맵보다 실행이 먼저입니다. ② "실제로 코드를 치고, 에러를 마주하고, 해결하는 과정의 반복이 실력을 만든다"는 점에서, AI에게 코드를 받은 뒤 "이 코드를 한 줄씩 설명해줘", "왜 이 방식을 썼어? 다른 방법은?"이라고 되묻는 습관이 최고의 개인 과외가 됩니다. ③ 마인드셋 전환도 중요합니다 — "AI가 나를 위해 무엇을 해줄 수 있지?"가 아니라 "내가 AI를 위해 무엇을 제공할 수 있지?"를 물어야 하며, AI에게는 명확한 맥락과 구체적인 요구사항이 필요합니다.

이 교재를 마친 뒤 추천하는 다음 3개 프로젝트

  1. 나만의 포트폴리오 사이트 — Lv.1 기술 + GitHub Pages 배포. 이 교재의 1~4단계를 그대로 한 번 더 복습
  2. 업무 자동화 도구 — 반복하는 업무 하나(파일 정리, 보고서 생성 등)를 Python으로. PRD 작성 → CLI 도구 실습
  3. 데이터가 저장되는 미니 서비스 — Next.js + Supabase + Vercel. 5단계에서 익힌 DB 연동을 실제 프로젝트로 확장하며 Lv.3로 넘어가는 관문 프로젝트

✅ 6단계를 모두 마쳤다면, 이제 이런 것을 할 수 있습니다

  • PRD 없이 무작정 코딩을 맡기지 않고, 목적·범위·기술 스택을 먼저 문서로 정리한 뒤 AI에게 지시할 수 있다 (2단계)
  • AI가 만든 코드를 로컬에서 직접 실행·테스트하고, 외부 API와 LLM API를 연결한 프로그램을 완성할 수 있다 (2단계)
  • GitHub 저장소를 직접 만들고, 6개 핵심 명령어로 커밋·푸시·풀을 반복하며 여러 PC를 오가며 작업할 수 있다 (3단계)
  • Issues·Pull Request·브랜치를 활용해 AI의 실험적 코드를 안전하게 격리하고 검토할 수 있다 (3단계)
  • PRD의 기술 스택 결정에 맞춰 Vercel 또는 GitHub Pages 중 알맞은 플랫폼을 골라 실제 주소로 배포할 수 있다 (4단계)
  • localStorage와 실제 DB(Supabase)의 차이를 이해하고, 필요에 따라 데이터 저장 방식을 직접 설계·전환할 수 있다 (5단계)
  • API 키·환경 변수·RLS 같은 보안 기본기를 지키며, "AI에게 맡기되 사람이 검토하는" 바이브 코딩 습관을 유지할 수 있다 (전 단계 공통)
  • 다음 목표를 스스로 정할 수 있다 — 6단계 로드맵 중 지금 내가 어디에 있는지, 다음에 무엇을 배워야 하는지 안다 (6단계)
