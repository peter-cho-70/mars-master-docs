1장

작업 강도에 맞는 모델·도구 운용

학습 목표 — 모든 작업에 가장 비싼 모델을 쓰는 습관에서 벗어나, 작업의 난이도·위험도에 따라 모델과 기능을 선택하는 기준을 세운다.

비기너 단계에서는 "어떤 모델이 좋은가"를 묻지만, 중급자는 "이 작업에 어떤 모델이면 충분한가"를 묻습니다. 오타 수정에 최상위 모델을 쓰는 것은 택시로 옆집에 가는 것과 같습니다 — 되긴 하지만, 사용량 한도와 비용이 실력의 일부인 중급자에게는 낭비입니다.

작업 강도예시권장
가벼움오타·문구 수정, 한 줄 스타일 변경, 단순 질문경량 모델 (Haiku급) / 자동 선택
보통기능 하나 추가, 버그 하나 수정, 리팩토링중간 모델 (Sonnet급) — 일상 작업의 기본값
무거움아키텍처 설계, 다중 파일 대규모 변경, 까다로운 디버깅최상위 모델 (Opus급) + 확장 사고(thinking)
💡 저자 의견

실전에서 더 중요한 건 모델 등급보다 "막혔을 때 모델을 올리는" 습관입니다. 중간 모델로 두 번 시도해서 같은 자리에서 맴돌면, 세 번째 시도를 반복하지 말고 (1) 대화를 새로 시작해 문제를 더 좁혀 설명하거나 (2) 상위 모델로 올려서 분석만 시키세요. "왜 안 되는지 원인만 찾아줘, 고치지는 마"라는 분업이 의외로 강력합니다 — 비싼 모델은 진단에, 중간 모델은 수술에 쓰는 겁니다.

📎 딸림 문서: 작업 강도별 모델 선택 가이드 (Cursor & Claude) Cursor와 Claude Code CLI 각각에서의 모델 선택, 기능 켜고 끄기, 진행 중 작업 이어가기, 미리 심어두는 설정까지 — 도구별 실전 세팅은 이 문서를 참고하세요.
2장

클로드 200% — 스킬, 확장 기능, 워크플로우

학습 목표 — "매번 다시 설명하는" 낭비를 없애는 장치(CLAUDE.md, 스킬, 서브에이전트, 훅 등)를 이해하고, 내 반복 작업 하나를 실제로 스킬로 만든다.

중급자의 핵심 전환은 AI를 '지켜보는' 사람에서 '지휘하는' 사람으로 바뀌는 것입니다. 지휘자는 단원에게 매번 악보를 다시 설명하지 않습니다 — 악보(스킬)와 연습 규칙(CLAUDE.md)을 미리 깔아둡니다.

핵심 도구 6가지 — 각각 "어떤 반복"을 없애는가

도구없애는 반복한 줄 요약
CLAUDE.md프로젝트 규칙을 매번 설명저장소에 두는 상시 지침서 — 코딩 컨벤션, 금지사항, 자주 쓰는 명령
스킬(Skills)특정 작업 절차를 매번 설명SKILL.md 폴더 하나로 전문 매뉴얼을 쥐여줌. 필요할 때만 로드(점진적 공개)
슬래시 명령자주 치는 긴 프롬프트/명령 한 번으로 정해진 프롬프트 실행
서브에이전트긴 탐색으로 컨텍스트 오염검색·리뷰 같은 부업무를 별도 에이전트에 위임, 결과만 받아옴
훅(Hooks)"끝나면 꼭 ~해줘" 부탁도구 실행 전후에 자동으로 걸리는 스크립트 — 포맷터, 테스트, 알림
MCP외부 도구 수동 복붙DB·브라우저·사내 시스템을 AI가 직접 조작하게 연결하는 표준

좋은 스킬의 4원칙 (요약)

  • SKILL.md는 짧게 — 긴 내용은 references/로 빼서 필요할 때만 읽히게.
  • description은 '방아쇠' — "무엇을 한다"가 아니라 "언제 나를 불러야 하는가"를 적는다.
  • Gotchas(자주 실패하는 지점) 섹션 필수 — 정상 절차보다 신뢰도에 더 기여한다.
  • 정밀 반복 작업은 스크립트로 — 매번 똑같아야 하는 일은 코드로 고정.
💡 저자 의견

여섯 도구를 한꺼번에 도입하려 하지 마세요. 제가 권하는 순서는 ① CLAUDE.md → ② 스킬 1개 → ③ 나머지는 필요가 생길 때입니다. CLAUDE.md는 10분 투자로 즉시 효과가 나고, 스킬은 "하루 한 번 이상 하는 일"이 생겼을 때 그 일 하나로 시작하면 됩니다. 훅과 MCP는 강력하지만, 무엇이 반복되는지 스스로 알기 전에 깔면 관리 부담만 늘어납니다. 도구가 아니라 반복이 먼저 보여야 합니다.

📎 원본 상세: 통합본 7장 — 클로드를 200% 활용하는 효율적 코딩 스킬 작성 최소 예시, 점진적 공개 3단계 구조, 6가지 확장 기능 각각의 설정법, 프로처럼 한 사이클 도는 워크플로우의 전체 상세는 분권 전 통합본 7장에 있습니다.
3장 ★ 이 교재의 척추

실전 사례 연구 — 방송 자동화 시스템 3주의 기록

학습 목표 — 실제 프로젝트에서 반복된 실패 패턴 6가지를 사례로 익혀, 같은 함정을 내 프로젝트에서 미리 피한다.

이 장의 사례는 전부 실화입니다. 뉴스 큐시트 제작 앱(MARS)과 방송 진행 앱(Master), 그리고 vMix·OBS·Blackmagic 라우터 같은 실제 방송 장비를 제어하는 시스템을 약 3주간 바이브 코딩으로 구축하면서, 9번의 커밋과 30여 건의 세션 기록에 남은 것들입니다. 교과서적 조언이 아니라, 실제로 방송 중에 터졌던 문제들입니다.

교훈 1 — "AI가 통과시킨 테스트"와 "실제로 되는 것"은 다르다

모의(mock) 테스트는 전부 통과했는데, 실제 장비에 붙이자 6가지가 연달아 깨졌다.
영상 끝 여유가 이론값(5프레임)으로는 매번 실패하고 실측 15프레임이 필요했고, 한국어판 OBS에서는 'Cut'이라는 영문 이름이 아예 매칭되지 않았으며, vMix는 잘못된 명령에도 "Success"를 반환하면서 아무것도 하지 않는 경우가 있었다.
중급자의 규칙: 외부 시스템(장비·API·결제·메일)에 의존하는 기능은 "mock 통과"를 완료로 치지 마세요. 실물 검증을 완료 조건에 처음부터 포함하고, AI에게도 "실제로 호출해서 결과를 읽어와 확인해줘"라고 요구하세요. 로그의 '성공'이 아니라 결과 상태를 직접 조회한 값이 근거입니다.

교훈 2 — 같은 증상 ≠ 같은 원인

"서버 1·2번만 깨지고 3번은 정상"이라는 동일한 증상이 4번 보고됐고, 그때마다 "이번엔 진짜 고쳤다"고 선언됐다.
서로 완전히 다른 원인 3개가 같은 증상을 만들고 있었다 — ① 빠른 연속 입력의 경쟁 상태, ② 리셋되지 않은 전역 추적 변수, ③ 빈 설정값이 조용히 1번 서버로 대체되는 폴백. 하나를 고쳐도 나머지가 같은 얼굴로 나타났다.
중급자의 규칙: 고쳤던 문제가 재발하면 "수정이 풀렸나"가 아니라 "다른 원인이 같은 증상을 내는 건 아닌가"부터 의심하세요. AI에게 시킬 때도 "지난번과 같은 버그야"라고 단정해서 주지 말고, 증상만 주고 원인 후보를 복수로 찾게 하세요.

교훈 3 — 조용한 기본값(fallback)은 미래의 부채다

"디졸브로 설정했는데 컷으로만 나간다"는 문제가 하루 사이 세 번 "완료" 선언되고도 실제 방송 중 재발했다.
값이 없으면 조용히 'Cut'으로 대체하는 코드(x || 'Cut')가 한 곳이 아니라 8곳 이상에 흩어져 있었다. AI는 발견된 그 자리만 고치고 "완료"라고 답했고, 같은 모양의 코드가 다른 경로에서 계속 발동했다.
중급자의 규칙: 폴백 버그를 하나 고치면 그 자리에서 끝내지 말고, AI에게 "같은 패턴을 프로젝트 전체에서 찾아줘"라고 바로 시키세요 (grep 한 번이면 됩니다). 그리고 새 코드를 시킬 때 "값이 없으면 조용히 기본값 쓰지 말고, 로그를 남기거나 명시적으로 실패하게 해줘"라고 요구하세요. 조용한 성공이 시끄러운 실패보다 비쌉니다.

교훈 4 — 검증 루프 자체에도 함정이 있다

"고쳤는데도 여전히 안 된다" — 그런데 사실 고쳐져 있었다.
브라우저가 수정 전 파일을 캐싱해서, 수정 후 검증이 옛날 코드를 테스트하고 있었다(이 프로젝트에서만 2번 발생). 별개로, 테스트 중 UI를 클릭한 것이 자동저장을 건드려 실제 사용자 데이터를 오염시킨 사고도 2번 있었다.
중급자의 규칙: ① 파일 수정 후 첫 재검증은 강력 새로고침 또는 ?v=2 캐시버스팅을 습관으로. ② UI를 통해 테스트했다면 화면만 보지 말고 저장소(localStorage·DB)에 테스트 잔여물이 남았는지까지 확인. ③ 앱 헤더에 빌드 시각을 표시해두면 "지금 뜬 게 최신인가"를 1초 만에 판별할 수 있습니다.

교훈 5 — 계산은 한 곳에만 두게 하라 (단일 소스 오브 트루스)

미리보기가 보여주는 자막 타이밍과 실제 방송에서 나가는 타이밍이 달라질 뻔했다.
처음엔 "실제 실행"과 "미리보기"가 각자 타이밍을 계산했다. 공식을 computeCGPlan() 하나로 뽑아내 양쪽이 같은 함수를 읽게 바꾼 뒤로는, 이 계열의 "미리보기와 실제가 다르다" 버그가 한 번도 발생하지 않았다.
중급자의 규칙: 같은 값을 두 군데서 계산하고 있다면 AI에게 "이 공식을 함수 하나로 추출해서 양쪽 다 그걸 쓰게 해줘"라고 시키세요. AI는 요청받은 화면만 만들며 계산을 복사하는 경향이 있어서, 사람이 이 지점을 지적해줘야 합니다. 복사된 공식은 반드시 어긋납니다 — 오늘이 아니면 다음 달에.

교훈 6 — 문서화는 정리가 아니라 속도다

"vMix의 컷 명령이 PGM·PVW를 맞바꾼다"는 사실을 3주 동안 세 번 새로 발견했다.
매번 다른 기능을 만들다 같은 벽에 부딪혔고, 이전 발견이 기록되지 않아 매번 처음부터 실측해야 했다. 반대로, 실제 사고들을 "증상→원인→조치"로 적은 문제해결 문서를 만든 뒤로는 같은 증상이 와도 몇 분 안에 진단이 끝났다.
중급자의 규칙: 외부 시스템의 반직관적 동작을 발견한 순간이 문서화의 타이밍입니다. CLAUDE.md나 프로젝트 문서에 한 줄 적어두면, 다음 세션의 AI가 그걸 읽고 같은 함정을 피합니다. AI는 세션이 끝나면 잊지만, 저장소의 문서는 잊지 않습니다 — 문서가 곧 AI의 장기기억입니다.
💡 저자 의견 — 이 장이 중급자 교재의 척추인 이유

여섯 교훈의 공통분모는 하나입니다: 중급자의 실력은 프롬프트를 잘 쓰는 능력이 아니라, AI의 "완료"를 검증하는 능력이라는 것. 이 프로젝트에서 재작업의 대부분은 "몰라서"가 아니라 "한 번 알아낸 것을 전체에 적용하지 않아서" 생겼습니다. 비기너와 중급자를 가르는 질문은 "만들 수 있나요?"가 아니라 "그게 정말 되는지 어떻게 알죠?"입니다.

4장

프로덕션 하드닝 — 프로토타입을 상품으로

학습 목표 — "동작하는 데모"와 "팔 수 있는 상품" 사이의 네 기둥(보안·성능·안정성·유지보수성)을 점검 목록으로 익힌다.

AI는 "요청한 기능"은 잘 만들지만 "실서비스에서 버티는 법"은 시키지 않으면 챙기지 않습니다. 반복 프롬프트를 거치며 인증 로직이 조용히 약해지고, UI를 바꾼 뒤 옛 API 엔드포인트가 열린 채 남고, 비밀 키가 클라이언트 코드로 새는 일이 실제로 관찰됩니다.

① 보안 — 가장 먼저

  • 모든 사용자 입력을 검증·정화 (주입 공격 방어)
  • 비밀 키는 환경 변수로 — 코드 하드코딩 금지, 클라이언트 노출 금지 (NEXT_PUBLIC_ 접두사 오남용 주의)
  • npm audit 정기 점검 — AI가 취약한 라이브러리를 고를 수 있음
  • 인증(로그인)과 권한(이 사용자가 이 데이터를 봐도 되는가)은 별개 — 모든 엔드포인트에서 권한 검사, Supabase면 RLS
  • 속도 제한(rate limiting)으로 남용·비용 폭탄 차단
  • AI 코드를 검토 없이 프로덕션에 올리지 않기 — /security-review 활용 + 사람 검토 병행

② 성능 — Core Web Vitals

LCP(로딩) 2.5초, INP(반응) 200ms, CLS(밀림) 0.1이 기준선. 효과 큰 순서: 대표 이미지 압축(WebP/AVIF) → 무거운 서드파티 스크립트 제거 → CDN+캐싱 → JS 번들 축소. 모든 <img>에 width·height 지정, 단 LCP 이미지에는 loading="lazy" 금지(가장 흔한 실수).

③ 안정성 · ④ 유지보수성

  • 에러를 "조용히 넘기지" 않기 — 사용자에겐 복구 방법을, 개발자에겐 로그를
  • 핵심 흐름(가입→결제→핵심 기능)에 최소한의 자동 테스트
  • 작은 PR 단위로 — AI에게도 "한 번에 하나의 관심사만" 시키기
💡 저자 의견

하드닝 항목을 출시 직전에 몰아서 하면 반드시 빠뜨립니다. 3장의 프로젝트에서 효과를 본 방식은 기능을 붙이는 시점마다 3문항 자가점검이었습니다: ① 실패해도 성공처럼 보이지 않는가? ② 파괴적 동작에 확인창이 있는가? ③ 화면 표시가 실제 상태를 따라가는가? — 이 세 질문만 습관이 되어도 출시 전 감사에서 발견되는 항목이 크게 줄어듭니다.

📎 원본 상세: 통합본 8장 — 프로토타입을 '상품'으로 보안 점검표 전체, Core Web Vitals 세부 수치와 개선 순서, 버전 관리 전략의 상세는 분권 전 통합본 8장에 있습니다.
5장

AI 에이전트 스택으로 확장하기

학습 목표 — 코딩 도구를 넘어, 24시간 일하는 에이전트(메시징 비서·자율 에이전트·로컬 LLM·VPS)의 지형도를 이해하고 나에게 맞는 조합을 고른다.

중급자의 다음 지평은 "내가 앉아 있을 때만 일하는 AI"에서 "내가 없어도 일하는 AI"로의 확장입니다. 메시징 게이트웨이형 비서(OpenClaw), 자가학습 자율 에이전트(Hermes), 내 컴퓨터에서 도는 로컬 LLM(Ollama), 그리고 이들을 24시간 살려두는 인프라(VPS)가 그 부품들입니다.

조합구성비용맞는 사람
A. 완전 로컬Ollama + 로컬 LLM₩0프라이버시 우선, 하드웨어 보유
B. 24/7 클라우드VPS + 에이전트 + API월 $6~ + API상시 자동화가 필요한 사람
C. 하이브리드민감한 건 로컬, 무거운 건 API중간실전 추천 — 대부분의 중급자
💡 저자 의견

에이전트 스택은 매력적이지만, 순서를 지키세요: 2장의 워크플로우(스킬·훅)로 "반복을 자동화하는 감각"을 먼저 익힌 뒤에 넘어와야 합니다. 자기 반복 작업이 뭔지 모르는 상태에서 24시간 에이전트를 세우면, 24시간 동안 뭘 시킬지 모르는 서버 비용만 나갑니다. 그리고 3장의 교훈이 여기서도 그대로 적용됩니다 — 에이전트가 "했다"고 보고하는 것과 실제로 된 것은 다릅니다. 무인 시스템일수록 검증 장치를 먼저 설계하세요.

📎 딸림 문서: AI 에이전트 스택 설명서 (OpenClaw · Hermes · Ollama · 로컬 LLM · Hostinger) 각 도구의 개념·아키텍처·설치 실습, 2026년 추천 로컬 모델, 하드웨어 가이드, 보안·비용 체크리스트, 호스팅 심층 비교까지 — 이 주제의 전체 상세를 다루는 별권입니다.

중급자 자가진단 체크리스트

아래에 대부분 "예"라고 답할 수 있으면, 이 교재의 목표에 도달한 것입니다.

운용

  • 작업 강도에 따라 모델을 바꿔 쓰고, 막히면 진단과 수술을 분업시킨다
  • CLAUDE.md가 있고, 하루 한 번 이상 하는 작업 하나가 스킬로 되어 있다

검증

  • AI의 "완료" 선언을 실물 결과 조회로 확인한다 (로그의 성공을 믿지 않는다)
  • 수정 후 첫 재검증에 캐시버스팅을 쓰고, 테스트 후 저장소 오염을 확인한다
  • 폴백 버그를 고치면 같은 패턴을 전체에서 즉시 찾아본다
  • 재발한 버그 앞에서 "다른 원인일 가능성"부터 의심한다

구조와 기록

  • 같은 계산이 두 곳에 있으면 함수 하나로 뽑아내게 시킨다
  • 외부 시스템의 반직관적 동작을 발견 즉시 문서에 적는다
  • 기능을 붙일 때마다 3문항(가짜 성공·확인창·표시 동기화)을 점검한다
  • 보안 4기둥을 출시 직전이 아니라 개발 중에 나눠 점검한다

다음 단계가 궁금하다면 — 이 교재의 사례가 된 실제 프로젝트처럼, 실물(사용자·장비·돈)이 걸린 프로젝트를 하나 시작하는 것이 가장 빠른 성장입니다. 장난감 프로젝트는 교훈 1~6을 가르쳐주지 않습니다. 실물만이 가르칩니다.
