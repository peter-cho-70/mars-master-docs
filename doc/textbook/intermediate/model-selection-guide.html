MODEL SELECTION GUIDE

작업 강도에 맞는
모델 고르기 — Cursor & Claude

모든 작업에 제일 똑똑한 모델을 쓸 필요는 없습니다. 질문에 답하면 지금 할 일에 맞는 모델을 바로 추천해 드립니다.

가벼움 — 빠른 모델로 충분
중간 — 균형형 모델 권장
무거움 — 최상위 추론 모델 필요

지금 뭘 하려고 하세요?

하나를 고르면 Cursor·Claude 각각의 추천 모델과 이유를 보여드립니다.

01 · CURSOR

Cursor에서 바이브 코딩할 때

Cursor 모델 드롭다운(Auto · Composer 2.5 · GPT-5.5 · Opus 4.8 · Gemini 3.1 Pro · Grok 4.3) 기준입니다. 작업 성격에 따라 나눠 쓰면 크레딧과 시간을 크게 아낄 수 있습니다.

작업추천 모델이유
PRD · 기획 · 아키텍처 설계 Opus 4.8 (또는 GPT-5.5) Plan Mode(Shift+Tab)에서 초안이 잘못되면 이후 모든 구현이 틀어집니다. 가장 신중한 모델을 여기에 씁니다.
기존 코드 대규모 리팩터링 Opus 4.8 여러 파일에 걸친 의존성 추론이 필요한 작업. 속도보다 정확도가 중요합니다.
새 기능 구현 (바이브 코딩) Composer 2.5 Cursor 자체 모델로 속도가 빨라 플랜을 여러 번 재실행하거나 실험하기 부담이 없습니다. 계획이 이미 서 있다면 여기서부터는 이 모델로 충분합니다.
버그 수정 · 테스트 작성 Composer 2.5 범위가 명확한 작업. Auto 모드로 둬도 대부분 이 정도 모델이 자동 선택됩니다.
간단한 질문 · 코드 설명 Auto "이 함수가 뭐 하는 거야?" 수준의 질답에 프론티어 모델을 수동 지정할 필요 없습니다. Auto가 크레딧을 아껴줍니다.
02 · CURSOR 안의 CLAUDE CODE

Cursor 터미널에서 Claude Code CLI 쓸 때

Cursor의 내장 터미널에서 claude 명령으로 Claude Code를 그대로 띄울 수 있습니다. 이때는 Cursor의 모델 드롭다운과 별개로, Claude Code 자체의 Opus·Sonnet·Haiku 중에서 골라야 합니다.

작업추천 모델이유
아키텍처 설계 · 어려운 디버깅 Opus 여러 파일에 걸친 의존성을 한 번에 붙잡고 있어야 하는 작업. 경쟁 레이스 컨디션, 미묘한 상태 관리 버그처럼 원인이 잘 안 보일 때.
일반적인 기능 구현 · 리팩터 Sonnet (기본값) 전체 작업의 80~90%를 Opus에 준하는 품질로 처리합니다. 특별한 이유가 없다면 항상 이걸로 시작하세요.
단순 반복 작업 Haiku 파일 전체에 변수명 바꾸기, 저작권 연도 수정, 로그 문구 정리처럼 기계적인 작업. Sonnet보다 2배가량 빠릅니다.
계획 → 구현이 이어지는 작업 opusplan 계획 단계는 Opus, 구현 단계는 자동으로 Sonnet으로 전환되는 하이브리드 모드. "뭘 바꿔야 할지 파악 → 실제로 바꾸기" 흐름에 딱 맞습니다.

모델은 세션 중간에도 바꿀 수 있고, 바꿔도 그동안의 대화 맥락은 그대로 유지됩니다.

터미널 명령어
Cursor 통합 터미널 안에서 claude 실행 중
/model              # 현재 모델 확인 + 목록 보기
/model opus         # Opus로 전환 — 어려운 문제, 설계
/model sonnet       # Sonnet으로 전환 — 기본 작업
/model haiku        # Haiku로 전환 — 단순 반복 작업
/model opusplan     # 계획은 Opus, 구현은 자동으로 Sonnet

# 세션 시작 전에 특정 모델로 바로 켜기
claude --model opus
claude --model sonnet

# 매번 켤 때마다 기본 모델을 고정하고 싶다면 (~/.zshrc 또는 ~/.bashrc)
export ANTHROPIC_MODEL="claude-sonnet-5"
💡 Sonnet으로 시작해서 막히는 지점에서만 /model opus로 올리고, 정리·포맷 작업은 다시 /model haiku로 내리는 식으로 세션 하나 안에서 오가는 게 가장 비용 효율적입니다.
03 · CLAUDE.AI

Claude 모델 등급별 쓰임새

원리는 Cursor와 같습니다 — 작업 난이도에 모델을 맞추는 것.

Haiku
단순 반복 작업. 요약, 번역, 형식 변환, 분류처럼 복잡한 추론이 필요 없는 일. 가장 빠르고 토큰당 비용이 낮습니다.
Sonnet
대부분의 기본 작업. 일상적인 코딩, 글쓰기, 데이터 정리. 특별히 어렵지 않다면 여기서 끝납니다.
Opus
복잡한 추론이 필요할 때. 아키텍처 설계, 까다로운 디버깅, 다단계 논리. 쉬운 일에 쓰면 비용 낭비, 정말 어려운 일엔 재작업을 줄여줍니다.
04 · 토큰 절약

기능 켜고 끄기 & 실전 습관

웹 검색·리서치는 최신 정보가 필요할 때만 켭니다. 특히 리서치(Deep Research)는 수십 번 검색을 돌리므로 토큰 소모가 가장 큽니다.
반복되는 코딩·크롤러 작업은 채팅에 코드를 붙여넣기보다 Claude Code처럼 레포를 직접 다루는 도구가 훨씬 효율적입니다.
자주 참고하는 자료(사이트 구조, 코딩 규칙 등)는 매번 붙여넣지 말고 프로젝트(Projects)에 한 번만 넣어두세요.
대화가 길어질수록 매 턴마다 이전 내용을 전부 다시 처리합니다. 주제가 바뀌면 새 대화를 시작하는 것이 가장 큰 절약입니다.
애매하게 물어 여러 번 되묻는 것보다 프롬프트를 구체적으로 쓰는 편이 왕복 토큰을 줄입니다.
05 · 세션 한도 도달했을 때

진행 중인 작업을 계속하는 방법

Claude Code의 5시간 롤링 윈도우나 주간 한도에 도달하면 더 이상 Claude API를 부를 수 없습니다. 하지만 진행하던 작업을 저장하고 여러 방법으로 계속할 수 있습니다.

1️⃣ 한도 상태 확인
세션이 응답 없어질 때 먼저 해야 할 일. /usage 또는 /cost 명령어를 실행해서 정확한 리셋 시간을 확인하세요.
한도 확인 명령어
Claude Code CLI 세션 중
/usage          # 구독 플랜: 현재 사용량 + 리셋 시간 표시
/cost           # API 키: 이번 세션 비용 계산 + 총 지출
/context        # 현재 컨텍스트 윈도우 사용률 (색상 그리드)

4가지 대처 방법 (우선순위순)

방법 1: 현재 세션에서 토큰 절약하기 (권장)
Claude Code CLI
# 한도가 될 때까지 시간이 조금 남아 있다면:

/compact              # 대화를 요약으로 압축 (대화 맥락 유지하며 토큰 절약)
                      # "API Error: Rate limit reached"가 안 나올 때까지 사용 가능

/compact focus on authentication fixes     # 특정 부분만 강조하며 압축

# 여전히 부족하면:
/model haiku          # 모델을 낮춰서 토큰 소비 줄이기
                      # 마지막 정리·수정 작업에만 사용
방법 2: 작업 저장 후 API 키로 계속하기
Terminal
# 진행 상황을 git에 커밋
git add .
git commit -m "WIP: [진행 상황 설명]"

# Anthropic API 키 설정 (없으면 console.anthropic.com에서 생성)
export ANTHROPIC_API_KEY="sk-ant-..."

# 같은 프로젝트에서 새로운 claude 세션 시작
# (이제 API 크레딧이 별도로 사용됨 — 구독 한도 무관)
claude
방법 3: 기존 세션 다시 불러오기 (나중에 계속)
Terminal 또는 Claude Code
# 리셋 시간이 지난 후, 진행하던 작업으로 바로 돌아가기

# 가장 최근 세션 불러오기
claude --continue         # (또는 -c)

# 선택지 리스트에서 고르기
claude --resume           # (또는 -r)

# CLI 안에서:
/resume                   # 최근 세션 목록 표시
방법 4: 완전히 새로 시작 (드물게)
Claude Code CLI
# 진행 상황이 더 이상 필요 없을 때만 사용 (되돌릴 수 없음)

/clear                    # 대화 히스토리 완전 삭제
                          # (코드 변경사항은 유지됨, CLAUDE.md도 다음 세션에서 로드)

/clear checkpoint-name    # 지금까지의 작업에 라벨을 붙이고 저장한 후 초기화

⏱️ 세션 한도를 애초에 아끼는 법

작업을 작은 세션으로 쪼개기: 한 번에 4시간을 쓰는 것보다 1.5시간씩 3번 나눠서 하는 게 낫습니다 (각 세션이 맑음).
Sonnet으로 기본값 유지: Opus는 진짜 필요할 때만 쓰세요. 일상적인 구현은 Sonnet으로 충분하고 비용이 60% 아닙니다.
/plan으로 계획 먼저: 막연히 "만들어줘"보다 "뭘 바꿔야 할까"를 먼저 물으면 오류를 줄여 오히려 토큰을 아낍니다.
CLAUDE.md에 규칙 고정: 매번 "우리 패턴은..."이라 설명하지 말고, CLAUDE.md에 한 번 써두면 모든 세션에서 자동으로 로드됩니다.
/compact를 습관화: 45분마다, 또는 주요 작업이 끝난 후 한 번씩 /compact를 실행하면 세션이 오래가고 응답 품질이 떨어지지 않습니다.
💡 한도 도달은 "설계 실패"가 아니라 작업이 깊다는 증거입니다. 처음부터 한도를 염두에 두고 /compact · /clear · 짧은 세션 · CLAUDE.md 등을 조합하면 얼마든지 큰 프로젝트를 진행할 수 있습니다.
06 · 사전 설정

매번 물어보지 않게 미리 심어두기

아래 두 텍스트를 각 도구의 설정에 붙여넣으면, 새 채팅을 열 때마다 이 기준이 자동으로 적용됩니다.

.cursor/rules/model-guide.mdc
Cursor → 프로젝트 루트에 저장
작업을 시작하기 전에 항상 작업 성격을 먼저 판단할 것:

- PRD, 기획, 아키텍처 설계, 대규모 리팩터링 → Plan Mode로 전환 제안하고
  Opus 4.8 또는 GPT-5.5 사용을 권장할 것
- 기능 구현, 버그 수정, 반복적인 코드 작성 → Composer 2.5 사용을 권장할 것
- 단순 설명, 코드 질문 → Auto 모드로 충분함을 알릴 것

모델을 직접 전환하지는 못하니, 답변 시작 부분에
"[권장 모델: ___]" 형식으로 한 줄만 표시할 것.
Custom instructions
Claude → 설정 → 맞춤형 지침 / 프로젝트 지침
새 대화 초반에 내가 하려는 작업이 불분명하면,
"간단한 질문 / 계획·기획 / 실제 구현·반복 작업" 중
어느 쪽에 가까운지 먼저 짧게 물어봐줘.

- 간단한 질문·설명 → Haiku급으로 충분하다고 안내
- 계획·기획·복잡한 설계 → Opus급을 권장
- 그 외 일반 작업 → Sonnet 기본 유지

그리고 웹 검색·리서치 같은 토큰 소모가 큰 기능은
정말 필요한 경우에만 켜라고 먼저 확인해줘.
