# 실전 문제해결 가이드 (Troubleshooting)

실제 방송·실장비 테스트에서 **한 번 이상 실제로 발생했던** 문제와 검증된 해결책 모음입니다.
증상으로 찾아보세요 — 각 항목은 `증상 → 원인 → 조치` 순서입니다.

> 이 문서의 모든 항목은 가상 시나리오가 아니라 2026-06~07 실제 운영·테스트 중 발생한 사례입니다.
> 이미 코드로 수정된 항목은 "수정됨" 표시와 함께, 재발 시 확인할 지점을 남겨뒀습니다.

## 1. 전환효과 (Cut/Mix/디졸브)

### "디졸브로 설정했는데 컷으로만 나간다"
실제 방송 중 보고된 증상. 원인이 **4가지**나 있으므로 순서대로 확인:

1. **Master 활동 로그부터 확인** — `전환효과 "X"를 찾을 수 없어 Cut으로 처리됩니다` 경고가 있으면:
   설정 탭 → 전환효과 설정에서 누군가 MIX/FADE/WIPE 프리셋을 삭제/개명한 것. MARS는 이 4개 이름을 그대로 보내므로 프리셋이 없으면 조용히 Cut으로 강등된다. 프리셋을 복구하면 해결.
2. **아이템에 전환In/Out이 아예 설정 안 된 경우** — 특히 **구버전 템플릿**으로 만든 큐시트.
   과거 템플릿 저장 시 transIn/transOut에 'Cut'이 박제되는 버그가 있었다(2026-07-05 수정). 이미 저장된 템플릿·아이템은 자동 교정되지 않으므로: 아이템 열어 전환 드롭다운 수동 수정 → 템플릿도 다시 저장.
3. **타입 기본값 오버라이드** — 타입 관리에서 해당 타입의 transIn/transOut을 바꾼 적이 있으면 코드 기본값보다 우선한다. 타입 관리 모달에서 확인.
4. 위 3개가 모두 아니면 코드 회귀 의심 — 그때 개발 이슈로.

### 전환 우선순위가 헷갈릴 때
**들어오는 아이템의 전환In이 무조건 이긴다.** 직전 아이템의 전환Out은 다음 아이템 진입에 영향을 주지 않는다(2026-07-04 확정 원칙). 전환Out이 실제로 동작하는 건 순차형(ANCHOR/REPORT)의 앵커→영상 내부 전환뿐이다.

### 타입별 기본 원칙 (기준값)
| 타입 | 전환In | 전환Out(앵커→영상) |
|---|---|---|
| REPORT(완제) | Cut | Cut |
| ANCHOR(단신) | Cut | **Mix** |
| CM | Mix | Mix |
| BRIDGE | Cut | Mix |
| TITLE | Fade | Cut |

단신이 Mix/Mix로 기록된 옛 문서가 있다면 그쪽이 틀린 것 — **Cut/Mix가 맞다** (2026-07-05 확정).

## 2. 자막 (CG / GT 타이틀)

### "자막 내용이 안 바뀌는데 로그는 전부 성공"
**가장 함정 같은 사례.** vMix GT 필드명 설정에 `.Text`까지 포함해 입력한 경우(`TextBlock1.Text`), 코드가 `.Text`를 한 번 더 붙여 `TextBlock1.Text.Text`가 되는데 — vMix는 이때도 "Success"를 반환하면서 아무것도 안 바꾼다. 오버레이 ON/OFF는 정상 동작해서 더 헷갈린다.
**조치**: 설정 탭 CG 필드명에는 **base 이름만** 입력 (`TextBlock1`, `.Text` 제외).

### "OO 실패: … is not a function" 같은 이상한 에러
실제 vMix 호출은 성공했는데 **그 뒤의 로그 표시 코드가 죽어서** 실패로 잘못 표시된 사례가 있었다(2026-07-02, 수정됨). 비슷한 에러가 다시 보이면: 자막이 실제로 화면에 나갔는지부터 확인 — 나갔다면 연동이 아니라 표시 코드 문제다.

### GT 인풋을 못 찾음
대소문자 오타가 제일 흔하다 (`GT-title1` vs `GT-Title1`). 설정 탭의 **🔍 감지** 버튼이 대소문자 무시 매칭 + 원클릭 교정을 제공한다.

### 오버레이 하나 껐는데 다른 자막도 꺼짐
서로 다른 자막 타입이 **같은 오버레이 슬롯 번호**를 공유하면 vMix 채널이 하나이므로 함께 꺼진다 — vMix 구조상 정상. 타입별로 다른 슬롯(1~8)을 배정할 것.

## 3. 서버/클립 재생 (SRV1~3)

### "다음 소재의 첫 프레임이 살짝 보인다"
vMix 리스트는 소재가 끝나면 **유예시간 0으로 즉시** 다음 항목으로 넘어간다(실측 확인). 설정의 "끝나기 (프레임) 전 정지"가 이걸 막는 장치 — **기본값 15프레임(약 0.5초)**. 5프레임은 실측에서 반복 실패했으니 줄이지 말 것.

### "SRV1/2만 깨지고 SRV3는 정상" 류의 슬롯 편중 증상
과거 두 가지 원인이 있었다(모두 수정됨):
1. SRV 슬롯의 vMix 인풋명이 비어 있으면 조용히 SRV1로 폴백 → 다른 기능과 같은 물리 인풋에서 충돌.
2. 중단된 큐시트 재생의 추적 변수가 남아 라디오 재생과 충돌.

재발 시 첫 확인: **설정 → 스위처 소스 관리에서 SRV1/2/3 모두 vMix 인풋명이 실제로 채워져 있는가.**

### 빠른 연속 단축키로 영상이 겹치거나 끊김
같은 SRV에 연속 재생 명령이 겹치면 "마지막 요청만 승리"하도록 수정됨(2026-07-06). 재발하면 새로 추가된 재생 경로가 세대 카운터(`_srvPlayGeneration`) 체크를 빠뜨렸는지 확인.

### 이 vMix에서 Function=Stop이 안 먹음
이 환경의 vMix는 `Function=Stop`이 무효(조용히 실패). **Pause를 쓴다** — 코드도 전부 Pause로 통일되어 있다.

## 4. 스위처 (PGM/PVW)

### "PGM만 선택했는데 PVW가 제멋대로 바뀐다"
**버그가 아니라 vMix의 정상 동작.** `Cut&Input=X`는 실제 스위처처럼 직전 PGM을 PVW로 되돌린다(버스 스왑). 과거엔 앱 표시가 1.5초 늦게 따라와서 "제멋대로 바뀌는" 것처럼 보였는데, 2026-07-10에 즉시 반영으로 수정됨. 재발하면 새로 추가된 컷 경로가 로컬 스왑 반영을 빠뜨렸는지 확인 (`setPGM`/`doCut`/`doAuto`/`_vmixCut`/`radioCutTo` 5곳이 기준 구현).

### 단축키가 갑자기 안 먹음
스위처 탭의 **⏻ 단축키 켜기/끄기** 토글 확인 — 실수 방지용 안전장치로, 꺼져 있으면 키보드만 막힌다(마우스 클릭은 항상 동작). 새로고침하면 항상 켜짐으로 시작.

## 5. OBS 버전 (obs/news-broadcast-system-obs.html)

### CUT을 눌렀는데 디졸브로 나감
**한국어 OBS**에서는 전환 이름이 "자르기"/"서서히 사라지기"라서 영문 'Cut' 매칭이 실패했던 사례. `transitionKind` 기반으로 수정됨(2026-07-09). 새 OBS 설치에서 재발하면 그 인스턴스의 전환 목록(`GetSceneTransitionList`)에 cut_transition이 존재하는지 확인.

### CG 브라우저 오버레이가 안 보임 (설정은 전부 정상인데)
macOS OBS의 **브라우저 소스 하드웨어 가속 버그**. 설정 → 고급 → "브라우저 소스 하드웨어 가속 사용" 토글 후 **OBS 완전 재시작**. 스크린샷 API가 빈 이미지를 반환하는 것도 같은 원인.

## 6. MARS ↔ Master 데이터

### "큐시트가 전반적으로 깨짐 / 타이틀부터 시작 안 함"
코드보다 **데이터부터** 확인 — 실제 사례에서 원인은 8개 아이템의 `clips:[]`가 비어 있던 것(MARS 영상 매칭 미완). 3계층 순서로 점검:
1. 큐시트 JSON에서 `clips` 비었는지 (TITLE/CM/REPORT류는 클립 필수)
2. 클립 파일명이 vMix 리스트의 실제 파일명과 **확장자 포함 정확히** 일치하는지 (경로 없이 파일명만)
3. Master의 인풋 매핑 검증 도구(설정 탭)

### 기사를 수정했는데 아이템/Master의 자막이 옛날 내용
기사 매칭은 **최초 1회 복사**라서 이후 기사 수정이 자동 반영되지 않는다(PD가 아이템에서 직접 고친 자막 보호 목적). **아이템매칭 탭의 "🔄 자막 새로고침"** 버튼으로 명시적으로 다시 끌어온다. 프롬프트도 같은 구조 — "프롬프트 새로고침" 사용.

### 수정했는데 화면이 그대로다
**브라우저 캐시.** 하드 리로드(Cmd+Shift+R) 하거나 주소 뒤 `?v=2`를 붙인다. 헤더의 **빌드 스탬프**(빌드 YYYY-MM-DD HH:MM)가 파일 수정 시각과 일치하는지로 최신 여부를 판별할 수 있다 — 이 문제로 "수정이 안 됐다"고 오판한 사례가 개발 중에도 2번 이상 있었다.

## 7. 진단이 막힐 때의 일반 원칙

- **로그의 "성공"을 믿지 말고 결과를 직접 확인** — vMix는 무효한 명령에도 Success를 반환하는 경우가 있다 (§2의 `.Text` 사례).
- **같은 증상 ≠ 같은 원인** — "SRV 깨짐"은 서로 다른 원인 3개가 같은 증상을 냈다. 한 번 고쳤던 문제가 재발하면 이전 수정이 풀린 게 아니라 다른 원인일 가능성부터.
- **장비의 실제 상태가 진실** — 앱 화면·로그보다 vMix `/api/` XML, OBS websocket 응답을 직접 확인하는 것이 최종 근거다.
