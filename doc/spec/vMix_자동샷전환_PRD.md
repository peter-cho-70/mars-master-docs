# PRD: vMix 마이크 레벨 기반 자동 샷 전환 (Auto Cam Switch)

## 0. 문서 정보
- 작성일: 2026-07-22
- 상태: Draft v0.1
- 대상 플랫폼: vMix (HTTP/TCP API 연동 외부 컨트롤러 — 별도 실행 프로그램/스크립트)

## 1. 개요
발언 중인 인물의 마이크 레벨이 일정 시간 이상 유지되면, 해당 마이크에 링크된 카메라(인물 원샷)로 vMix 프로그램 화면을 자동 전환하는 기능. 다자 대담·인터뷰·회의 중계처럼 스위처를 상시 조작할 인력이 부족한 상황에서 샷 전환을 자동/반자동화하는 것이 목적.

## 2. 문제 정의
- 현재 vMix는 오디오 레벨에 따라 카메라를 자동으로 전환하는 네이티브 기능이 없음 (Audio Follow는 오디오 믹싱용이지 비디오 스위칭용이 아님).
- 다자 출연 프로그램에서 스위처 1인이 모든 발언자를 실시간으로 캐치해 컷하기 어려움 → 발언자와 다른 화면이 송출되는 딜레이/미스컷 발생.
- 완전 자동화 시 순간적 잡음(헛기침, 겹침 발언)에도 반응해 컷이 난발될 위험 → 자동/반자동/수동 단계별 제어가 필요.

## 3. 목표
| 목표 | 성공 지표 |
|---|---|
| 발언자 인식 → 자동 컷 전환 | 발언 시작 후 전환까지 지연 1초 이내 |
| 불필요한 컷 난발 방지 | 5초 내 동일 입력 재전환(채터링) 0건 |
| 운영자 개입 여지 확보 | 반자동 모드에서 추천 샷 클릭 1회로 전환 가능 |

## 4. 대상 사용자
- 소규모 인력으로 다자 대담·인터뷰·강연을 vMix로 스위칭하는 방송기술 스태프
- 스위처 조작 인력이 없거나 부족한 1인 운영 환경

## 5. 유저 시나리오
1. 출연자별 카메라(원샷)와 마이크 입력을 사전에 1:1 매핑해둔다.
2. 방송 시작 시 모드를 자동/반자동/수동 중 선택한다.
3. **자동 모드**: A가 발언을 시작해 마이크 레벨이 임계값 이상으로 일정 시간 유지되면, A의 카메라로 프로그램이 자동 컷 전환된다.
4. **반자동 모드**: 동일한 조건이 감지되면 해당 카메라가 "추천 샷"으로 하이라이트되고, 운영자가 클릭(또는 단축키)하면 컷된다. 일정 시간 내 미클릭 시 추천만 유지되고 자동 전환은 되지 않는다.
5. **수동 모드**: 마이크 레벨 감지는 화면에 표시되지만 전환에는 관여하지 않는다. (기존 수동 스위칭 + 레벨 모니터링 보조 기능)

## 6. 기능 요구사항

### 6.1 카메라-마이크 매핑 설정
- vMix Input 목록을 불러와 "카메라 Input ↔ 마이크 Input(또는 카메라에 임베드된 오디오 채널)"을 매핑하는 설정 화면
- N:1 매핑 지원 여부는 v1에서는 1:1만 지원 (확장 여지는 8절 참고)

### 6.2 모드 선택
- 자동 / 반자동 / 수동 3단계, 방송 중 실시간 전환 가능
- 모드별 아이콘/색상으로 현재 상태를 명확히 표시 (오조작 방지)

### 6.3 트리거 로직
- **임계값(Threshold)**: 마이크 레벨이 설정 dB(또는 Amplitude) 이상일 때 "발언 중"으로 판정
- **유지시간(Hold Time)**: 임계값 초과 상태가 N ms 이상 지속되어야 트리거 (순간 잡음 필터링, 기본값 300~500ms 권장)
- **디바운스/쿨다운(Cooldown)**: 컷 전환 직후 N초간은 재전환 금지 (채터링 방지, 기본값 1.5~2초 권장)
- **동시 발언(오디오 겹침) 처리**: 레벨 단독 판정이 아닌 **가중치(Weight) × 레벨(Amplitude) 점수**로 판정
  - 기본 가중치: 카메라/출연자별로 사전 설정 (예: 진행자 1.2, 메인 게스트 1.0, 서브 게스트 0.8)
  - **대본/큐시트 연동 가중치**: 사전 입력된 진행 순서(예: "질문 3번은 게스트B 차례")에 따라 해당 시점에는 지목된 출연자의 가중치를 일시 상향 조정 → 겹침 상황에서도 "지금 답변할 차례인 사람"이 우선적으로 컷되도록 유도
  - 점수가 임계값 이상인 입력 중 최고점을 최종 선택, 동점 시 유지시간이 더 긴 쪽 우선
- **무음 구간 처리**: 전원 임계값 미만일 경우 현재 샷 유지(전체 샷/직전 샷으로 자동 컷하지 않음 — v1 범위 밖, 11절 참고)

### 6.4 사전 설정 (프리쇼 구성)
방송 시작 전 아래 항목을 미리 입력해 판정 로직과 UI 추천 샷에 활용합니다.
- **큐시트/대본 등록**: 질문 순서, 각 질문의 예상 답변자, 코너 구성을 순서 리스트로 입력. 자동/반자동 모드에서는 현재 큐 항목에 따라 6.3의 대본 연동 가중치가 자동 적용됨
- **질문 수**: 프로그램에 예정된 총 질문/코너 수를 등록해 진행률 표시 및 큐 자동 진행(다음 큐로 넘어가는 기준)에 활용
- **기본 샷 형식 프리셋**: 오프닝/질문/답변/클로징 등 구간별 기본 샷(원샷/투샷/풀샷) 패턴을 미리 정의, 큐 전환 시 참고 샷으로 UI에 제시
- **카메라 샷 구조 매핑**: 카메라별 역할(예: CAM1=진행자 원샷, CAM2=게스트A 원샷, CAM3=게스트B 원샷, CAM4=전체 풀샷)을 사전 등록해 6.1 매핑 및 판정 로직에서 참조

### 6.5 UI/컨트롤 패널
- 실시간 오디오 레벨 미터 (카메라별 라벨링)
- 현재 프로그램 샷과 "다음 추천 샷" 표시
- 모드 전환 버튼, 임계값/유지시간/쿨다운 슬라이더
- 수동 오버라이드 버튼 (모드와 무관하게 항상 즉시 개입 가능해야 함 — 안전장치)

## 7. 비기능 요구사항
- **지연시간**: 감지~전환 전체 파이프라인 1초 이내
- **안정성**: vMix 연결 끊김 시 자동 재접속, 끊긴 동안은 수동 모드로 자동 전환(폴백)
- **크로스플랫폼**: Mac 환경에서도 동작 (기존 개발 환경 우선순위 반영)
- **로깅**: 컷 전환 이력(시각, 입력, 트리거 사유)을 로그로 남겨 사후 리뷰 가능하게 함

## 8. 기술 아키텍처 (초안)
vMix는 오디오 레벨 변화를 실시간으로 푸시(subscribe)하는 네이티브 API가 없으므로, 외부 컨트롤러가 **폴링(polling) 방식**으로 상태를 가져오는 구조가 현실적입니다.

- **상태 조회**: vMix HTTP API(`/api`)의 XML State를 짧은 주기(예: 50~100ms)로 폴링하여 입력별 오디오 미터 값(Amplitude 0~1)을 확인
- **전환 명령**: 조건 충족 시 vMix Function API(`Cut`, `CutDirect`, 또는 `Fade`)를 HTTP/TCP로 호출해 해당 Input을 프로그램으로 전환
- **컨트롤러 구현**: 별도 상주 프로그램(스크립트/앱)이 폴링→판정→명령 전송 로직을 담당 (vMix 자체 스크립팅보다 외부 컨트롤러가 임계값/모드 UI 구현에 유리)
- **폴링 주기와 지연시간 트레이드오프**: 주기가 짧을수록 반응은 빠르지만 vMix API 부하 증가 — 실측 테스트로 최적 주기 조정 필요

### 8.1 컨트롤러 단말(하드웨어) 구성

```
카메라 + 마이크 (SDI/NDI/USB 입력)
        │
        ▼
vMix PC (Windows) — vMix 실행, HTTP/TCP API 제공
        │  ▲
        │  │  LAN · HTTP/TCP API
        │  │  (레벨 폴링 / Cut 명령)
        ▼  │
컨트롤러 단말 — 미니PC / 라즈베리파이 등
        │
        ▼  로컬 웹 UI
오퍼레이터 UI — 태블릿 · 브라우저 (반자동 조작)
```

- **vMix는 Windows 전용**이므로 vMix 본체는 반드시 Windows PC에서 실행됩니다. 자동 컷 컨트롤러는 이 vMix PC와 **네트워크(LAN)로 API 통신만** 하면 되므로 별도 플랫폼에서 독립적으로 구동 가능합니다.
- **컨트롤러 단말 후보**
  - **vMix PC와 동일 PC에서 상주 프로세스로 실행**: 네트워크 홉이 없어 지연이 가장 낮고 구성이 단순함. 다만 vMix와 자원을 공유하므로 부하 관리 필요
  - **별도 미니PC(Intel NUC, Mac mini 등)**: 유선 LAN으로 vMix PC와 연결, 가중치·대본 로직과 웹 UI 서버를 독립적으로 운영 가능
  - **라즈베리파이 등 초소형 SBC**: 폴링+로직+웹서버 정도의 경량 작업이라 사양 부담이 적어 저비용 상시 구동에 적합. 유선 이더넷 연결 권장(WiFi는 폴링 주기 대비 지연 편차 리스크)
- **오퍼레이터 UI**는 컨트롤러 단말이 제공하는 로컬 웹페이지를 태블릿·노트북·모니터가 달린 별도 PC의 브라우저로 접속해 사용 — 반자동 모드의 "추천 샷 클릭" 조작이 이 화면에서 이뤄짐
- **네트워크 요구사항**: vMix PC ↔ 컨트롤러 단말 ↔ 오퍼레이터 UI 모두 동일 LAN(가급적 유선)에 있어야 하며, 리허설 단계에서 실측 지연을 검증 필요

## 9. 적용 사례: 경제·시사 토크쇼 (4캠, 진행자+출연자2)
카메라 4대(진행자/출연자1/출연자2 원샷 + 전체 풀샷), 주요 화면 구성이 **갈라치기(진행자-출연자 1:1 스플릿) + 풀샷** 중심인 프로그램에 적용할 경우, 기존 설계를 아래와 같이 확장합니다.

### 9.1 샷 라이브러리 확장
1:1 마이크-카메라 매핑만으로는 부족하며, 카메라 2대를 합성한 별도 vMix Input이 추가로 필요합니다.
- CAM1=진행자 원샷, CAM2=출연자1 원샷, CAM3=출연자2 원샷, CAM4=전체 풀샷
- 갈라치기 Input 별도 제작(Layers로 카메라 2대를 좌우 합성한 고정 Input): 진행자-출연자1, 진행자-출연자2
- 출연자1-출연자2가 직접 논쟁하는 경우가 있다면 그 조합도 별도 갈라치기 Input으로 준비할지 검토

### 9.2 갈라치기 판별 로직 (6.3 확장)
"가장 가중치 높은 한 명"을 원샷으로 컷하던 기존 로직에, 대화 상황 판별을 추가합니다.
- **대화 핑퐁 감지**: 짧은 시간(예: 4~6초) 내 두 사람이 번갈아 발언하면 해당 조합의 갈라치기 Input으로 전환
- **단독 발언 유지**: 한 사람만 일정 시간(예: 3초) 이상 단독으로 발언을 유지하면 그 사람 원샷으로 전환
- **전원 겹침**: 세 명이 동시에 겹치면 전체 풀샷으로 전환 (기존 10절에서 v1 범위 밖이었던 규칙을 이 프로그램에서는 포함)

### 9.3 큐시트 연동 강화 (6.4 확장)
질문 순서가 사전에 정해진 경우가 많은 토크쇼 특성상, 큐 항목마다 "기본 샷 타입"을 지정해두면 판별 정확도가 올라갑니다.
- 큐 항목별 기본 샷 지정 (예: 3번 질문=진행자-출연자2 갈라치기)
- 큐 전환 시 해당 갈라치기 Input을 추천 샷으로 우선 표시
- 자유토론 구간은 큐 지정 없이 9.2의 판별 로직만으로 동작

### 9.4 운영 모드 권장
시사 토크쇼는 논쟁 중 오버랩 발언이 잦아 완전 자동보다 반자동이 안전할 수 있습니다.
- 갈라치기처럼 "두 화면을 합성해야 하는 판단"은 반자동(추천 후 클릭)으로 시작
- 단순 원샷 전환(한 명이 명확히 계속 말할 때)만 자동 모드로 두는 하이브리드 설정 고려
- 리허설에서 핑퐁 감지 시간(4~6초)과 단독발언 유지시간(3초) 값을 실측 튜닝

### 9.5 컴퓨터 화면(PC/그래픽) 소스 처리
경제·시사 토크쇼는 차트·통계·PPT 등 컴퓨터 화면을 풀샷 또는 인물과의 분할 화면으로 자주 사용합니다. 이 소스는 특정 마이크와 연결되어 있지 않아 기존 오디오 트리거 로직만으로는 전환 시점을 판단할 수 없습니다.
- **샷 라이브러리 추가**: PC 화면 풀샷 Input, PC화면+진행자 갈라치기 Input, PC화면+출연자 갈라치기 Input을 9.1의 카메라 조합과 별도로 준비
- **전환 트리거**: 마이크 레벨 기반이 아닌 **큐시트 지정(9.3) 또는 오퍼레이터 수동 트리거**로 전환 — "몇 번째 질문에서 화면 자료 사용" 여부를 사전 큐에 등록해두면 해당 큐 도달 시 추천 샷으로 표시
- **자동 전환 억제(Lock)**: PC 화면이 떠 있는 동안에는 마이크 레벨 기반 자동 컷이 임의로 인물 샷으로 끊고 들어가지 않도록 "화면 표시 중 자동전환 잠금" 옵션 필요 — 설명이 끝났음을 오퍼레이터가 해제하거나, 다음 큐로 넘어갈 때 해제
- **인물 결합 시 우선순위**: PC화면+인물 갈라치기 상태에서 발언자가 바뀌면(예: 진행자→출연자2) 결합된 인물 쪽만 교체하고 PC 화면은 유지하는 옵션 고려

### 9.6 유튜브 라이브 트렌드 반영 확장 아이디어
2026년 유튜브 라이브 제작 트렌드(가로/세로 동시 송출, 종료 후 AI 하이라이트 쇼츠 자동 생성, 실시간 시청자 반응 연동)와 최근 방송기술 트렌드(AI 실시간 자막)를 반영하면, 이 자동 컷 시스템이 이미 가진 "누가 언제 말했는지"라는 데이터를 아래처럼 확장 활용할 수 있습니다.

- **세로형(쇼츠) 동시 송출용 자동 리프레임**: 가로용 프로그램 출력과 별도로, 현재 발언자의 원샷 카메라만 9:16으로 크롭한 vMix Output을 하나 더 만들어 동시 송출. 자동 컷 로직이 이미 "지금 누가 말하는가"를 판정하고 있으므로 별도 판별 로직 추가 없이 출력만 분기하면 됨
- **하이라이트 구간 자동 마킹**: 자동 컷 로그(발언자 전환 시각, 발언 유지시간, 갈라치기 발생 빈도)를 방송 종료 후 AI 쇼츠 생성 도구에 타임스탬프 메타데이터로 넘겨, 대화가 활발했던(전환이 잦았던) 구간을 하이라이트 후보로 우선 제시
- **시청자 채팅 반응 연동 트리거**: 유튜브 라이브 채팅에서 특정 인물 언급이나 반응(도배성 댓글)이 급증하면, 반자동 모드의 "추천 샷"에 해당 인물 클로즈업을 우선순위로 띄우는 옵션 — 참여형 스트리밍 트렌드에 대응
- **실시간 화자 자막(이름표) 자동 연동**: 발언자 전환 시 vMix Titles의 하단 자막(이름/직함)도 API로 함께 갱신 — 카메라 전환과 자막 전환의 싱크가 어긋나는 문제를 구조적으로 방지

## 10. 제약사항 및 리스크
- vMix에 오디오 레벨 실시간 이벤트 API가 없어 폴링 기반 구현이 불가피 → 완벽한 실시간성은 보장 어려움
- 마이크 게인/노이즈게이트 설정이 카메라마다 다르면 임계값 하나로 대응 어려움 → 카메라별 개별 임계값 설정 필요
- 다수 출연자가 짧게 주고받는 대화(예능형 토크)에서는 자동 모드가 오히려 방해될 수 있음 → 반자동/수동 전환을 쉽게 만드는 것이 중요
- 갈라치기 Input 조합이 늘어날수록(3인 이상 프로그램 등) 사전 제작해야 할 vMix Input 수가 조합별로 증가함
- PC 화면처럼 마이크와 무관한 소스가 섞이면 오디오 레벨만으로는 전환 판단이 불완전 → 큐시트/수동 트리거와의 하이브리드 설계가 필수


## 11. 향후 확장 (v1 범위 밖)
- N:1 매핑(한 카메라에 여러 마이크가 걸린 경우 레벨 합산/최대값 처리)
- 전원 무음 시 자동으로 전체 샷(와이드)으로 전환
- 발언 패턴 학습 기반 예측 전환 (누가 다음에 말할지 힌트)
- 갈라치기 조합을 정적 Input이 아닌 동적 Layer 재배치로 실시간 생성 (출연자 수가 유동적인 프로그램 대응)
