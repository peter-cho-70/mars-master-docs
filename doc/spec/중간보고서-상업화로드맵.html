지금까지 완료된 부분: MARS(제작)·Master(진행) 두 축의 뉴스 큐시트 파이프라인은 vMix·OBS 두 장비 모두에서 실기기 검증까지 마쳤고, Blackmagic Videohub 라우터 연동도 1단계(크로스포인트 전환)까지 실장비로 확인됐습니다. 이 부분은 상업화 논의와 무관하게 제품으로서 이미 동작하는 실체입니다.

로그인/계정 시스템은 코드가 0%입니다. doc/login.md는 Next.js + Supabase + Vercel 스택의 구현 가이드이지만, 저장소 어디에도 package.json·Next.js·Supabase 관련 코드가 존재하지 않습니다 — 계획서만 있고 착수 전입니다.

더 중요한 문제는, 이 계획서가 기존 PRD와 서로 다른 백엔드 스택을 가정하고 있다는 점입니다. 최초 PRD(NEWS_BROADCAST_PRD.md)는 Phase 2 백엔드로 Node.js+Express+PostgreSQL+JWT를 이미 예정해뒀는데, 이번 login.md는 완전히 다른 Next.js+Supabase 체계를 전제합니다. 또한 이 제품의 핵심 가치(vMix/OBS/Videohub 같은 LAN 전용 장비의 실시간 제어)는 애초에 클라우드에서 직접 닿을 수 없는 영역이라, "로그인이 붙은 SaaS"가 되더라도 지금의 server.py 같은 현장 로컬 에이전트는 반드시 남아야 합니다. 이 구조적 결정이 서지 않은 채로 로그인 구현에 먼저 들어가면 다음 주 작업이 되돌아올 위험이 큽니다 — 아래 "결정이 필요한 것" 항목을 먼저 보시길 권합니다.

Section 3 · Critical

로그인 구현 전에 먼저 결정해야 하는 것

아래 두 가지는 코드를 짜기 전에 방향이 서야 하는 사업/아키텍처 결정입니다. 결정 없이 login.md 순서대로 바로 구현에 들어가면, 나중에 사업 모델이 바뀔 때 되돌리는 비용이 큽니다.

결정 1 상업화 모델: 온프레미스 라이선스 vs 클라우드 SaaS vs 하이브리드

현재 제품의 본질은 한 방송사의 로컬 네트워크 안에서 vMix·OBS·Videohub 같은 물리 장비를 직접 제어하는 도구입니다. 이 특성 때문에 "로그인"이 의미하는 바가 사업 모델에 따라 완전히 달라집니다.

A. 온프레미스 라이선스 판매 — 방송사마다 독립 설치(지금과 동일한 구조). 로그인은 사실상 "제품 잠금/라이선스 키 검증" 수준이면 충분 — 지금의 일일 비밀번호 잠금을 정식 라이선스 체계로 승격하는 정도. Supabase 멀티프로바이더 소셜 로그인은 과한 투자일 수 있음.
B. 클라우드 SaaS (login.md가 전제하는 모델) — 방송사별 계정, 소셜 로그인, 구독. 다만 LAN 하드웨어 제어는 클라우드에서 직접 닿지 않으므로, 방송 현장엔 지금의 server.py 같은 로컬 에이전트가 그대로 남고 클라우드는 "계정·설정·과금" 계층만 담당하는 하이브리드 구조가 됨.
권장: B의 하이브리드 형태로 결정하되, 1단계 로컬 계정만 먼저 검증 — 소셜 로그인 3종(구글/카카오/네이버)까지 한 번에 벌이기보다, 로컬 계정으로 "계정이 무엇을 잠그는가"가 실제로 명확해진 뒤 확장하는 편이 login.md 자체의 작성 원칙("1단계를 먼저 완전히 검증한 뒤 2단계로")과도 일치함.
결정 2 백엔드 스택 정리: 기존 PRD(Node+Express+PostgreSQL) vs 신규 login.md(Next.js+Supabase)

두 문서가 서로 다른 백엔드를 가정한 채 저장소에 함께 존재합니다. 어느 한쪽이 다른 쪽을 대체하는 것인지, 아니면 로그인/계정만 Supabase로 분리하고 큐시트·자막 등 나머지 데이터는 기존 PRD대로 PostgreSQL+Express로 가는 것인지 문서상 명시가 없습니다. 이 정리 없이 진행하면 나중에 두 백엔드가 중복 관리되는 상태로 굳어질 위험이 있습니다.
