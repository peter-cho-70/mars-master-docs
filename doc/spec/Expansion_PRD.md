# Expansion_PRD.md — 마스터(Master) 외부 장비 제어 확장 PRD

**버전**: v1.0
**작성일**: 2026-07-10
**대상 시스템**: 마스터 (큐시트 기반 방송 제어 프로그램)
**연관 시스템**: 마스 (제작 소재·자막·큐시트 준비 시스템)

---

## 1. 배경 및 개요

### 1.1 시스템 구조 요약

| 역할 | 시스템 | 설명 |
|---|---|---|
| 준비 | **마스 (MAS)** | 제작에 필요한 내용, 자막, 소재(클립·그래픽)를 준비하고 큐시트를 생성 |
| 실행 | **마스터 (Master)** | 마스가 만든 큐시트를 읽어 OBS, vMix, Blackmagic, Sony 등 외부 장비를 제어하여 실제 방송을 수행 |

### 1.2 현재 상태 (As-Is)

- OBS 연동 완료 (obs-websocket)
- vMix 연동 완료 (TCP API :8099 / HTTP API :8088)
- Blackmagic Design 라우터(Videohub) 네트워크 연결 및 단독 동작 테스트 완료 — **마스터 프로그램과의 연동은 미구현**

### 1.3 확장 목표 (To-Be)

1. **1단계**: Blackmagic Videohub 라우터를 마스터의 큐시트 액션으로 제어
2. **2단계**: VMU(Video Matrix Unit), AMU(Audio Matrix Unit) 등 타사 매트릭스 장비를 동일한 방식으로 제어
3. **장기**: Sony 스위처/라우터 등 추가 장비 확장

### 1.4 핵심 설계 원칙

> **"장비를 하나 추가할 때마다 마스터 본체 코드를 고치지 않는다."**

- 마스터 본체는 **추상화된 액션**(예: `router.route(input, output)`)만 알고, 실제 프로토콜 변환은 **디바이스 드라이버**가 담당한다.
- Videohub, VMU, AMU, Sony는 각각 하나의 드라이버 구현체일 뿐이다.
- 방송 중 장비 통신 장애가 발생해도 **방송(송출) 자체는 절대 중단되지 않는다.** (OnAir Connect 라이선스 설계에서 확립한 원칙과 동일)

## 2. 목표 / 비목표

### 2.1 목표 (Goals)

- G1. 큐시트의 한 행(row)으로 라우터 크로스포인트 전환을 실행할 수 있다.
- G2. 장비 연결 상태를 UI에서 실시간으로 확인할 수 있다. (연결됨 / 재연결 중 / 오프라인)
- G3. 라우터의 입출력 라벨을 자동으로 읽어와 큐시트 작성 시 "이름"으로 선택할 수 있다. (포트 번호 암기 불필요)
- G4. 새 장비 추가 시 드라이버 파일 1개 + 설정 등록만으로 확장 가능하다.
- G5. 모든 제어 명령은 실행 로그(성공/실패/응답시간)를 남긴다.

### 2.2 비목표 (Non-Goals)

- 라우터 펌웨어 업데이트, IP 설정 변경 등 장비 관리 기능 (Videohub Setup 유틸리티의 영역)
- 영상 신호 자체의 모니터링(멀티뷰어) — 별도 프로젝트
- 이번 단계에서 Sony 프로토콜 완전 구현 (인터페이스 호환성만 확보)

## 3. 전체 아키텍처

```
┌─────────────────────────────────────────────────────────┐
│                     마스터 (Master)                      │
│                                                         │
│  ┌───────────┐   ┌──────────────┐   ┌───────────────┐  │
│  │ 큐시트     │ → │ 액션 실행기   │ → │ Device Manager │  │
│  │ (from 마스)│   │ (Executor)   │   │ (연결/상태관리) │  │
│  └───────────┘   └──────────────┘   └───────┬───────┘  │
│                                             │          │
│              Device Driver Interface (공통 규약)         │
│         ┌──────────┬──────────┬─────────┬───┴─────┐    │
│         ▼          ▼          ▼         ▼         ▼    │
│    ┌────────┐ ┌────────┐ ┌────────┐ ┌───────┐ ┌──────┐│
│    │ OBS    │ │ vMix   │ │Videohub│ │ VMU   │ │ AMU  ││
│    │ Driver │ │ Driver │ │ Driver │ │Driver │ │Driver││
│    └───┬────┘ └───┬────┘ └───┬────┘ └───┬───┘ └──┬───┘│
└────────┼──────────┼──────────┼──────────┼────────┼─────┘
         │WebSocket │TCP/HTTP  │TCP:9990  │TCP/시리얼│...
         ▼          ▼          ▼          ▼        ▼
       OBS        vMix     Blackmagic   VMU      AMU
                            Videohub
```

### 3.1 구성 요소 정의

| 구성 요소 | 책임 |
|---|---|
| **Device Manager** | 장비 등록, 연결 수립, 상태 감시(heartbeat), 자동 재연결, 이벤트 브로드캐스트 |
| **Device Driver** | 프로토콜 변환. 공통 인터페이스를 구현하며 장비별 통신 세부사항을 캡슐화 |
| **액션 실행기 (Executor)** | 큐시트 행을 해석해 해당 드라이버의 메서드를 호출, 결과를 로그와 UI에 반영 |
| **장비 설정 저장소** | 장비 목록(이름, 종류, IP, 포트, 옵션)을 JSON/DB로 영속화 |

### 3.2 기존 OBS/vMix 코드의 리팩터링

현재 OBS·vMix 연동이 마스터 본체에 직접 붙어 있다면, 이번 확장에서 **동일한 드라이버 인터페이스 아래로 이동**시키는 것을 권장한다. 단, 리스크 최소화를 위해:

1. 먼저 인터페이스와 Videohub 드라이버를 신규 구조로 구현 (기존 코드 무변경)
2. Videohub가 안정화된 후 OBS → vMix 순으로 드라이버 구조로 마이그레이션
3. 마이그레이션 전후 큐시트 실행 결과가 동일함을 회귀 테스트로 확인

## 4. 디바이스 드라이버 공통 인터페이스

모든 드라이버가 구현해야 하는 최소 규약. (TypeScript 기준 예시 — 실제 언어에 맞게 변환)

```typescript
interface DeviceDriver {
  // === 생명주기 ===
  connect(): Promise<void>;          // 연결 수립 + 초기 상태 동기화
  disconnect(): Promise<void>;
  readonly status: 'connected' | 'reconnecting' | 'disconnected';

  // === 능력 선언 (capability) ===
  readonly capabilities: DeviceCapability[];
  // 예: ['video_route'], ['audio_route'], ['scene_switch', 'overlay'] ...

  // === 명령 실행 ===
  execute(action: DeviceAction): Promise<ActionResult>;

  // === 상태 조회 ===
  getState(): DeviceState;           // 현재 라우팅 테이블, 라벨 등

  // === 이벤트 ===
  on(event: 'stateChanged' | 'connected' | 'disconnected',
     handler: (payload: any) => void): void;
}
```

### 4.1 왜 capability 방식인가

- 큐시트 검증 시 "이 장비가 이 액션을 지원하는가"를 사전에 확인 가능
- VMU는 `video_route`만, AMU는 `audio_route`만, Videohub는 둘 다 선언하는 식으로 자연스럽게 확장
- UI에서 장비 종류에 따라 가능한 액션만 노출

### 4.2 공통 액션 스키마 (라우팅 계열)

```json
{
  "device": "main_router",
  "action": "route",
  "params": {
    "domain": "video",
    "output": 5,
    "input": 2
  },
  "options": {
    "timeoutMs": 2000,
    "retry": 1,
    "onFailure": "warn_and_continue"
  }
}
```

- `device`: 장비 설정 저장소에 등록된 논리 이름 (IP가 바뀌어도 큐시트 불변)
- `domain`: `video` | `audio` — Videohub는 video, AMU는 audio, 복합 장비는 둘 다
- `input`/`output`: 0-base 포트 번호 또는 라벨 문자열 허용 (라벨이면 드라이버가 번호로 해석)
- `onFailure`: `warn_and_continue`(기본) | `pause_cuesheet` — **방송 무중단 원칙**에 따라 기본값은 경고 후 진행

## 5. Phase 1 — Blackmagic Videohub 드라이버

### 5.1 프로토콜 개요 (Videohub Ethernet Protocol)

- **전송**: TCP, 포트 **9990**, 평문 텍스트 프로토콜 (줄바꿈 `\n`, 블록 종료는 빈 줄)
- **연결 직후**: 장비가 프리앰블(프로토콜 버전, 장비 정보, 입출력 라벨, 현재 라우팅 테이블 전체)을 자동 전송
- **명령 예시** — 출력 5번에 입력 2번을 연결 (0-base):

```
VIDEO OUTPUT ROUTING:
5 2

```

- **응답**: 성공 시 `ACK` + 변경된 라우팅 블록 재통지, 실패 시 `NAK`
- **푸시 방식**: 다른 컨트롤러(하드웨어 패널 등)가 라우팅을 바꿔도 변경 블록이 자동 수신됨 → 마스터 UI가 항상 실제 상태와 동기화 가능

> **주의**: 프로토콜 세부(지원 블록, 라벨 변경 명령 등)는 구현 착수 시 Blackmagic 공식 문서 *"Videohub Ethernet Protocol"* (Videohub 매뉴얼 부록)을 기준으로 최종 확인할 것.

### 5.2 드라이버 구현 요구사항

| ID | 요구사항 | 우선순위 |
|---|---|---|
| VH-1 | TCP 연결 및 프리앰블 파싱 (장비 모델, 입출력 수, 라벨, 라우팅 테이블) | P0 |
| VH-2 | `route(output, input)` 실행 및 ACK/NAK 처리 | P0 |
| VH-3 | 상태 푸시 수신 → 내부 상태 갱신 → `stateChanged` 이벤트 발행 | P0 |
| VH-4 | 연결 끊김 감지 및 지수 백오프 자동 재연결 (1s → 2s → 4s → 최대 30s) | P0 |
| VH-5 | 라벨 기반 라우팅 (`"CAM 1" → "vMix IN 3"` 형태를 포트 번호로 해석) | P1 |
| VH-6 | 다중 크로스포인트 일괄 전환 (하나의 명령 블록에 여러 행) — 살보(salvo) 기능의 기반 | P1 |
| VH-7 | 명령 큐잉: 연속 명령 시 이전 ACK 수신 후 다음 전송 (순서 보장) | P1 |
| VH-8 | 오프라인 시 명령 처리 정책: 즉시 실패 반환 + 로그 (큐에 쌓아두지 않음 — 뒤늦은 전환은 방송사고) | P0 |

### 5.3 상태 머신

```
[Disconnected] --connect()--> [Connecting] --preamble 수신--> [Connected]
     ▲                            │ 실패                        │ 소켓 끊김
     │                            ▼                            ▼
     └──── 재연결 포기(수동) ── [Reconnecting] ◄────────────────┘
                                  (지수 백오프, 무한 재시도)
```

- `Reconnecting` 중 큐시트가 해당 장비 액션을 만나면: `onFailure` 정책에 따라 처리하고 **큐시트의 나머지 장비 액션은 정상 진행**

### 5.4 파싱 시 주의점 (테스트에서 검증할 것)

- TCP는 스트림이므로 블록이 패킷 경계와 일치하지 않음 → **빈 줄 기준 블록 단위 버퍼링 파서** 필수
- 라벨에 공백·한글 포함 가능 → 라벨 파싱은 "첫 토큰 = 번호, 나머지 전체 = 라벨"로 처리
- 대형 라우터(40×40 이상)는 프리앰블이 김 → 파서가 블록 수에 무관하게 동작해야 함

## 6. Phase 2 — VMU / AMU 확장

### 6.1 사전 조사 체크리스트 (구현 전 필수)

VMU/AMU는 제조사·모델에 따라 프로토콜이 다르므로, 드라이버 설계 전에 다음을 확정한다:

- [ ] 정확한 제조사/모델명 및 프로토콜 문서 확보
- [ ] 통신 방식: TCP / UDP / RS-422·485 시리얼 / SNMP 여부
- [ ] 크로스포인트 명령 형식과 응답(ACK) 유무
- [ ] 상태 푸시 지원 여부 (미지원 시 폴링 주기 설계 필요)
- [ ] 오디오 매트릭스(AMU)의 경우: 채널 단위 라우팅인지, 스테레오 페어 단위인지, 레벨 제어 포함인지
- [ ] 동시 접속 컨트롤러 수 제한 (기존 하드웨어 패널과 병행 사용 가능한지)

### 6.2 설계 원칙

- Videohub 드라이버에서 만든 **공통 라우팅 액션 스키마(4.2절)를 그대로 재사용**한다. `domain: "audio"`만 다를 뿐 큐시트 작성 방식은 동일해야 한다.
- 상태 푸시가 없는 장비는 드라이버 내부에서 폴링(예: 2초)으로 상태를 갱신하되, 인터페이스상으로는 동일하게 `stateChanged` 이벤트로 노출한다. → 마스터 본체는 푸시/폴링 차이를 모른다.
- 시리얼 전용 장비가 있다면 **시리얼-TCP 게이트웨이(장비 서버)** 사용을 우선 검토 — 마스터가 웹 기반이라면 직접 시리얼 접근이 어렵기 때문 (OnAir Connect의 로컬 에이전트 패턴 재사용 가능)

### 6.3 Sony 등 추가 장비 (장기)

- Sony 라우터/스위처는 별도 제어 프로토콜 체계를 가지므로 문서 확보 후 별도 드라이버로 구현
- 이 단계에서는 **인터페이스가 Sony를 수용할 수 있는지만 검증** (capability에 `switcher_me`, `switcher_key` 등 스위처 계열 액션 추가 여지 확인)

## 7. 큐시트 통합

### 7.1 큐시트 행 확장

기존 OBS/vMix 액션 행과 동일한 구조에 라우터 액션을 추가한다. 예시:

| 순번 | 시각/트리거 | 장비 | 액션 | 파라미터 | 실패 정책 |
|---|---|---|---|---|---|
| 12 | 수동 GO | vmix_main | Cut | Input 3 | pause |
| 13 | +0.5s | main_router | route | CAM 2 → vMix IN 1 | warn_and_continue |
| 14 | +0s | amu_1 | route(audio) | MIC 3 → PGM L/R | warn_and_continue |

### 7.2 큐시트 사전 검증 (마스 → 마스터 인수 시)

방송 시작 전 큐시트 로드 단계에서 자동 검증:

1. 참조된 장비 논리 이름이 모두 등록되어 있는가
2. 각 액션이 해당 장비의 capability에 포함되는가
3. 라벨 기반 파라미터가 실제 장비 라벨과 매칭되는가 (연결된 경우)
4. 검증 실패 항목은 **행 단위로 경고 표시**하되 큐시트 로드 자체는 허용 (리허설 중 장비 미연결 상황 고려)

### 7.3 복합 액션 (매크로/살보)

- "CM 진입" 같은 하나의 큐에 vMix 전환 + 라우터 전환 + AMU 뮤트를 묶는 **매크로**를 지원
- 매크로 내 액션은 병렬 실행이 기본, 필요 시 순차(딜레이 포함) 지정 가능
- 매크로는 마스에서 정의하고 큐시트에서는 매크로 이름으로 참조

| UI-1 | 장비 관리 화면 | 장비 추가/편집(이름, 종류, IP, 포트), 연결 테스트 버튼 |

| UI-3 | 라우팅 매트릭스 뷰 | Videohub 현재 크로스포인트를 그리드로 표시, 수동 전환 가능 (큐시트 외 비상 조작용) |
| UI-4 | 실행 로그 | 액션별 타임스탬프, 소요시간(ms), 성공/실패, 실패 사유 |
| UI-5 | 큐시트 검증 결과 | 로드 시 경고 행 하이라이트 |

## 9. 비기능 요구사항

| 항목 | 기준 |
|---|---|
| 전환 지연 | 큐 실행 → 라우터 명령 송신까지 50ms 이내 (동일 LAN 기준) |
| 재연결 | 장애 복구 후 30초 이내 자동 재연결 및 상태 재동기화 |
| 무중단 | 특정 드라이버 예외가 마스터 본체/타 드라이버에 전파되지 않음 (드라이버 격리) |
| 로그 보존 | 방송 1회분 실행 로그 파일 저장 (사후 방송사고 분석용) |
| 보안 | Videohub 프로토콜은 무인증 평문 → 장비는 **폐쇄 방송망/VLAN에만 배치**, 마스터 외 접근은 방화벽으로 차단. 설정 문서에 명시 |

## 10. 구현 로드맵

### Phase 1 — Videohub (권장 순서)

1. **인터페이스 정의**: `DeviceDriver`, 액션 스키마, Device Manager 골격 (드라이버 0개 상태로 단위 테스트)
2. **Videohub 파서 단독 개발**: 실제 장비 프리앰블을 캡처(예: `nc <ip> 9990 > dump.txt`)해 픽스처로 저장 → 파서를 픽스처 기반 테스트로 먼저 완성
3. **드라이버 연결/명령/재연결** 구현 → 실제 장비로 통합 테스트
4. **큐시트 액션 연동** + 실행 로그
5. **UI**: 상태 표시등 → 매트릭스 뷰 순
6. **리허설 시나리오 테스트** (11절)

### Phase 2 — VMU/AMU

7. 6.1 체크리스트 완료 → 프로토콜 문서 기반 드라이버 설계 리뷰
8. 드라이버 구현 (Videohub와 동일 패턴) + audio domain 큐시트 검증
9. 매크로(복합 액션) 기능

### Phase 3 — 기존 코드 통합 및 확장

10. OBS/vMix를 드라이버 구조로 마이그레이션 (회귀 테스트 필수)
11. Sony 등 추가 장비 검토
