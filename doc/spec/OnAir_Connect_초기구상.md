조사해보니 vMix 자체에도 "Web Controller"라는 기능이 있긴 한데, 그것 때문에 오히려 불편 포인트가 더 명확하게 드러납니다. 지금 만드시는 "웹에서 편하게, vMix 설정 없이"라는 방향이 정확히 이 틈새를 파고드는 거예요.

확인된 핵심 불편 포인트
1. 원격 접속이 사실상 "같은 네트워크" 안에서만 됨
vMix 포럼에 반복적으로 올라오는 질문이 "같은 네트워크가 아닌 곳에서 웹 컨트롤러 접속이 되나요?"입니다. 기본적으로 인터넷을 통한 외부 접속 자체가 막혀 있고기본적으로 활성화되어 있어 웹 및 TCP API에 대한 모든 인터넷 접근을 차단하며, 원격 접속이 필요하면 이 옵션을 끄기 전에 반드시 방화벽과 적절한 보안 정책을 갖추라고 강력히 권고합니다. 즉 집에서 교회 PC를 제어하려면 포트포워딩·방화벽 설정을 직접 해야 하는데, 이게 비전문가에게는 거의 불가능한 장벽입니다. 
vMix

2. "설정"과 "제어"가 분리되지 않음
웹 컨트롤러는 어디까지나 데스크탑 vMix 안에서 미리 만들어둔 단축키(Shortcut)를 누르는 용도지, 새 단축키를 웹에서 만들 수는 없습니다숏컷 화면을 보면 프로덕션에서 설정해둔 모든 단축키가 나타나는데, 필요한 단축키를 미리 설정해두면 그 버튼들만으로 전체 프로덕션을 제어할 수 있어 매우 편리하다는 게 vMix의 설계 전제입니다 — 결국 "누군가는 데스크탑 vMix를 열고 사전 설정을 해야 한다"는 전제가 그대로 남아있는 거죠. 학생/교회/유튜버 입장에선 이 사전 설정 자체가 진입장벽입니다. 
vMix

3. UI가 초보자용으로 재설계된 게 아니라 데스크탑의 축소판
웹 컨트롤러는 스마트폰의 가장 작은 화면에서도 쓸 수 있도록 압축된, 실제 vMix 인터페이스의 축소 버전입니다. 즉 vMix 용어(Input, Overlay, PGM/PRV, Stinger 등)를 그대로 노출하는 구조라, "우리 교회 카메라 이름"이 아니라 "Input 3" 같은 식으로 보입니다. 
vMix

4. 방화벽/포트/링크 관련 잡다한 실패 사례가 많음
포럼에 "웹 컨트롤러 링크가 안 열려요", "듀얼 네트워크 카드 환경에서 안 됨" 같은 글이 꾸준히 올라옵니다 — 네트워크 지식이 없는 사용자가 부딪히는 전형적인 실패 지점입니다.

5. 보안 설정을 수동으로 해야 함
비밀번호 보호나 페이지별 접근 권한도 설정 탭에서 비밀번호를 지정해 활성화해야 하는데, 안 하면 비밀번호가 비어 있으면 접근 설정과 무관하게 로그인이 전혀 요구되지 않는 위험한 기본값이라 보안 사고 가능성도 있습니다.

1. 원격 통신 구조의 취약 포인트
지금 아키텍처는 대략 [웹 브라우저] ↔ [클라우드 릴레이 서버] ↔ [로컬 에이전트] ↔ [vMix TCP/HTTP API] 4단 구조가 될 텐데, 구간마다 위험이 다릅니다.

① 로컬 에이전트 ↔ vMix API 구간
vMix는 기본적으로 Web/TCP API에 대한 외부 인터넷 접근을 차단하도록 설계되어 있고, 이걸 열려면 방화벽 정책을 직접 갖춰야 한다고 vMix 스스로 경고합니다. 즉 vMix API 자체는 인증 메커니즘이 거의 없는 로컬 신뢰 기반 프로토콜입니다. 이 말은 에이전트가 vMix와 통신하는 포트가 로컬 밖으로 절대 노출되면 안 되고, 에이전트는 반드시 localhost 또는 vMix가 설치된 그 PC 안에서만 API를 호출해야 한다는 뜻입니다. (원격 접속은 에이전트→클라우드 구간에서 처리하고, vMix API 구간은 격리)

② 로컬 에이전트 ↔ 클라우드 릴레이 구간 (가장 위험한 신규 구간)

에이전트 위장(impersonation): 페어링 시 발급하는 토큰/코드가 재사용 가능하거나 만료가 없으면, 탈취된 코드로 누군가 다른 사람의 교회/학교 vMix를 조작할 수 있습니다. 페어링 코드는 1회성·단시간 만료가 필수입니다.
세션 하이재킹: WebSocket 연결 토큰이 길게 살아있으면, 토큰 유출 시 공격자가 생방송 도중 화면을 임의로 전환하거나 자막을 바꿔치기할 수 있습니다(방송 사고로 직결). 짧은 TTL + 리프레시 구조가 필요합니다.
MITM: 반드시 wss://(TLS)만 허용, 가능하면 데스크탑 에이전트에 인증서 핀닝을 걸어 가짜 릴레이 서버로의 우회 연결을 막아야 합니다.
릴레이 서버 단일장애점: 릴레이가 다운되면 전국 모든 고객의 원격 제어가 동시에 마비됩니다. 데이터 보안뿐 아니라 "생방송 중 서비스 가용성"이 곧 보안 이슈와 동급으로 취급돼야 합니다.
③ 웹 브라우저 ↔ 릴레이 구간

멀티테넌시 권한 검사: 가장 치명적인 버그 유형은 "내 계정으로 로그인했는데 다른 교회의 vMix 인스턴스를 제어할 수 있는" 상황입니다. 프론트엔드에서 버튼을 숨기는 것만으로는 부족하고, 서버가 매 명령마다 "이 사용자가 이 device_id를 소유했는가"를 반드시 재검증해야 합니다.
XSS/토큰 저장: 세션 토큰을 localStorage에 두면 XSS 한 방에 탈취되니 httpOnly 쿠키 + CSRF 토큰 조합을 권장합니다.
④ 명령 인젝션
자막 텍스트, 커스텀 함수 호출 필드처럼 사용자가 직접 입력하는 값을 그대로 vMix API 명령 문자열에 이어붙이면, 입력값 안에 추가 API 파라미터를 끼워넣어 의도치 않은 함수(녹화 중지, 스트리밍 종료, 심지어 Quit 등)를 호출당할 수 있습니다. 화이트리스트 기반 함수 매핑이 필수입니다.

⑤ 민감정보(RTMP 스트림 키)
멀티스트리밍 기능을 넣는다면 유튜브/치지직 스트림 키는 절대 브라우저로 평문 전달·로그 기록하면 안 됩니다. 서버/에이전트 단에서만 다루고 브라우저에는 마스킹된 값만 노출하세요.

⑥ 자동 업데이트 공급망
에이전트가 자동 업데이트 기능을 가진다면, 업데이트 바이너리에 코드 서명을 반드시 걸어야 합니다. 안 그러면 교회/학교 PC에 악성코드를 심는 공급망 공격 경로가 생깁니다.

⑦ 오프라인 폴백
인터넷이 끊겨도 같은 네트워크 안에서는 기존처럼 로컬 제어가 계속 되도록 설계하는 게 좋습니다. 예배/행사 도중 인터넷 장애로 방송이 완전히 멈추면 신뢰를 잃습니다.

2. 구독/라이선스 모델 설계 시 중요한 점
① 라이선스 바인딩 단위 결정
지금 구조(로컬 에이전트 + vMix PC)에서는 사용자 계정 단위보다 "디바이스/에이전트" 단위 바인딩이 자연스럽습니다 — vMix 자체도 PC 단위 라이선스이므로 이 관행과 맞습니다. 다만 교회/학교는 담당자가 자주 바뀌므로 "기관 계정 아래 여러 디바이스"를 등록하는 사이트 라이선스형이 사용자 단위보다 마찰이 적습니다.

② 오프라인 유예(grace period)가 필수
라이브 방송/예배 도중 라이선스 서버 통신이 실패한다고 서비스가 멈추면 치명적입니다. 라이선스 검증은 "매 명령마다"가 아니라 로컬에 서명된 라이선스 파일을 캐싱해두고 주기적(예: 24~72시간)으로만 재검증하며, 실패 시에도 일정 기간 계속 동작하게 설계해야 합니다. 소프트웨어 라이선싱 업계에서도 오프라인/에어갭 환경 지원 여부를 필수 평가 항목으로 꼽습니다.

③ 결제 실패 시 즉시 차단 금지
일반 SaaS는 결제 실패 시 바로 서비스를 끊지만, 이 제품은 "일요일 예배 30분 전에 결제 실패로 잠김" 같은 시나리오가 최악의 이탈 요인이 됩니다. 던닝(dunning) 유예 기간을 넉넉히 잡고, 잠기더라도 로컬 LAN 제어만은 살려두는 등 단계적 제한이 안전합니다.

④ 요금제 계층은 "원격 여부"로 가르는 게 자연스러움
로컬 네트워크 제어는 무료/기본 티어, 원격(클라우드 릴레이) 접속과 다중 기기 동시 세션은 유료 티어로 나누는 구조가 이 제품 특성과 잘 맞습니다. 동시 접속 세션 수, 등록 가능 vMix 인스턴스 수를 요금제 차등 기준으로 쓰면 관리가 쉽습니다.

⑤ 체험판 어뷰징 방지
무료 체험을 기기 지문(하드웨어 ID) 기반으로 제한하지 않으면 재설치로 무한 체험이 가능해집니다.

⑥ 직접 구현 vs 라이선싱 플랫폼
자체 개발(JWT 기반 entitlement + 서명된 오프라인 라이선스 파일)로 충분히 시작 가능하지만, 나중에 규모가 커지면 Keygen, Cryptlex 같은 전문 라이선싱 SaaS로 이전하는 경로도 고려해두면 초기 설계 부담이 줄어듭니다.

우선순위를 매기자면: ②(멀티테넌시 권한 검사)와 ②(오프라인 유예) 이 두 가지가 지금 단계에서 가장 먼저 설계에 반영해야 할 항목입니다. 하나는 보안 사고, 하나는 신뢰 상실로 직결되거든요. 페어링/인증 흐름을 구체적으로 어떻게 짜고 계신지 알려주시면 그 부분 설계를 더 파고들어 드릴 수 있어요.
