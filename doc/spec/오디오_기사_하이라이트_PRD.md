# 실시간 오디오 인식 기반 기사 하이라이트 기능 PRD

## 0. 문서 정보
- 작성일: 2026-07-22
- 상태: Draft v0.1
- 대상 플랫폼: MARS/Master 뉴스 큐시트 시스템 — 기존 기사매칭/프롬프터/자막 IN-OUT 인프라(뉴스큐시트 아이템 편집창)의 확장 기능

## 1. 개요

뉴스 진행 중 진행자가 읽는 기사 오디오를 실시간으로 인식하여, 사전에 확보된 기사 원문 텍스트 상에서 현재 낭독 중인 대략적인 위치를 자동으로 하이라이트 표시하는 기능. 큐시트에 등록된 뉴스 아이템의 기사를 사전에 로드해 두고, 방송 진행에 맞춰 실시간으로 해당 기사에 표시를 갱신한다.

## 2. 배경 및 문제 정의

- 현재는 진행자가 어느 문장을 읽고 있는지 스태프가 수동으로 화면을 넘기거나 표시해야 함
- 실시간 STT(음성 인식) 자체에는 필연적으로 지연(delay)이 발생하며, 이를 완전히 제거하는 것은 불가능
- 완벽한 싱크보다는 "평균 지연 시간을 사전 보정하여 대략적인 위치"를 안정적으로 보여주는 것이 실사용에 더 적합

## 3. 목표 (Goals)

- 큐시트 아이템에 연결된 기사 원문을 방송 전 미리 로드
- 방송 중 마이크(진행자) 오디오를 실시간 STT로 변환
- 변환된 텍스트를 원문 기사와 매칭하여 현재 낭독 위치를 추정
- 평균 지연 시간만큼을 사전 보정하여 하이라이트 위치를 조정
- 문서 화면에 현재 위치를 시각적으로 하이라이트

## 4. 비목표 (Non-goals)

- 100% 정확한 워드 단위 싱크 (지연 보정을 적용한 "대략적 위치"가 목표)
- 진행자 발화 내용의 교정/자동 수정
- 기사 원문과 실제 발화가 크게 다를 경우(애드리브, 생략)의 완전한 대응 — 별도 fallback 정책으로 처리

## 5. 사용자 흐름

1. 방송 전: 큐시트에서 해당 뉴스 아이템의 기사 원문을 시스템에 로드
2. 기사 원문을 문장/구간 단위로 사전 분절(인덱싱)
3. 방송 시작, 진행자가 기사를 낭독
4. 마이크 오디오가 실시간 STT 엔진으로 전송되어 텍스트 스트림 생성
5. 생성된 텍스트 조각을 원문과 매칭 → 대략적 위치 계산
6. 평균 지연 시간만큼 보정 적용
7. 문서 화면(오퍼레이터/서브 모니터 등)에서 해당 위치가 하이라이트로 표시되고, 낭독이 진행됨에 따라 순차적으로 이동
8. 해당 아이템 종료 시 다음 큐시트 아이템의 기사로 자동 전환

## 6. 핵심 기능 요구사항

### 6.1 큐시트 연동 및 사전 로딩
- 큐시트 아이템과 기사 원문 텍스트를 매핑하여 관리
- 현재 진행 중인 아이템을 자동 감지하거나 수동 선택 가능
- 기사 원문은 문장 단위로 분절, 각 문장에 순번(인덱스) 부여

### 6.2 실시간 오디오 인식(STT)
- 마이크 입력을 실시간 스트리밍으로 STT 엔진에 전달
- 짧은 청크(예: 1~2초) 단위로 부분 인식 결과(partial transcript) 수신
- 한국어 뉴스 낭독 특성(속도, 발음)에 적합한 STT 엔진 선정 필요

### 6.3 텍스트 매칭 및 위치 추정
- STT로 인식된 텍스트 조각을 원문 기사와 비교하여 가장 유사한 위치를 탐색 (fuzzy matching / sliding window 방식)
- 노이즈, 오인식, 조사 누락 등을 감안한 유사도 기반 매칭 (완전 일치가 아닌 근사 매칭)
- 이전 매칭 위치를 기준으로 탐색 범위를 좁혀 처리 속도와 정확도 향상 (역행 방지 로직 포함)

### 6.4 지연 보정
- STT 처리 지연 + 매칭 처리 지연을 사전 측정하여 평균 지연 값 산출
- 하이라이트 표시 시점에 평균 지연만큼 보정하여, 실제 낭독 위치에 가깝게 표시되도록 조정
- 지연 값은 설정 가능한 파라미터로 운영 중 조정 가능해야 함

### 6.5 하이라이트 표시 UI
- 현재 낭독 중으로 추정되는 문장(또는 구간)을 문서 화면에서 시각적으로 강조
- 낭독이 진행됨에 따라 하이라이트가 자연스럽게 다음 문장으로 이동
- 매칭 실패/불확실 구간에 대한 표시 방식 필요 (예: 이전 위치 유지 또는 옅은 표시)

## 7. 기술 아키텍처 (제안)

```
[마이크 입력]
   → [실시간 STT 엔진] (스트리밍 인식, partial transcript)
   → [텍스트 매칭 모듈] (원문 기사 인덱스와 유사도 매칭)
   → [지연 보정 모듈] (평균 지연 값 적용)
   → [하이라이트 렌더링] (문서 화면 갱신)

[큐시트 시스템] → [기사 원문 사전 로딩 및 문장 분절 인덱스 생성]
```

- STT 엔진: 스트리밍 지원 + 한국어 인식 정확도를 기준으로 후보 비교 필요 (클라우드형 vs 온프레미스형)
- 매칭 알고리즘: 문자열 유사도 기반(예: 편집 거리, n-gram 매칭) 슬라이딩 윈도우
- 실시간 처리를 위한 파이프라인은 비동기/스트리밍 구조로 설계

## 7-1. 하드웨어/인프라 요구사항

실시간 처리를 위해서는 STT 방식에 따라 요구되는 하드웨어 조건이 크게 달라진다. 크게 "클라우드 STT API 사용" 방식과 "온프레미스(자체 GPU 서버) STT 구동" 방식 두 가지로 나눠 검토할 수 있다.

### (A) 클라우드 STT API 방식 (예: 네이버 클로바 스피치, Google/Azure STT 등)
- 네이버 클로바 스피치의 경우 실시간 스트리밍 자막 기능을 이미 상용 제공 중이며, 어절 단위로 즉시 텍스트를 추출하는 것이 특징
- 이 방식은 별도의 고성능 GPU 서버가 필요 없고, 안정적인 네트워크 연결(방송 스튜디오 → 클라우드) 및 오디오 입력 인터페이스만 있으면 됨
- **최소 요구 사양**
  - 일반 사무용 PC/서버 수준(멀티코어 CPU, 8~16GB RAM)으로 충분 — STT 연산 자체는 클라우드에서 처리
  - 안정적인 유선 인터넷 회선 (지연/끊김 최소화가 중요, 방송용은 이중화 회선 권장)
  - 오디오 인터페이스: 마이크 오디오를 네트워크로 스트리밍할 수 있는 캡처 장비(예: 오디오 인터페이스 + 인코더)
- **장점**: 하드웨어 투자 부담 적음, 유지보수 용이, 한국어 인식 정확도 검증됨(뉴스 자막 생성에 이미 활용 중인 사례 있음)
- **단점**: 네트워크 상태에 따라 지연이 변동될 수 있음, 사용량 기반 과금, 방송 데이터가 외부로 전송됨(보안/내부정책 검토 필요)

### (B) 온프레미스 GPU 서버 방식 (예: Whisper 계열 자체 구동)
자체 서버에서 STT를 구동하려면 실시간(RTF < 1, 체감상 여유를 두려면 RTF 0.2~0.5 이하) 처리가 가능한 GPU가 필요하다.

- **모델 크기별 GPU 참고 기준** (Whisper Large-v3 기준, faster-whisper/INT8 양자화 사용 시)
  - 최소 사양: VRAM 6GB급 GPU (예: RTX 3060 12GB) — 실시간 처리 가능하나 여유가 크지 않음
  - 권장 사양: VRAM 12~24GB급 GPU (예: RTX 4070~4090, RTX 3090) — 안정적 실시간 처리 + 여유 마진 확보, 동시 다중 스트림 처리 가능
  - 다중 진행자/다채널 동시 처리가 필요하면 VRAM 24GB 이상 또는 GPU 2장 이상 구성 고려
- **본 서비스 특성상 고려사항**
  - 뉴스 진행자 1인 음성만 실시간 처리하는 단일 스트림 기준이라면 중급 GPU(RTX 4060~4070Ti급, VRAM 12GB)로도 충분히 실시간 처리 가능
  - 정확도(전문용어/고유명사 인식)를 높이기 위해 Large 계열 모델을 쓸 경우 VRAM 여유를 더 확보하는 것이 안전
  - CPU만으로는 Large 모델 기준 실시간 처리가 어려우므로 GPU 필수
- **서버 구성(참고)**
  - CPU: 8코어 이상 (오디오 전처리, 매칭 로직, 렌더링 파이프라인 동시 처리)
  - RAM: 32GB 이상 권장
  - 저장장치: NVMe SSD (모델 로딩/캐시 속도)
  - 네트워크: 방송 시스템(큐시트, 문서 표시 장치)과의 내부망 연결 안정성 확보
- **장점**: 데이터가 외부로 나가지 않음(보안), 네트워크 상황에 지연이 좌우되지 않음, 장기적으로 사용량 과금 부담 없음
- **단점**: 초기 하드웨어 투자 비용, GPU 서버 운영/유지보수 필요, 모델 업데이트/튜닝을 자체적으로 관리해야 함

### (C) 채택 방향 (수정): 오픈소스 경량 STT 우선 검토
본 기능은 "정확한 전사"가 아니라 "대략적 위치 추정"이 목적이므로, 인식 정확도가 다소 낮아도 무방하다. 이 경우 네이버 클로바 스피치 같은 종량제 클라우드 API보다, 무료 오픈소스 경량 STT를 자체 서버(CPU)에서 구동하는 방식이 비용 효율이 훨씬 높다. 클로바 스피치 구현 방안은 참고용으로 7-2에 남겨두고, 가성비 대안은 7-3을 참고.

## 7-2. 네이버 클로바 스피치(CLOVA Speech) 구현 방안

### 7-2-1. 서비스 선택
- 네이버 클라우드는 STT로 **CLOVA Speech**와 **CLOVA Speech Recognition(CSR)** 두 가지를 제공
  - CSR: 60초 이내의 짧은 명령형 음성 인식에 최적화 (비서/챗봇용) → 본 기능에는 부적합
  - CLOVA Speech: 장문 인식 + **실시간 스트리밍 인식** 기능 제공, 문장 자동 분리·타임스탬프 지원 → 본 기능에 적합
- 따라서 CLOVA Speech의 "실시간 스트리밍 인식"(gRPC 기반) 기능을 사용

### 7-2-2. 사전 준비 절차
1. 네이버 클라우드 플랫폼(NCP) 콘솔 가입 및 CLOVA Speech 서비스 이용 신청
2. 콘솔에서 Application 등록 → 사용할 API(실시간 스트리밍 포함)가 선택되어 있는지 확인 (미선택 시 429 Quota Exceed 오류 발생)
3. 인증 정보 발급: Client ID / Client Secret(Secret Key)
4. 콘솔 Domain 설정에서 **InvokeURL** 확인 (API 호출에 사용)
5. Classic/VPC 중 이용 환경 결정 (실시간 스트리밍은 두 환경 모두 지원)

### 7-2-3. 인증 및 연결 방식
- REST 기반 기능(장문 인식 등): Client ID + Client Secret을 요청 헤더에 포함해 인증
- **실시간 스트리밍 인식**: gRPC 프로토콜 사용, Secret Key 기반 인증
- 연결 관리 권장사항(네이버 공식 가이드 기준)
  - 하나의 음성 인식 스트림(예: 하나의 뉴스 아이템 낭독 구간) 동안 gRPC 채널 커넥션을 유지
  - 예: 라이브 방송이라면 방송 종료 시점에 `channel.shutdown`/`channel.close`로 커넥션 종료
  - 송수신 상황에 따른 timeout 로직 적용 권장 (연결 유실 대비)

### 7-2-4. 오디오 캡처 → 인식 → 매칭 파이프라인
1. **오디오 캡처**: 진행자 마이크(또는 믹서 출력)를 오디오 인터페이스로 캡처
2. **청크 전송**: 캡처된 PCM 오디오를 일정 크기(수백 ms 단위)로 잘라 gRPC 스트림을 통해 CLOVA Speech로 순차 전송 (지원 오디오 포맷/샘플레이트는 최신 API 가이드에서 확인 필요)
3. **실시간 인식 결과 수신**: 서버에서 어절 단위로 형성된 텍스트를 스트리밍 응답으로 수신, 문장 자동 분리 및 타임스탬프 활용
4. **텍스트 매칭 모듈 연동**: 수신된 인식 텍스트 조각을 사전 로딩된 기사 원문과 실시간 비교(fuzzy matching, sliding window)하여 대략적 위치 추정
5. **지연 보정 적용**: (오디오 캡처~인식 결과 수신까지 걸리는 네트워크·처리 지연)을 사전 실측한 평균값으로 보정하여 하이라이트 위치 조정
6. **하이라이트 렌더링**: 추정 위치를 문서 화면에 반영

### 7-2-5. 인프라 구성 (클로바 방식 기준)
- 고성능 GPU 서버 불필요 — STT 연산은 전량 네이버 클라우드에서 처리
- 필요한 자체 구성 요소
  - 오디오 캡처 및 gRPC 클라이언트 역할을 수행할 경량 서버/PC (일반 사무용 사양으로 충분)
  - 매칭 로직 + 하이라이트 렌더링 서비스 (동일 서버에 통합 가능)
  - 안정적인 유선 인터넷 회선 (방송 사고 예방을 위한 이중화 회선 권장)
  - 필요시 사내망-NCP 간 전용선(Direct Connect) 검토로 지연 변동성 최소화

### 7-2-6. 비용 구조
- CLOVA Speech는 사용 시간 기준 종량 과금 (예: 화자 인식 미사용 시 10초당 5원 수준 — 실제 요금은 변동될 수 있으므로 콘솔에서 최신 요금표 확인 필요)
- 일일 방송 시간 × 요금 단가로 월 예상 비용을 사전 산정하여 운영 예산에 반영

### 7-2-7. 리스크 및 대응 방안
- gRPC 연결 끊김/네트워크 장애 대비 자동 재연결 로직 필요
- 장애 발생 시 fallback: 직전 하이라이트 위치 유지 또는 수동 전환 모드로 전환
- 실제 방송 투입 전 리허설을 통해 지연 시간을 실측하고, 7-2-4의 보정값을 튜닝
- 방송 데이터가 외부(네이버 클라우드)로 전송되므로, 보안/내부 정책 검토 및 필요시 데이터 처리 계약 확인

## 7-3. 가성비 대안 비교 (정확도보다 "위치 추정"이 목적인 경우)

본 기능은 완벽한 전사(transcription)가 아니라 "대략적인 낭독 위치 추정"이 목적이므로, 반드시 클로바 스피치 수준의 최고 정확도가 필요하지 않다. 이 경우 비용을 크게 낮출 수 있는 대안이 있다.

### 비교 요약

| 옵션 | 비용 구조 | 정확도 | 인프라 | 비고 |
|---|---|---|---|---|
| **네이버 클로바 스피치** | 종량제, 10초당 약 5원 (시간당 약 1,800원 수준) | 매우 높음 | GPU 불필요 | 최고 정확도가 필요할 때만 적합, 위치 추정용으로는 과함 |
| **RTZR STT (리턴제로)** | 월 10시간 무료, 이후 시간당 약 1,000원 | 높음 (자체 벤치마크상 국내 상위권) | GPU 불필요, 클라우드/온프레미스 선택 가능 | gRPC/WebSocket 실시간 스트리밍 지원, 클로바 대비 저렴 |
| **Vosk (오픈소스, 자체 구동)** | **무료** (라이선스 비용 없음, 인프라 비용만 발생) | 중간 (소형 모델 기준, 완벽한 전사엔 부족) | CPU만으로 구동 가능 (GPU 불필요) | 오프라인 동작, 위치 추정처럼 "대략적 정합"이 목적인 용도에 적합 |
| **Whisper 소형 모델 자체 구동 (tiny/base, faster-whisper)** | **무료** (인프라 비용만) | 중간~높음 | 저사양 GPU 또는 고성능 CPU | Vosk보다 정확도는 좋으나 약간의 리소스 필요 |

### 권장: Vosk 기반 자체 구동으로 전환
- **한국어 경량 모델**(`vosk-model-small-ko-0.22`)이 공개되어 있으며, CPU만으로 실시간 스트리밍 인식이 가능
- 완전 오프라인 동작 → 네트워크 지연/장애 리스크 없음, 방송 데이터가 외부로 전송되지 않아 보안 이슈 해소
- 사용량과 무관하게 **추가 과금 없음** (일회성 인프라 구축 비용만 발생)
- 단, 정확도는 클로바/RTZR 대비 낮으므로, 매칭 모듈에서 fuzzy matching 허용 오차를 넉넉하게 설정해야 함 (애초에 "정확한 문장 인식"이 아니라 "근사 위치 추정"이 목적이므로 실사용상 문제가 크지 않을 가능성이 높음)
- 실사용 리허설을 통해 매칭 정확도/지연 특성을 검증한 뒤, 부족하면 RTZR STT(저비용 유료 옵션)로 단계적 업그레이드하는 방식을 권장

### 단계별 전환 전략
1. **1단계 (PoC)**: Vosk 소형 한국어 모델로 자체 구동 프로토타입 제작, 매칭 정확도/지연 실측 → 비용 없이 빠른 검증
2. **2단계**: PoC 결과 정확도가 부족하면 RTZR STT(월 10시간 무료, 이후 저렴한 종량제)로 전환 검토
3. **3단계(선택)**: 그래도 부족하면 클로바 스피치 등 고정확도 유료 API 사용 (최후 옵션, 비용 부담 가장 큼)

## 7-3. 가성비 대안: 오픈소스 경량 STT (권장)

"정확한 문장 전사"가 아니라 "원문 기사 내 대략적 위치 추정"이 목적이라는 점을 고려하면, 인식 정확도(WER)를 다소 희생하더라도 완전 무료·자체 구동 방식이 훨씬 합리적이다. 클로바 스피치는 시간당 종량 과금이라 방송 시간이 누적될수록 비용이 계속 늘어나지만, 아래 방식은 초기 서버 구축 이후 추가 비용이 거의 발생하지 않는다.

### 7-3-1. 후보 비교

| 구분 | Vosk (경량 오프라인) | Whisper 경량 모델 (tiny/base/small, whisper.cpp·faster-whisper) | 네이버 클로바 스피치 (비교용) |
|---|---|---|---|
| 비용 | 완전 무료(오픈소스) | 완전 무료(오픈소스) | 시간당 종량 과금 |
| 실행 방식 | 완전 오프라인, 네이티브 스트리밍 지원 | 청크 단위 스트리밍(약간의 지연 존재) | 클라우드 gRPC 스트리밍 |
| 하드웨어 | CPU만으로 충분, 초경량(모델 약 50MB) | CPU로도 실시간 가능(tiny/base/small), 구형 PC도 가능 | 없음(클라우드 처리) |
| 한국어 지원 | `vosk-model-small-ko` 등 한국어 모델 제공 | 다국어 모델에 한국어 포함, 정확도는 모델 크기에 비례 | 한국어 특화, 정확도 최상 |
| 정확도(WER) | 중간 수준 (일상 대화 기준 사용 가능한 수준) | tiny/base는 낮음~중간, small은 준수한 수준 | 매우 높음 |
| 지연 | 매우 낮음(네이티브 스트리밍, 거의 즉시 부분 결과 반환) | 청크 크기에 따라 수백 ms ~ 1~2초 | 낮음(클라우드 처리, 네트워크 지연 포함) |
| 보안 | 오디오 데이터가 외부로 전송되지 않음 | 오디오 데이터가 외부로 전송되지 않음 | 데이터가 네이버 클라우드로 전송됨 |

**위치 추정용으로는 Vosk 한국어 경량 모델(`vosk-model-small-ko`) 또는 Whisper small(faster-whisper/whisper.cpp, CPU 구동)을 1순위로 권장.** 뉴스 기사처럼 정형화된 문어체 원고를 낭독하는 상황은 일상 대화보다 인식 난이도가 낮아, 경량 모델로도 fuzzy matching에 필요한 수준의 텍스트 조각은 충분히 확보 가능하다.

### 7-3-2. 권장 구성: Vosk 기반
- 모델: `vosk-model-small-ko-0.22` (약 50MB, 한국어 경량 모델)
- 실행: Python `vosk` 패키지 + `KaldiRecognizer`로 마이크 오디오를 실시간 스트리밍 인식
  - 마이크 입력 → 16kHz PCM 청크 → `rec.AcceptWaveform(chunk)` → 부분/최종 인식 결과(JSON) 수신
- 네이티브 스트리밍 API를 제공하므로 별도의 청크 버퍼링 설계 부담이 적음
- 완전 오프라인 동작이므로 인터넷 회선 품질에 지연이 좌우되지 않음(방송 안정성 측면에서 유리)

### 7-3-3. 권장 구성: Whisper 경량 모델 기반 (정확도를 좀 더 확보하고 싶을 때)
- 모델: `small` 또는 `base` (다국어, 한국어 포함), faster-whisper(CTranslate2) 또는 whisper.cpp 런타임 사용
- CPU만으로도 최신 프로세서 기준 실시간 처리 가능 (RAM 2~4GB 수준이면 충분)
- Vosk보다 인식 품질이 높은 편이나, 청크 분할 방식 특성상 약간의 지연(수백 ms ~ 1~2초)이 발생 → 6장의 "지연 보정" 값에 반영 필요
- 정확도가 부족하면 `medium`으로 상향 검토 가능하나, 이 경우 GPU가 있으면 더 안정적(7-1의 온프레미스 GPU 섹션 참고)

**프로토타입 구현 완료 → 독립 테스트 도구로 승격**: 이 경로(faster-whisper 기반)로 CLI 프로토타입이 먼저 작성됐고(`main.py`/`stt_stream.py`/`matcher.py`), 이후 마이크 입력 + 웹 하이라이트 UI를 갖춘 독립 실행 도구로 확장돼 저장소 루트의 [`audio-highlight/`](../../audio-highlight/README.md)(문서 폴더 밖 — 실제 실행 코드이므로 소스 저장소에만 존재, 공개 문서 사이트에는 배포되지 않음)에 있다. `matcher.py`가 6.3절의 "현재 위치 근처 문장만 탐색(forward_window) + 유사도 임계값 미만이면 이전 위치 유지(fallback)" 판정 로직을 그대로 구현한다. Vosk PoC를 거치지 않고 바로 이 경로로 착수된 상태이며, `python server.py` 실행 후 브라우저(`localhost:8787`)에서 마이크로 직접 낭독하며 실시간 하이라이트를 확인할 수 있다 — 실행 방법은 `audio-highlight/README.md` 참고.

### 7-3-4. 인프라 및 비용
- 서버: 기존 사무용 PC 또는 미니 서버(8코어 CPU, RAM 16GB 내외)로 충분 — 별도 GPU, 클라우드 API 계약 불필요
- 네트워크: 외부 API 호출이 없으므로 인터넷 회선 품질과 무관하게 지연이 안정적
- 비용: 초기 서버(이미 보유 장비 활용 가능) 외 추가 운영비 사실상 0원 — 클로바 스피치처럼 방송 누적 시간에 비례해 비용이 늘어나지 않음
- 보안: 오디오/기사 데이터가 외부로 전송되지 않아 사내 정책상으로도 유리

### 7-3-5. 트레이드오프 및 대응
- 인식 오류(오탈자, 고유명사 오인식) 가능성이 클로바 스피치 대비 높음 → 매칭 알고리즘에서 fuzzy matching 허용 범위를 넉넉히 잡아 보완
- 잡음이 많은 환경(스튜디오 배경음 등)에서는 경량 모델일수록 정확도 저하 폭이 큼 → 마이크 품질/노이즈 게이팅으로 보완
- 최종 검증: 실제 뉴스 원고 낭독 샘플로 Vosk와 Whisper small을 함께 테스트하여 매칭 성공률을 비교 후 택일 권장

## 8. 비기능 요구사항

- **지연 시간**: STT + 매칭 + 렌더링까지 전체 파이프라인 지연을 측정하고 목표치 설정 필요 (실사용 테스트를 통해 평균 지연 값 산출)
- **정확도**: 문장 단위 위치 추정 정확도 목표 설정 (완전 일치가 아닌 "근접 구간" 기준)
- **안정성**: 방송 중 STT 엔진 오류/네트워크 문제 시 이전 하이라이트 위치 유지(fallback)
- **자원 사용**: 실시간 처리이므로 CPU/네트워크 부하를 고려한 최적화 필요

## 9. 리스크 및 제약

- 진행자가 원고를 그대로 읽지 않고 애드리브를 하는 경우 매칭 정확도 저하 가능
- STT 인식 오류(전문 용어, 고유명사)로 인한 매칭 실패 가능성
- 네트워크 기반 STT 사용 시 지연 변동성 존재 → 평균값 보정의 한계
- 큐시트와 기사 원문 매핑이 사전에 정확히 되어있지 않으면 기능 전체가 무력화됨

## 10. 성공 지표 (안)

- 하이라이트 위치와 실제 낭독 위치 간 평균 오차(문장 수 기준)
- 매칭 실패율(원문과 매칭되지 않는 발화 비율)
- 전체 파이프라인 평균 지연 시간
- 실사용(스태프) 만족도/운용 편의성

## 11. 향후 확장 가능성

- 다수 진행자/출연자 음성 분리(화자 분리) 후 각각 매칭
- 애드리브 구간 자동 감지 및 별도 처리
- 매칭 실패 시 자동 학습을 통한 정확도 개선
