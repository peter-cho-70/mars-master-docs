Recurring Failure Patterns

같은 모양의 오류가 반복해서 발견됐습니다

30개에 가까운 세션 기록을 모아보면, 오류는 무작위가 아니라 6가지 정도의 반복되는 형태로 나뉩니다. 빈도가 높을수록 "한 번 고쳤다고 끝나지 않았던" 문제입니다.

01 안전해 보이는 기본값이 진짜 문제를 가림 8회 이상 재발

값이 없을 때 조용히 다른 값으로 대체하는 코드(x || fallback)가, 실제로는 "설정이 안 됐다"는 신호를 지워버려서 완전히 다른 곳에서 엉뚱한 증상으로 재발했습니다.

  • 전환효과(전환In/Out) || 'Cut' 폴백 — openItemModal, structureItemToCueItem, cuesheetStructureOnly, Master openItemEditor, toMasterCuesheet/importMasterCuesheet, MARS 자체 ITEM_TYPES 사본, NPS_NEW_ITEM_DEFAULTS['타'], 모호한 '특' 코드까지 — 하루 사이 총 8곳 이상이 개별적으로 발견됨
  • _vmixSrvInput(n)의 map['srv'+n] || map.srv1 — SRV2/3 미설정 시 조용히 SRV1로 대체되어 라디오 모드와 물리적으로 같은 인풋을 충돌시킴
  • 오늘 발견: get_local_ip()가 VPN 인터페이스를 기본값으로 골라 vMix가 접근 못 하는 CG 오버레이 URL을 등록할 뻔함
비용: 전환효과 폴백 하나만으로 2026-07-04~05 사이 최소 3차례의 별도 "수정 완료" 선언이 있었고, 그중 하나는 실제 방송 도중 사용자가 직접 발견해 보고한 건이었습니다.
02 로컬 상태가 장비의 실제 부수효과를 따라가지 못함 3회 재발 · 동일 근본원인

앱이 "이렇게 될 것"이라고 낙관적으로 기록해둔 상태와, 실제 장비(vMix/OBS)가 스스로 일으키는 부수효과가 서로 다른 줄 모른 채 3번 각각 새로 발견됐습니다.

  • OBS 포팅 초기: doCut()가 pgmSrc=pvwSrc로 "복사"만 하고, 실제 스위처처럼 이전 PGM을 PVW로 되돌리지 않음
  • 단축키 배지 작업 중: refreshSwitcherUI()가 폴링마다 has-hotkey 클래스를 지워, 단축키가 1~2초 뒤 슬쩍 사라짐
  • 오늘: vMix의 실제 Function=Cut&Input=X는 PGM·PVW 버스를 항상 맞바꾸는데, setPGM/doCut/doAuto/_vmixCut/radioCutTo 5곳 전부 이를 로컬 상태에 반영하지 않아 "PGM만 눌렀는데 PVW가 제멋대로 바뀐다"로 보고됨 — 그중 _vmixCut()은 큐시트 GO/앵커→영상 전환의 실제 엔진이라 방송 중 매 아이템마다 재현되는 문제였음
비용: 세 번 모두 "장비가 실제로 무엇을 하는지"를 각 세션이 처음부터 다시 실측해야 했습니다 — 원칙 자체가 문서화된 것은 오늘이 처음입니다.
03 Mock 테스트는 통과, 실기기에서만 드러남 6건

이 프로젝트에서 가장 까다로웠던 버그들은 전부 로직이 아니라 "실제 장비의 타이밍/레이스"였고, 예외 없이 mock 테스트는 이를 잡아내지 못했습니다.

  • endStopFrames=5(167ms)가 이론상 맞았지만 실측에서 매번 실패 — 15(500ms)로 재조정 후 안정화
  • vMix 리스트는 자동전환 시 유예시간이 전혀 없다는 사실 — 1회성 setTimeout 설계 자체를 "굵게 대기 + 실시간 정밀 폴링"으로 재설계해야 했음
  • Restart 직후 vMix가 duration=0을 잠깐 반환하는 레이스 — 재시도 로직 필요
  • OBS 한국어 로케일에서 transitionName:'Cut' 하드코딩이 전혀 매칭 안 됨 — transitionKind 기준으로 전환해야 했음
  • GT 필드명에 .Text를 이미 포함해 입력하면 SelectedName이 중복되어 vMix가 "성공"을 반환하면서도 실제로는 아무 것도 안 바뀜
  • 이번 세션: vMix Cut 호출 직후 곧바로 상태를 읽으면 아직 스왑 전 상태가 보임 — 실제로는 약 1초 뒤에나 안정화
비용: 이런 버그는 코드 리뷰로는 원천적으로 안 보입니다 — 실측 없이는 "고쳤다"는 확신 자체가 틀린 확신이었습니다.
04 "해결 완료"로 닫았지만 실사용에서 재발 SRV 겹침 버그 · 3회 재수정

동일 증상이 서로 다른 근본원인 3개에서 각각 발생할 수 있었기 때문에, 첫 수정 후에도 같은 증상으로 계속 재열렸습니다.

  • 1차: 빠른 연속 단축키가 비동기 단계를 서로 덮어씀 → 세대 카운터(generation counter)로 해결
  • 2차: _vmixPollInputName 전역 변수가 중단된 재생 후 리셋 안 되고 남아있다가, 완전히 무관한 라디오 재생과 충돌 → 소유권 리셋 추가
  • 3차: _vmixSrvInput의 SRV1 폴백(패턴 01) 자체가 별도 충돌 경로였음 → 폴백 제거
  • 그 뒤로도 재발 보고가 있어 근본 설계(1회성 타이머)를 폴링 기반으로 교체하고서야 안정화(패턴 03과 동일 건)
비용: "SRV1/2 깨짐, SRV3만 정상"이라는 같은 문장이 최소 4번 반복 보고됐고, 그때마다 이전 수정을 재검증하는 데 시간이 들었습니다.
05 브라우저·JS 고유의 함정 4건

도메인 로직과 무관하게, 언어/플랫폼 자체의 반직관적 동작이 조용한 실패를 만들었습니다.

  • opener.appMode가 항상 undefined — let 변수는 window에 붙지 않음(접근자 함수 getAppMode() 필요)
  • 템플릿 리터럴 안의 </script>가 브라우저 HTML 토크나이저에 의해 바깥 스크립트를 조기 종료시킴
  • style.cssText=''가 무관한 레이아웃 스타일(margin-left:auto)까지 통째로 지움
  • 이번 세션: Playwright로 설정 탭을 클릭하려다 선택자가 다른 버튼("vMix 자동 설정")에 매칭되어 실제 vMix에 잘못된 명령이 나감(다행히 vMix가 거부해 무해)
비용: 대부분 발견 자체는 빨랐지만("왜 아무 반응이 없지"), 원인이 도메인 로직이 아니라 언어 함정이라는 걸 알기까지 우회 탐색 시간이 들었습니다.
06 Feature-first 개발로 쌓인 UX 안전장치 부채 한 세션에 30건 동시 발견

기능은 동작하지만 "실패해도 성공처럼 보이는" 종류의 문제가 한 번에 대량으로 쌓여 있었습니다 — 개별 버그라기보다 개발 순서(기능 먼저, 안전장치 나중)의 결과입니다.

  • vMix 미연결 상태에서도 CUT/AUTO가 성공 토스트를 표시
  • CG 큐 항목 하나를 끄면 전체 오버레이가 꺼짐
  • LIVE 배지가 실제 방송 여부와 무관하게 고정 표시
  • 방송 중 탭을 실수로 닫아도 막는 장치 없음(beforeunload 부재)
  • 처음부터/서버 불러오기 등 파괴적 동작에 확인창 없음, 삭제에 undo 없음
비용: 두 개 앱 전체를 훑는 전담 감사 세션이 따로 필요했습니다 — 기능 추가 시점에 나눠 처리했다면 하루짜리 대규모 감사가 필요 없었을 종류의 항목들입니다.
더 빠르고 좋은 바이브 코딩을 위한 제안

같은 시간에 더 적게 반복하려면

아래는 영향이 큰 순서대로 정리했습니다. 공통적으로, 이번 프로젝트의 재작업 대부분은 "몰라서"가 아니라 "한 번 안 이후 전수 적용을 안 해서" 발생했습니다.

01
폴백 버그를 하나 고치면, 같은 모양을 그 자리에서 전수 grep
전환효과 || 'Cut' 폴백은 발견될 때마다 "이번엔 진짜 완료"로 닫혔지만 8곳 넘게 나눠 발견됐습니다. 같은 파일과 자매 파일(MARS/Master)에서 || 뒤에 리터럴 기본값이 오는 자리를 한 번에 훑는 게, 나중에 재보고를 받고 다시 찾는 것보다 항상 저렴합니다.
02
장비의 "진짜 동작 원칙"은 발견 즉시 한 곳에 원칙으로 박아둔다
vMix의 Cut/Fade+Input이 PGM·PVW를 항상 맞바꾼다는 사실은 이번이 3번째 발견입니다. 매번 새 함수가 이 부수효과를 모른 채 추가됐기 때문입니다. "이 장비 API를 호출하는 새 코드를 짤 때 반드시 확인할 것" 목록이 있었다면 3번이 아니라 1번으로 끝났을 문제입니다.
03
스위처/타이밍류는 실기기 검증을 완료 조건에 처음부터 포함
이 프로젝트의 가장 심각했던 버그 6건은 전부 mock 테스트 통과 후 실기기에서만 드러났습니다. "일단 mock으로 검증 후 나중에 실기기로 확인"이 아니라, 장비 부수효과에 의존하는 기능은 실기기 확인 자체를 1차 완료 조건으로 삼는 편이 재작업 횟수를 줄입니다.
04
Playwright 재검증은 캐시버스팅을 기본값으로
"고쳤는데도 여전히 깨짐"이 최소 2번은 사실 브라우저가 편집 전 스크립트를 캐싱한 결과였습니다. 파일을 수정한 뒤의 첫 재검증은 ?v=n 캐시버스팅을 습관적으로 붙이고, 의심스러우면 fn.toString()으로 실제 로드된 코드를 확인합니다.
05
UI로 테스트한 뒤엔 항상 자동저장 오염 여부까지 확인
selectItem/renderCuesheet 같은 "보기용" 함수가 실제로는 조용히 localStorage에 저장까지 하는 바람에, 테스트용 임시 데이터가 실제 운영자의 자동저장을 두 번 오염시켰습니다. UI 경로로 테스트했다면 메모리 변수뿐 아니라 localStorage까지 확인하는 걸 표준 절차로 둡니다.
06
공유 가변 상태엔 처음부터 소유자/세대 키를 넣는다
_vmixPollInputName 전역 변수 하나를 큐시트와 라디오가 동시에 썼다가 충돌한 뒤에야 _srvPlayGeneration 같은 소유권 카운터 패턴을 배웠습니다. 이후 새로 만드는 공유 상태에도 "누가 마지막으로 썼는지"를 처음부터 넣으면 같은 계열 버그가 재발하지 않습니다.
07
기능 출시 시점에 안전장치 3문항을 곁들인다
"실패해도 성공처럼 보이는가?", "파괴적 동작에 확인창이 있는가?", "표시가 실제 상태를 따라가는가?" — 이 세 가지만 기능 추가 시점에 짧게 자가점검했다면, 30개를 한꺼번에 훑는 대규모 감사 세션이 따로 필요 없었을 것입니다.
