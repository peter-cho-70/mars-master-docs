신규 기획 3종 교차 분석 · 2026-07-23

vMix 연동 프로그램,
다음 방향

최근 doc/에 들어온 3개 문서 — vMix 최신 트렌드·이슈 분석, 마이크 레벨 기반 자동 샷전환 PRD, 오디오 인식 기반 기사 하이라이트 PRD — 를 기존 안정화 로드맵과 상업화 기획 위에 겹쳐 놓고, 이 프로젝트의 "vMix 연동 프로그램"이 실제로 어느 방향을 향해야 하는지 정리한 문서다.

종합 결론 세 문서는 하나의 로드맵이 아니라 서로 다른 두 트랙(사내 실사용 / 상업화)의 재료다. 공통점은 딱 하나 — 셋 다 "vMix 상태를 읽고 판단해서 명령을 보낸다"는 같은 패턴을 요구한다는 것. 이 공용 코어를 먼저 뽑아내는 것이 두 신규 기능 PRD보다 먼저 와야 한다.
근거 문서 3종 교차 분석 기존 P0-P3 로드맵과 정합 사내 트랙 vs 상업화 트랙 분리 2026-07-23 기준

지금 들어온 세 문서, 정직하게 분류하면

세 문서를 그대로 이어 붙이면 방향이 흐려진다. 각 문서가 실제로 무엇에 대한 재료인지부터 구분한다.

01 · 상황 인식

vMix 트렌드·이슈 분석

2026-07-04 리포트의 후속판. vMix 27→29 버전 흐름과 국내 사용처를 보강했지만, 새로운 불편사항 발견보다는 "07-04 리포트의 가설이 아직 유효한가"를 재확인하는 성격이 강하다. 문서 스스로도 포럼 로그인 제약으로 조사가 제한적이었다고 밝힌다.

02 · 신규 PRD (사내 우선)

자동 샷전환 PRD

일반적인 외부 컨트롤러로 문서 서두를 열지만, 9절의 실제 적용 사례가 "경제·시사 토크쇼(진행자+출연자2, 갈라치기)"다. 이건 정확히 MARS/Master가 이미 다루는 시사 프로그램 포맷이다 — Track 2(상업화) 아이디어에서 출발했지만 가장 먼저 쓸 곳은 Track 1(사내)이다.

03 · 신규 PRD (사내 전용)

오디오 기사 하이라이트 PRD

뉴스 큐시트·기사 원문·프롬프터에 직결된 기능이라 처음부터 Track 1 전용이다. 교회/학교/유튜버 대상 Track 2와는 접점이 거의 없다 — 굳이 잇는다면 "설교·발표 원고 트래킹" 정도의 먼 응용에 그친다.

04 · 이미 있던 로드맵

Cursor AI 다음 일정 (07-19)

"새 기능을 더 얹기 전에 이미 되는 방송 경로를 반복 증명"이 결론이었다. 이번 3개 문서는 정확히 그 "더 얹을 새 기능" 후보군이다 — 그 로드맵의 P0-P3 통과 조건과 무관하게 착수하면 같은 문제(가짜 성공, 설정 드리프트)가 새 기능에도 반복된다.

이 문서가 하는 일 세 문서를 "다음에 뭘 만들지 목록"으로 합치지 않는다. 대신 어느 트랙에 속하는지 먼저 가르고, 사내 트랙(Track 1)에 속하는 두 PRD를 기존 P0-P3 로드맵 뒤에 P3.5(코어 추출)→P4→P5로 이어 붙이며, 상업화 트랙(Track 2)은 별도 워크스트림으로 명시적으로 분리한다.

두 갈래로 갈라지는 지점

OnAir Connect 초기 구상이 이미 정확히 짚었다 — "지금 개발 중인 코어 엔진(vMix API 연동부)에 타겟별 프리셋/UI 레이어만 얹으면 하나의 코드베이스로 여러 시장을 커버할 수 있다." 이 프로젝트는 실제로 하나의 코어와 두 개의 소비처로 이루어져 있다.

TRACK 1 · 사내 실사용

MARS/Master 뉴스·시사 방송

실제 vMix·OBS·Videohub를 매일 돌리는 현재 시스템. 큐시트, 전환, 서버 재생, CG, LIVE 시퀀스까지 검증된 실사용자가 있다.

이번 3문서 중 이 트랙 소관: 자동 샷전환(9절 시사토크쇼 적용), 오디오 기사 하이라이트 전체.

TRACK 2 · 상업화 (OnAir Connect)

교회·학교·유튜버 대상 제품

원격 접속 간소화·설정 없는 자동 컨트롤 패널이 핵심 차별화. 중간보고서·login.md가 이미 별도 로드맵을 갖고 있다.

이번 3문서 중 이 트랙 소관: vMix 트렌드·이슈 분석(포지셔닝 갱신), 자동 샷전환의 "개인 유튜버용 저가 버전" 원안.

섞으면 생기는 위험 자동 샷전환 PRD 한 문서 안에 "외부 컨트롤러/별도 실행 프로그램"(Track 2 어투)과 "MARS 아이템 편집창에 CAM/SRV/DVE 소스 사전 설정"(Track 1 실제 구조, live-relay-item-sequence-engine 메모리 참고)이 섞여 있다. 어느 트랙으로 만들지 정하지 않고 착수하면 아키텍처가 반반씩 섞인 결과물이 나온다. 아래 로드맵은 Track 1으로 확정하고 시작하되, 로직을 재사용 가능하게 만들어 Track 2 이식 여지를 남기는 절충안이다.

핵심 통찰 — 세 문서가 공통으로 요구하는 것

표면적 기능은 다르지만, 자동 샷전환과 오디오 기사 하이라이트는 구조적으로 같은 파이프라인을 요구한다.

Master의 기존 코드에는 이미 이 패턴이 기능마다 각자 구현되어 있다 — 스위처 폴링(_vmixPollTick), 서버 클립 조기전환 폴링, LIVE 시퀀스 단계 진행 폴링이 모두 별도 타이머·별도 상태로 존재한다(master-clip-pause-resume-on-cut, live-relay-item-sequence-engine 메모리 참고). 두 신규 PRD를 각자 새 폴링 루프로 얹으면, 이미 07-19 로드맵 P2가 지적한 "탭 백그라운드 타이머 스로틀링" 같은 문제가 폴링 루프 개수만큼 반복된다.

그래서 결론 07-19 로드맵의 P3 항목 "vMix/OBS 공통 큐시트·타입·자막 로직을 공유 모듈로 분리"를 "vMix 상태 폴링 + 판정 + 명령 코어"로 범위를 넓혀서 먼저 뽑아낸다. 이 모듈 하나가 자동 샷전환의 임계값/가중치/쿨다운 로직 전체를 흡수할 수 있고, 이후 다른 자동화 기능이 추가돼도 폴링 루프를 새로 만들 필요가 없어진다.

통합 로드맵 — P0-P3 다음, P3.5~P5

아래 단계는 기존 P0-P3(즉시 복구 → 회귀 테스트 → 실장비 내구 검증 → 구조 정리)가 모두 통과된 뒤에만 착수한다. P0-P3의 원칙 그대로 — 앞 단계 통과 조건 없이는 다음 단계 신규 기능에 손대지 않는다.

P3.5(선행조건). 폴링+명령 코어 모듈 추출

P3 직후, 신규 기능 착수 전

P4/P5가 실제로 재사용할 수 있는 형태로, 지금 산발적인 vMix 상태 폴링 로직을 하나로 모은다.

  • 주기적 vMix XML State 폴링 → 파서 → 콜백 디스패치를 하나의 서비스로 통합(주기, 대상 필드는 설정 가능)
  • "임계값 초과 + 유지시간 + 쿨다운" 판정 로직을 재사용 가능한 순수 함수로 분리 (자동 샷전환 §6.3을 최초 소비자로 구현)
  • 탭 백그라운드/네트워크 단절 시 폴백 규칙을 이 모듈 안에 한 번만 구현해 모든 소비 기능이 상속받게 함
통과 조건 — 기존 스위처/서버클립 폴링 최소 1곳이 이 모듈로 이전되어 회귀 없이 동작.

P4. 자동 샷전환 MVP — 시사 토크쇼 포맷

P3.5 이후

PRD의 M1~M4를 그대로 따르되, 적용 대상은 PRD 9절의 4캠 시사 토크쇼 포맷(진행자+출연자2)으로 고정한다 — 실장비·실사용자가 이미 있는 쪽부터 검증한다.

  • M1: PoC — P3.5 코어로 오디오 레벨 폴링·임계값 판정 검증 (실vMix)
  • M2: 수동/반자동 모드 UI + Cut 명령 연동, 카메라-마이크 매핑을 MARS 아이템 편집창(CAM/SRV/DVE 사전 설정)에 결합
  • M3: 자동 모드 + 갈라치기 판별(핑퐁 감지) — 기존 DVE1-3 슬롯 재사용 가능 여부 먼저 확인 (신규 Input 제작은 최후 수단)
  • M4: 실전 리허설 및 파라미터 튜닝(핑퐁 4~6초, 단독발언 3초 등 PRD 기본값에서 시작)
통과 조건 — PRD §3 성공 지표(전환 지연 1초 이내, 5초 내 채터링 0건) 실장비 리허설로 확인.

P5. 오디오 기사 하이라이트 — PoC 실장비 검증부터

P3.5와 독립 병행 가능 · PoC 코드 이미 존재

vMix 폴링 코어와 무관한 별도 파이프라인(STT+텍스트매칭)이라 P4와 동시 진행 가능. PRD §7-3의 3단계 전환 전략 중 1단계가 이미 코드로 구현되어 있다 — doc/spec/Whisper/(faster-whisper 기반 CLI PoC). Vosk를 거치지 않고 §7-3-3 Whisper 경로로 바로 착수된 상태이며, "PoC 제작"이 아니라 "PoC 실장비 검증"이 다음 순서다.

  • 0단계(완료): matcher.py가 forward-window 탐색 + 유사도 임계값 미달 시 이전 위치 유지(fallback)를 이미 구현 — PRD §6.3 판정 로직과 1:1 대응
  • 1단계: 실제 PC(인터넷+마이크 필요 — 샌드박스는 모델 다운로드·마이크 접근 불가)에서 --wav 녹음 검증 → --mic 실시간 순으로 지연·정확도 실측, --delay-comp/--min-score 튜닝
  • 2단계: CPU 실시간 여유(RTF)가 부족하면 tiny/base로 하향하거나 Vosk로 교체 검토 — 반대로 정확도가 부족하면 RTZR STT(월 10시간 무료)로 승급 검토
  • 3단계: 큐시트 연동(README "다음 단계" 1~3) — --article 텍스트 파일을 MARS 기사매칭(matchArticle) 결과로 교체, 마이크 소스를 방송 믹서 출력으로 교체, 하이라이트 출력을 문서 화면 웹소켓으로 연동
통과 조건 — 실제 뉴스 원고 낭독 샘플에서 매칭 실패율·평균 오차(문장 수 기준)가 목표치 이하로 실측 확인.
순서를 지켜야 하는 이유 P3.5를 건너뛰고 P4·P5를 각자 새 폴링/새 상태로 구현하면, 나중에 P3.5로 되돌리는 리팩터링이 두 기능을 동시에 건드리는 더 위험한 작업이 된다. 코어 추출은 "나중에 정리"가 아니라 "지금 먼저"다.

Track 2(상업화)와의 관계 — 대체가 아니라 병행

P4~P5는 전부 Track 1(사내) 소관이다. Track 2(OnAir Connect) 상업화는 이미 별도 로드맵 A~F와 login.md의 백엔드 스택 결정을 축으로 진행 중이며, 이 문서가 그 순서를 바꾸지 않는다.

이번 트렌드·이슈 분석 문서가 Track 2에 실제로 기여하는 것은 두 가지뿐이다.

  • 포지셔닝 갱신 — vMix 29의 OMT·Instant Replay·POG 등은 우리 차별화 포인트(원격 접속 간소화)와 직접 겹치지 않는다. 07-04 리포트의 "핵심 차별화는 원격 접속 간소화"라는 결론은 이번 조사로도 반박되지 않았다 — 즉 기존 결론을 유지해도 된다.
  • 관찰 대상 등록, 실행 아님 — "AIffirmatron"(AI 소셜 스트림 상호작용) 같은 vMix 자체의 AI 실험 기능은 지금 당장 대응할 필요는 없지만, Track 2 로드맵 후반부(중간보고서 로드맵 E~F)에서 "AI 기반 차별화"를 검토할 때 참고 사례로 남겨둔다.
자동 샷전환의 Track 2 이식 여지 PRD 원안이 "개인 유튜버용 저가 자동 스위칭"에서 출발했으므로(초기 구상 대화록 참고), P4에서 Track 1용으로 구현할 때 카메라-마이크 매핑과 판정 로직을 MARS 아이템 편집창 UI와 분리 가능한 형태로 짜두면, Track 2 착수 시점에 같은 판정 엔진을 재사용해 UI 레이어만 새로 씌울 수 있다. 지금 단계에서 Track 2를 동시에 만들 필요는 없다 — 재사용 가능한 경계만 지켜두면 된다.

교차 리스크 — PRD 각자의 리스크에 없던 것

영역리스크근거
하드웨어 두 PRD 모두 "카메라별 마이크 Input"을 전제하는데, 현재 실vMix 입력 구성에 카메라별 전용 마이크 채널이 매핑돼 있는지 미확인 상태 real-vmix-environment 메모리 — cam1~4/rs1~6 vmixInput이 실제 구성과 어긋난 전례 있음
폴링 부하 P4의 오디오 레벨 폴링(50~100ms)이 기존 스위처·서버클립 폴링과 같은 vMix 인스턴스에 얹히면 응답 지연이 누적될 수 있음 PRD §8 "폴링 주기와 지연시간 트레이드오프 — 실측 필요"
갈라치기 Input 증식 PRD §10이 스스로 인정 — 출연자가 늘수록 조합별 갈라치기 Input을 다 만들어야 함. 기존 DVE1-3이 몇 조합까지 흡수 가능한지 먼저 재고 조사 필요 PRD §9.1, §10
큐시트-기사 매핑 전제 P5는 "큐시트와 기사 원문이 정확히 매핑돼 있지 않으면 기능 전체가 무력화"됨 — MARS 기사매칭이 실제로 항상 완료된 상태로 방송에 들어가는지 확인 필요 PRD §9 리스크, golabon-segye-cuesheet-audit 메모리(과거 매칭 공백 사례)

완료 기준 재확인 — 새 기능도 예외 없음

07-19 로드맵이 정한 5가지 완료 기준은 P4·P5에도 그대로 적용한다. 이미 두 PRD가 이 기준을 어느 정도 예견하고 있다는 점은 좋은 신호다.

정상 경로 — 발언자 인식 후 자동 컷, 낭독 위치 하이라이트가 실화면에서 동작.
실패 경로 — vMix 연결 끊김, STT 매칭 실패, 마이크 무음 상태에서 가짜 전환/가짜 하이라이트를 만들지 않음(PRD가 이미 폴백을 명시).
회귀 테스트 — P3.5 코어 모듈에 대해 mock 기반 판정 로직 테스트를 남겨, 다음 수정 뒤에도 같은 명령으로 재검증.
실제 상태 확인 — "전환됐다"는 판단을 앱 표시가 아니라 vMix XML State의 실제 PGM 변화로 확인.
운영 복구 — 리허설 중 네트워크 단절·서버 재시작 후 수동 모드로 안전 복귀하는지 실측.
