1 개요

MARS/Master는 방송사 뉴스 큐시트 제작부터 실제 vMix 스위처 진행까지를 자동화하는 2인 1조(기자/PD ↔ 기술감독) 시스템입니다. MARS가 기사 작성·큐시트 제작을 담당하고, Master가 그 큐시트를 실제 방송 진행(전환효과·자막·서버클립·라우터·스위처)으로 실행합니다. 둘 다 server.py 하나가 서빙하는 순수 HTML/JS 정적 페이지이며, 별도 빌드 과정이나 프레임워크 없이 브라우저에서 바로 동작합니다.

방송 하드웨어(vMix, Blackmagic Videohub)는 전부 실제 장비와 직접 통신하며 동작을 검증했습니다 — 이 문서에서 "실장비 검증"이라고 표시된 항목은 목킹이 아니라 진짜 vMix REST API·Videohub TCP 프로토콜로 확인한 것입니다.

2 왜 이 프로그램을 만들었나

vMix는 그 자체로 매우 강력한 방송 소프트웨어입니다. 이 프로젝트는 vMix를 대체하려는 것이 아니라, vMix의 REST API를 전부 그대로 사용하면서 그 위에 다른 인터페이스를 얹은 것입니다. 왜 굳이 그럴까요 — vMix를 그냥 켜서 바로 방송해도 되는데 말입니다.

vMix의 화면과 용어는 범용 방송 소프트웨어의 언어로 돼 있습니다. Input 번호, Overlay 채널, Layers, SelectedIndex 같은 개념을 알아야 하고, 어느 인풋이 어느 카메라인지, 지금 어느 서버 슬롯이 비어있는지, 전환마다 얼마나 걸리는지를 운영자가 머릿속으로 계속 추적해야 합니다. 뉴스 방송 현장에서 실제로 필요한 건 그런 게 아니라 "이 리포트는 컷으로", "이 단신은 디졸브로", "이 자리엔 기자 카메라" 같은, 방송 자체의 언어입니다.

MARS/Master는 그 번역기입니다. 운영자는 vMix의 존재 자체를 몰라도 되고, 화면 뒤에서는 똑같은 vMix REST API가 그대로 호출됩니다 — 기능이 줄어드는 게 아니라, 그 기능에 접근하는 언어가 이 방송국의 워크플로우에 맞게 바뀔 뿐입니다.

하고 싶은 일vMix를 직접 쓴다면MARS/Master로 쓴다면
화면분할(DVE) 소스 바꾸기 Layers 편집창을 직접 열고, 레이어 순번·API 파라미터(SetLayer, 1부터 세는 번호, 콤마 결합 문법)를 정확히 알아야 함 설정 탭에서 "이 자리엔 카메라, 이 자리엔 서버" 드롭다운 선택 한 번 — 방송 중 자동 반영
자막 켜고 끄기 Overlay 채널 번호·GT 템플릿 필드명을 정확히 알아야 함 "제목자막" 카드에 문구 입력 → 전송 버튼 한 번
다음 아이템으로 넘어갈 때 이전 자막·화면분할 내용을 운영자가 매번 수동으로 정리 자동 정리 — 다음 아이템은 항상 빈 보드에서 시작
서버 클립이 다음 파일로 넘어가는 사고 운영자가 타이밍 감으로 직접 컷해서 방지 정밀 타이머 + 이중 안전장치로 자동 방지
이 방송국만의 새 워크플로우가 필요할 때 vMix 자체 기능 범위 안에서만 가능 바이브 코딩으로 당일 추가 가능 — 4절 DVE 레이어 기능이 실제 사례
왜 "바이브 코딩"이 이걸 가능하게 했나 전통적으로는 이런 방송국 전용 커스텀 제어 레이어를 만들려면 전담 개발팀과 수개월, 그리고 실제 방송 하드웨어에 대한 깊은 경험이 필요합니다. 이 프로젝트는 한 사람이 평범한 말로 필요한 걸 설명하고, 실제 vMix 장비에 바로 연결해 검증하고, 실사용 중 나온 버그를 그날그날 고치는 방식으로 만들어졌습니다 — 오늘 하루 만에 자막 화면 하나, 자막 파이프라인 하나, 화면분할 원격 제어 기능 하나가 전부 실장비 검증까지 끝난 것이 그 증거입니다(4·6절 참고). "vMix를 몰라도 vMix의 기능을 100% 쓴다"는 더 이상 이론이 아니라, 이 시스템이 매일 검증하고 있는 사실입니다.

실제 하드웨어

MARS가 완성한 큐시트를 저장하면 Master가 불러와 진행합니다. 파일 기반 핸드오프라 두 프로그램이 같은 네트워크의 다른 PC에서 동작해도 되고, MARS를 재전송해도 Master가 방송 중 이미 고쳐둔 값은 보존됩니다. server.py는 정적 파일 서빙, vMix REST API 프록시(브라우저의 CORS 제약 우회), 그리고 Videohub·자막 등 여러 화면 사이의 얇은 릴레이 큐 역할을 합니다.

설계 원칙 같은 로직을 두 번째 화면에 다시 구현하지 않습니다. 자막담당 페이지도, CG 스튜디오도 결국 Master의 기존 함수를 그대로 재사용하도록 얇은 큐만 얹었습니다 — 두 구현이 따로 있으면 반드시 갈라지기 때문입니다.

4 핵심 기능 요약

아이템 타입(리포트/단신/CM/브리지/중계 등)마다 전환효과·카메라·서버 규칙이 데이터로 정의돼 있어 설정 탭에서 직접 수정할 수 있고, Cut/Mix/Fade/Wipe 4종은 필수 기본값이라 삭제해도 자동으로 복구됩니다.

자막(CG) — 3개의 계층

① 표준 8종 CG(제목자막·이름자막·긴급자막·티커 등)는 vMix GT 템플릿에 매핑되고 IN/OUT 타이밍을 타임라인으로 지정합니다. ② CG 스튜디오신규는 자막담당을 위해 새로 디자인한 화면 — 8종 카드 상시 표시, 즉시전송/불러오기 분리, 단축키까지 갖췄습니다. ③ 커스텀 HTML 템플릿신규은 vMix GT를 아예 거치지 않고 자막담당이 직접 HTML/CSS로 만든 템플릿을 표시(IN)/내리기(OUT)로 제어하는 완전히 별도 경로입니다 — Master가 꺼져 있어도 동작합니다.

서버(SRV) 클립 재생

vMix List 인풋이 클립 하나가 끝나면 자체적으로 다음 항목으로 넘어가 버리는 것을 막기 위해 정밀 정지 타이머·0.3초 고속 감시·1.5초 폴링·디졸브 유예시간까지 여러 겹의 안전장치를 쌓았습니다. 이 영역은 프로젝트에서 가장 많이 재발한 버그 영역이기도 합니다 — 5절 패턴 ①·②가 이 이야기입니다.

LIVE 중계/출연 시퀀스 · DVE 레이어 원격 제어 신규

중계·출연은 "앵커멘트 → [앵커 갈라치기 → 화면 → 서버 갈라치기] × 질문 수 → 마무리" 시퀀스로 진행됩니다. "갈라치기" 화면(DVE)은 vMix의 Layers 기능으로 미리 구성해둔 화면분할 합성 인풋인데, 이제 설정 탭에서 각 DVE 레이어에 기본 소스를 등록해두면 LIVE 시퀀스가 단계를 넘어가기 직전 자동으로 올바른 소스를 채워 넣습니다. 

Function=SetLayer&Input=<합성 인풋>&Value=<레이어번호(1부터)>,<새 소스>
실장비에서만 드러난 위험 존재하지 않는 소스 이름을 가리키면 vMix는 "성공"이라고 응답하면서 그 레이어를 통째로 지워버립니다. 그래서 지금은 실제 vMix 인풋 목록과 먼저 대조해 존재할 때만 명령을 보내도록 바꿨습니다 — 6절에서 이 발견 과정을 자세히 다룹니다.

외부 장비 연동

Blackmagic Videohub(16×16 라우터)는 TCP 상시 연결로 크로스포인트를 중계하고, 카메라/서버/DVE/라우터 소스 전체는 switcherSources 배열 하나로 통합 관리됩니다 — 소스를 늘리려고 스위처 UI 10여 곳을 일일이 손대던 예전 방식에서 벗어났습니다.

5 반복해서 발견된 오류 패턴

개별 버그는 매번 다른 증상이었지만, 근본 원인을 모아보면 몇 가지 패턴으로 수렴합니다. 같은 프로젝트를 계속 다룰 사람(사람이든 AI든)이 다음에 비슷한 기능을 만들 때 참고할 수 있도록 정리합니다.

패턴 01

vMix는 틀린 값을 줘도 "성공"이라고 답한다

존재하지 않는 인풋 이름을 SetText/OverlayIn/SetLayer 등에 넘겨도 vMix REST API는 HTTP 200과 "성공" 메시지를 돌려줍니다 — 심지어 SetLayer는 그러면서 실제로는 레이어를 지워버리기까지 합니다. HTTP 응답만으로는 절대 신뢰할 수 없고, 실제 vMix 상태(XML)를 다시 읽어 대조해야만 진짜 확인이 됩니다. verifyVMixInputMap(), applySourceLayers()의 존재 검증, _vmixEnsurePaused()가 전부 이 교훈에서 나왔습니다.

패턴 02

"이 명령을 보내면 이 상태가 된다"는 가정은 실장비에서 자주 깨진다

Cut이 리스트 재생을 자체 트리거하거나, Restart가 재생을 재개하거나, Restart와 Play를 너무 붙여 보내면 재생 명령이 씹히는 등 — 문서에 없는 실제 하드웨어 동작이 반복적으로 코드의 암묵적 가정을 깼습니다. 이번 세션의 DVE SetLayer 파라미터 재발견(6절)도 같은 계열입니다: 처음 만든 코드는 문서를 확인하지 않고 그럴듯한 이름으로 추측해서 썼고, 실제로는 한 번도 동작한 적이 없었습니다.

패턴 03

안전장치 사이의 경합 — 하나를 고치면 다른 하나가 새로 드러난다

서버 클립 안전장치는 0.3초 감시·1.5초 폴링·정밀 정지 타이머·유예시간까지 겹겹이 쌓여 있는데, 새 보호막을 하나 추가할 때마다 "그 보호막이 아직 유예시간을 안 보고 있어서" 생기는 새로운 경합이 나타났습니다(07-17→07-18 사이 같은 증상이 세 번 재발). 여러 타이머/폴링이 같은 상태를 동시에 감시할 때는 새 방어막을 추가하면 기존 방어막 전부가 그 새 상태를 알아야 한다는 점을 놓치기 쉽습니다.

패턴 04

"자동이 디폴트"는 방송에서 위험하다

CG 자동 타이밍, 서버 자동 교번, 조기 전환 등 여러 기능에서 "아무것도 지정 안 하면 자동으로 나간다"는 기본값이 실제로는 "방송 중 저절로 자막이 튀어나온다"는 사고로 이어졌습니다. 지금은 명시적으로 켠 경우만 자동, 그 외엔 항상 수동 확인을 거치도록 기본값을 반전시켰습니다.

패턴 05

두 번째 화면을 만들 때 첫 번째 화면의 방어 로직을 빠뜨리기 쉽다

"대기 단계에서 재생이 시작되면 안 된다"는 방어가 armSelectedItemOnPGM() 한 곳에만 있고, 같은 문제를 겪을 수 있는 다른 세 지점(라디오 등록영상, 다음 아이템 프리뷰, 앵커 단계 사전로드)에는 빠져 있었습니다. 같은 패턴을 쓰는 곳이 여럿이면 하나를 고칠 때 "이 방어가 필요한 다른 곳이 더 있는가"를 항상 같이 점검해야 합니다.

패턴 06

목킹 테스트는 로직 버그를 잡고, 실장비 테스트는 타이밍·문서 오류를 잡는다

이 프로젝트의 버그 대부분은 Playwright + vMix 목킹으로 실제 함수 호출 순서를 검증하며 잡혔지만, "Restart와 Play를 너무 붙이면 재생이 안 됨"이나 "SetLayer 파라미터 자체가 틀림" 같은 문제는 목킹으로는 재현되지 않고 실장비에서만 드러났습니다. 둘 다 필요하고, 서로 대체할 수 없습니다.
