지금까지 무엇을 만들었고, 무엇이 틀렸었나

1 개요

MARS/Master는 방송사 뉴스 큐시트 제작부터 실제 vMix 스위처 진행까지를 자동화하는 2인 1조(기자/PD ↔ 기술감독) 시스템입니다. MARS(nps-news-system.html)가 기사 작성·큐시트 제작을 담당하고, Master(news-broadcast-system.html)가 그 큐시트를 실제 방송 진행(전환효과·자막·서버클립·라우터·스위처)으로 실행합니다. 둘 다 server.py 하나가 서빙하는 순수 HTML/JS 정적 페이지이며, 별도 빌드 과정이나 프레임워크 없이 브라우저에서 바로 동작합니다.

방송 하드웨어(vMix, Blackmagic Videohub)는 전부 실제 장비와 직접 통신하며 동작을 검증했습니다 — 이 문서에서 "실장비 검증"이라고 표시된 항목은 목킹이 아니라 진짜 vMix REST API·Videohub TCP 프로토콜로 확인한 것입니다.

MARS가 완성한 큐시트를 POST /api/cuesheet로 저장하면, Master가 GET /api/cuesheet로 불러와 진행합니다. 파일 기반 핸드오프라 두 프로그램이 같은 네트워크의 다른 PC에서 동작해도 되고, MARS를 여러 번 재전송해도 Master가 이미 방송 중 고쳐둔 값(예: LIVE 시퀀스 단계 재정의)은 보존하도록 병합 로직이 따로 있습니다.

server.py — 세 가지 역할

  • 정적 파일 서빙 — MARS·Master·CG 스튜디오 등 모든 HTML/JS/미디어
  • vMix REST API 프록시 — 브라우저는 CORS 때문에 vMix를 직접 못 부르므로 /proxy/vmix가 대신 중계
  • 얇은 릴레이 큐 — Videohub(TCP 상시연결), 자막담당 CG 요청, 자막 종류 이름 동기화, 커스텀 자막 상태까지 전부 같은 패턴: "로직은 한 곳(Master)에만 구현하고, 다른 화면은 서버를 거쳐 요청만 넣는다"
설계 원칙 같은 로직(예: vMix GT 필드 매핑, 서버 클립 안전장치)을 두 번째 화면에 다시 구현하지 않습니다. 자막담당 페이지도, CG 스튜디오도, 8종 표준 자막을 실제로 내보낼 때는 전부 Master의 기존 함수를 그대로 재사용하도록 얇은 큐만 얹었습니다 — 두 구현이 따로 있으면 반드시 갈라지기 때문입니다.

4 자막(CG) 시스템 — 3개의 계층

자막 요구사항이 커지면서 서로 다른 목적의 세 파이프라인이 생겼습니다. 셋 다 공존하며, 운영 방식에 따라 골라 쓸 수 있습니다.

① 표준 8종 CG (vMix GT 타이틀)

제목자막·이름자막·긴급자막·티커·하단자막·시간자막·위치자막·코너자막 8종 — 각각 vMix GT(.gtzip) 템플릿의 필드명·오버레이 슬롯에 매핑됩니다. 자막 IN/OUT 타이밍을 아이템 편집창에서 타임라인 드래그로 지정할 수 있고, 큐시트 진행에 맞춰 자동/수동으로 내보낼 수 있습니다. Master의 CG 큐 패널, 별도 PC용 cg-operator.html, 그리고 새로 만든 cg-studio.html "입력" 탭까지 세 화면 모두 결국 같은 sendCGVMix()를 실행합니다.

② CG 스튜디오 신규 07-18

기존 cg-operator.html을 대체하는 게 아니라 나란히 존재하는 새 자막담당 화면. 8종 카드가 모두 항상 보이고, 큐시트 아이템에 미리 등록된 자막을 "즉시전송"(바로 발사)과 "불러오기"(카드에 담아 다듬은 뒤 전송) 두 갈래로 분리했습니다. Alt+1~8 단축 포커스, 이 브라우저 전용 즐겨찾기 문구, 2단계 확인이 걸린 "전체 해제" 버튼, 그리고 /api/cg-state를 실시간 폴링해 카드 상태를 지금 실제로 켜져 있는 값으로 미리 채워두는 기능까지 포함합니다.

③ 커스텀 HTML 자막 템플릿 신규 07-18

vMix GT/Master를 아예 거치지 않는 세 번째 경로. 자막담당이 직접 HTML/CSS로 자막 템플릿을 만들고({{필드명}} 자리표시자로 입력 필드 자동 생성), 체크무늬 배경의 실시간 미리보기로 확인한 뒤 표시(IN)/내리기(OUT)로 직접 제어합니다. 새 서버 엔드포인트 /api/custom-templates(템플릿 정의, 파일 영속화)와 /api/custom-cg-state(지금 화면 상태)를 cg-studio.html이 직접 호출하고, 별도 렌더러 cg-custom-overlay.html이 그 상태를 폴링해 그대로 그립니다 — Master가 꺼져 있어도 동작합니다.

실사용 중 발견된 버그 아이템이 끝나도 이전 자막이 화면에 남아 다음 아이템 자막과 잠깐 겹쳐 보이는 문제가 있었습니다. 원인은 아이템 전환 시 호출되는 clearCGAutoTimers()가 이름 그대로 "예약된 타이머만" 취소했지, 이미 화면에 나가 있는 자막 자체는 꺼주지 않았기 때문입니다. 이 함수에 clearAllCGVMix()/clearAllCGBrowser() 호출을 추가해 아이템이 바뀌는 모든 지점에서 자막판이 항상 빈 상태로 시작하도록 고쳤습니다.

5 서버(SRV) 클립 재생 — 안전장치의 역사

SRV1~3(vMix VideoList 인풋)에서 영상을 재생하는 기능은 이 프로젝트에서 가장 많이 재발한 버그 영역입니다. vMix List 인풋은 한 클립이 끝나면 자체적으로 다음 리스트 항목으로 넘어가 버리는데, Master는 "지정한 클립까지만" 재생돼야 하므로 여러 겹의 안전장치를 쌓아왔습니다.

  • 정밀 정지 타이머 — vMix가 보고하는 실측 duration/position으로 클립이 정확히 언제 끝나는지 계산해 선제적으로 Pause
  • 0.3초 고속 감시 + 1.5초 폴링 — 정밀 타이머가 못 미더울 때(재생 직후 duration을 0으로 보고하는 경우 등)의 이중 백업
  • 디졸브 유예시간 — Mix/Fade/Wipe 전환 중에는 안전장치가 "인덱스가 벗어났다"고 오판하지 않도록 전환 소요시간만큼 유예
  • _vmixEnsurePaused() — Pause/Restart 명령을 보낸 뒤 실제 state를 다시 읽어 확인하고, 아직 재생 중이면 재시도(07-18)

이 영역의 버그가 특히 끈질겼던 이유는, "Pause+Restart를 걸면 반드시 정지 상태가 된다"는 가정 자체가 실제 vMix 하드웨어에서 항상 보장되지 않았기 때문입니다 — Cut이 리스트 재생을 자체 트리거하거나, Restart 자체가 재생을 재개하는 사례가 실측으로 여러 번 확인됐습니다. 8절의 반복 패턴 ①이 이 이야기입니다.

6 LIVE 중계/출연 시퀀스 · DVE 레이어 원격 제어

중계(기자 연결)와 출연(스튜디오 게스트)은 "앵커멘트 → [앵커 갈라치기 → 중계/기자 화면 → 서버 갈라치기] × 질문 수 → 마무리"라는 가변 길이 시퀀스로 진행됩니다. MARS가 기사의 질문 수를 넘기면 Master가 기본 단계 목록을 자동 생성하고, 기술감독이 아이템 편집창에서 단계를 개별 추가/삭제/순서변경/소스변경할 수 있습니다 — MARS는 사전 설정, Master가 방송 중 최종 결정권을 갖는 구조입니다.

DVE 레이어 원격 제어 신규 07-18

"갈라치기" 화면(DVE1/2/3)은 vMix의 Layers 기능으로 미리 구성해둔 화면분할 합성 인풋입니다. 기존에는 이 화면 "전체"를 컷 대상으로만 썼는데, 서버(SRV1~3)가 방송 중 교번하는 것과 달리 DVE 안의 레이어는 고정이라 실제 온에어 서버와 어긋날 수 있는 구조적 공백이 있었습니다. 이제 설정 탭에서 각 DVE의 레이어(순번·이름·기본 소스)를 등록해두면, LIVE 시퀀스가 각 단계로 넘어가기 직전에 자동으로 올바른 소스를 그 레이어에 채워 넣습니다. 

Function=SetLayer&Input=<합성 인풋>&Value=<레이어번호(1부터)>,<새 소스>
실장비에서만 드러난 위험 존재하지 않는 소스 이름을 가리키면 vMix는 "성공"이라고 응답하면서 그 레이어를 통째로 지워버립니다(단순 무시가 아니라 파괴적 실패). 그래서 지금은 "먼저 보내고 나중에 확인"이 아니라 실제 vMix 인풋 목록과 먼저 대조해 존재할 때만 명령을 보내는 순서로 바꿨습니다. 9절에서 이 발견 과정을 자세히 다룹니다.

방송 화면 전환 없이 미리 확인할 수 있도록, 다음 단계의 DVE는 온에어로 나가기 전에 레이어를 채워 vMix 프리뷰(PVW) 버스에 올려둡니다 — 지금 온에어 중인 DVE는 이 과정에서 절대 건드리지 않습니다. 아이템이 끝나면(다음 아이템으로 전환) 그 아이템이 썼던 DVE들의 레이어를 기본값으로 되돌려, 특정 질문에서 재정의했던 값이 다른 상황에 남아있지 않게 합니다 — 4절의 자막 정리와 같은 원칙입니다.

7 외부 장비 연동

Blackmagic Videohub(16×16 라우터)는 server.py가 TCP 9990 상시 연결을 유지하고 /api/videohub/*로 상태 폴링·크로스포인트 전환을 중계합니다. 아이템별로 item.router={output,input}을 지정해두면 방송 진행에 맞춰 자동으로 회선을 전환하고, 아이템 편집창의 TAKE 버튼으로 저장/방송과 무관하게 즉시 전환도 가능합니다.

스위처(카메라/서버/DVE/라우터 소스 전체)는 switcherSources 배열 하나로 통합 관리됩니다 — 예전에는 cam1-4/srv1-3 7개가 스위처 HTML·색상·인풋맵·단축키 등 10곳 넘게 개별 하드코딩돼 있어 소스를 늘리려면 그 전부를 손대야 했습니다. 지금은 이 배열만 바꾸면 PGM/PVW 버튼·팝업·단축키 표시가 전부 따라갑니다.

8 반복해서 발견된 오류 패턴

개별 버그는 매번 다른 증상이었지만, 근본 원인을 모아보면 몇 가지 패턴으로 수렴합니다. 같은 프로젝트를 계속 다룰 사람(사람이든 AI든)이 다음에 비슷한 기능을 만들 때 참고할 수 있도록 정리합니다.

패턴 01

vMix는 틀린 값을 줘도 "성공"이라고 답한다

존재하지 않는 인풋 이름을 SetText/OverlayIn/SetLayer 등에 넘겨도 vMix REST API는 HTTP 200과 "성공" 메시지를 돌려줍니다 — 심지어 SetLayer는 그러면서 실제로는 레이어를 지워버리기까지 합니다. HTTP 응답만으로는 절대 신뢰할 수 없고, 실제 vMix 상태(XML)를 다시 읽어 대조해야만 진짜 확인이 됩니다. verifyVMixInputMap(), applySourceLayers()의 존재 검증, _vmixEnsurePaused()가 전부 이 교훈에서 나왔습니다.

패턴 02

"이 명령을 보내면 이 상태가 된다"는 가정은 실장비에서 자주 깨진다

Cut이 리스트 재생을 자체 트리거하거나, Restart가 재생을 재개하거나, Restart와 Play를 너무 붙여 보내면 재생 명령이 씹히는 등 — 문서에 없는 실제 하드웨어 동작이 반복적으로 코드의 암묵적 가정을 깼습니다. 이번 세션의 DVE SetLayer 파라미터 재발견(9절)도 같은 계열입니다: 처음 만든 코드는 문서를 확인하지 않고 그럴듯한 이름으로 추측해서 썼고, 실제로는 한 번도 동작한 적이 없었습니다.

패턴 03

안전장치 사이의 경합 — 하나를 고치면 다른 하나가 새로 드러난다

서버 클립 안전장치는 0.3초 감시·1.5초 폴링·정밀 정지 타이머·유예시간까지 겹겹이 쌓여 있는데, 새 보호막을 하나 추가할 때마다 "그 보호막이 아직 유예시간을 안 보고 있어서" 생기는 새로운 경합이 나타났습니다(07-17→07-18 사이 같은 증상이 세 번 재발). 여러 타이머/폴링이 같은 상태를 동시에 감시할 때는 새 방어막을 추가하면 기존 방어막 전부가 그 새 상태를 알아야 한다는 점을 놓치기 쉽습니다.

패턴 04

"자동이 디폴트"는 방송에서 위험하다

CG 자동 타이밍, 서버 자동 교번, 조기 전환 등 여러 기능에서 "아무것도 지정 안 하면 자동으로 나간다"는 기본값이 실제로는 "방송 중 저절로 자막이 튀어나온다"는 사고로 이어졌습니다. 지금은 명시적으로 켠 경우만 자동, 그 외엔 항상 수동 확인을 거치도록 기본값을 반전시켰습니다.

패턴 05

두 번째 화면을 만들 때 첫 번째 화면의 방어 로직을 빠뜨리기 쉽다

"대기 단계에서 재생이 시작되면 안 된다"는 방어가 armSelectedItemOnPGM() 한 곳에만 있고, 같은 문제를 겪을 수 있는 다른 세 지점(라디오 등록영상, 다음 아이템 프리뷰, 앵커 단계 사전로드)에는 빠져 있었습니다. 같은 패턴을 쓰는 곳이 여럿이면 하나를 고칠 때 "이 방어가 필요한 다른 곳이 더 있는가"를 항상 같이 점검해야 합니다.

패턴 06

목킹 테스트는 로직 버그를 잡고, 실장비 테스트는 타이밍·문서 오류를 잡는다

이 프로젝트의 버그 대부분은 Playwright + vMix 목킹으로 실제 함수 호출 순서를 검증하며 잡혔지만, "Restart와 Play를 너무 붙이면 재생이 안 됨"이나 "SetLayer 파라미터 자체가 틀림" 같은 문제는 목킹으로는 재현되지 않고 실장비에서만 드러났습니다. 둘 다 필요하고, 서로 대체할 수 없습니다.
