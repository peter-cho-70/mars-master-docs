Cursor AI가 본
다음 일정

저장소의 개발 문서, 실제 구현, 장애 기록과 테스트 자산을 서로 대조했다. 이 문서는 지금까지 만든 기능의 목록이 아니라, 다음에 무엇을 어떤 순서로 끝내야 하는지를 방송 위험도 기준으로 정리한 실행 계획이다.

Cursor AI verdict 새 기능을 더 얹기 전에, 이미 되는 방송 경로를 반복해서 증명할 수 있는 체계를 먼저 만든다.
검토 기준 2026-07-19 문서·코드 교차 검토 실장비 위험 우선 4주 실행안

지금 상태가 보내는 네 가지 신호

MARS와 Master는 이미 실제 vMix·OBS·Videohub를 움직이는 수준에 도달했다. 문제는 기능의 부족보다 같은 기능이 다음 수정 뒤에도 그대로 동작하는지 자동으로 확인하기 어렵다는 것이다.

01 · PRODUCT

프로토타입을 넘어섰다

큐시트, 전환, 서버 재생, 자막, 라우터와 LIVE 시퀀스까지 실제 운용 범위가 넓다.

02 · RISK

오류는 경계에서 생긴다

브라우저 상태↔장비 상태, MARS↔Master, 재생 타이머↔전환 타이머 사이의 경합이 핵심 위험이다.

03 · TEST

검증 이력이 테스트 자산은 아니다

실장비·Playwright 검증 기록은 풍부하지만 저장소에서 반복 실행 가능한 자동화는 Videohub 중심이다.

04 · SCALE

확장 전에 기준선을 고정해야 한다

VMU·AMU·로그인·DB를 시작하면 현재의 설정 이중화와 회귀 위험도 함께 커진다.

가장 위험한 착각 화면에 성공 토스트가 떴거나 HTTP 200을 받았다고 실제 방송 장비가 바뀐 것은 아니다. 이 프로젝트에서는 존재하지 않는 vMix 인풋과 잘못된 일부 파라미터가 조용히 무시된 전례가 반복됐다.

다음 개발 일정 — 안정화에서 확장으로

일정은 날짜를 채우는 순서가 아니라 각 단계의 통과 조건으로 움직인다. 앞 단계가 통과되지 않으면 다음 단계의 신규 기능은 착수하지 않는다.

P0. 지금 막힌 경로부터 복구

1–2일

사용자가 바로 만나는 실패와 실제 장비 설정 드리프트를 먼저 제거한다.

  • 브라우저 PDF 파서의 pdf.js 배포 파일과 OBS판 상대경로 확인
  • 실제 vMix 인풋 목록 기준으로 cam1~4, rs1~6, SRV, DVE 매핑 전수 대조
  • 긴급·위치·코너 자막의 GT 인풋/필드/오버레이 슬롯 확정
  • MARS→Master 전송에서 ANC·PD·방송시각·전체 길이 메타 보존 확인
통과 조건 — PDF 1종 정상 로드, 입력 매핑 검증 0건 실패, CG 8종 실화면 확인.

P1. 핵심 회귀 테스트를 저장소에 남기기

1주차

한 번 성공한 검증을 다음 수정 뒤에도 같은 명령으로 다시 돌릴 수 있게 만든다.

  • 실제 PDF 3종을 정답 JSON과 비교하는 골든 테스트
  • MARS→server.py→Master 데이터 왕복 E2E 테스트
  • vmixCall 목킹으로 SelectIndex→Restart→Play→전환 호출 순서 고정
  • LIVE 자동 단계와 CUSTOM 단계의 질문별 클립 선택 회귀 테스트
  • 큐시트·CG·버전·Videohub API 스모크 테스트
통과 조건 — P0/P1 자동 테스트가 한 명령으로 실행되고, 실패 시 어느 경로가 깨졌는지 바로 보인다.

P2. 실제 방송 조건에서 내구성 검증

2주차

목 테스트가 잡지 못하는 vMix 타이밍과 브라우저 수명주기 문제를 집중적으로 확인한다.

  • Cut/Mix/Fade/Wipe × TITLE/CM/ANCHOR/REPORT/LIVE 조합 리허설
  • SRV1→2→3 교번, 같은 서버 연속 사용, 짧은 클립, 멀티클립 반복
  • 디졸브 연속 20회와 60–90분 전체 큐시트 내구 테스트
  • Master·MARS·CG Studio·오버레이 동시 폴링 부하
  • 탭 백그라운드, 네트워크 단절, server.py 재시작 후 복구
통과 조건 — 잘못된 다음 소재 0회, 의도하지 않은 Pause 0회, 상태 복구 후 운영자 개입 없이 장비 상태와 재동기화.

P3. 경계를 정리하고 확장 준비

3–4주차

현재 동작을 지킨 뒤에만 구조를 단순화하고 다음 제품 단계를 결정한다.

  • vMix/OBS 공통 큐시트·타입·자막 로직을 공유 모듈로 분리
  • 타입·CG 라벨·장비 매핑을 서버의 단일 설정 원본으로 통합
  • 큐시트 저장 검증, 원자적 교체, 백업·복원과 설정 버전 도입
  • 방송 VLAN 접근 통제, API 토큰 또는 IP 허용목록 적용
  • Node+PostgreSQL과 Next.js+Supabase 중 상업화 백엔드 방향 결정
통과 조건 — 기존 P0/P1 테스트를 모두 유지한 채 공통 로직과 설정 원본이 하나로 줄어든다.

앞으로의 개발 완료 기준

새 기능과 버그 수정은 다음 다섯 항목을 모두 만족해야 완료로 이동한다.

정상 경로 — 요청한 기능이 UI에서 실제로 동작한다.
실패 경로 — 연결 끊김·설정 누락·잘못된 파일에서 가짜 성공을 표시하지 않는다.
회귀 테스트 — 같은 오류를 자동으로 다시 재현하고 수정 후 통과한다.
실제 상태 확인 — 앱 표시가 아니라 vMix XML·OBS 응답·Videohub 상태로 결과를 확인한다.
운영 복구 — 새로고침·서버 재시작·네트워크 복구 후 안전한 상태로 돌아온다.
좋은 다음 개발의 모습 기능 수가 빠르게 늘어나는 것보다, 어제 되던 방송이 오늘도 같은 테스트로 증명되는 것이 더 중요한 진척이다.

안정화 전에는 잠시 미룰 것

아래 항목은 중요하지 않아서가 아니라, 현재 경계를 고정한 뒤 시작해야 재작업이 줄어든다.

VMU·AMU 신규 연동현재 장비 제어 패턴과 실패 규칙을 테스트로 고정한 뒤 복제한다.
로그인·소셜 계정온프레미스/SaaS/하이브리드와 백엔드 스택 결정이 먼저다.
MOS·NRCS 통합현재 큐시트 스키마와 타입 레지스트리를 하나로 정리한 뒤 연결한다.
대규모 UI 개편방송 핵심 함수의 선택자와 상태 흐름을 회귀 테스트로 보호한 뒤 진행한다.
