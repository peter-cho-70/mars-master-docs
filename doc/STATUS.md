| DVE 레이어 — 사전 준비(프리암)·아이템 종료 시 초기화·화면 확인 기능 추가 (07-18) | 사용자 요청 3가지: "DVE 설정이 변화 가능하니까 사전에 미리 준비가 되어야 한다", "자막처럼 해당 아이템이 끝나면 다음 아이템에 영향을 주지 않도록", "DVE 값으로 소재가 정상적으로 화면이 나오는지 확인하는 기능". ① `prearmLiveSeqPhaseDve()` 신설 — LIVE 시퀀스가 각 단계로 실제로 컷하기 "직후" 곧바로 다음 단계의 DVE 레이어를 미리 채워 넣고(지금 온에어 중인 DVE는 절대 건드리지 않는 가드 포함, updatePvwLookaheadVMix()의 curSrvName 가드와 동일 원리) PreviewInput으로 PVW에 올려 vMix 프리뷰 모니터로 미리 확인할 수 있게 함 — `startLiveSeqVMix()`(첫 질문 진입 시)·`advanceLiveSeq()`(다음 단계 진행마다) 양쪽에 연동. ② `resetLiveSeqUI()`(아이템 전환 시 selectItem()이 항상 호출하는 지점, 자막의 clearCGAutoTimers()와 같은 자리)가 이번 아이템이 실제로 썼던 DVE들을 레이어 "기본값"으로 되돌리도록 확장 — 자막과 달리 DVE 레이어는 끄는 개념이 없어 "기본값 복귀"로 같은 효과를 냄, 어떤 질문에서 재정의(phase.layers)해둔 값이 이후 그 DVE를 재사용하는 다른 상황에 남아있지 않게 함. ③ `applySourceLayers()`가 SetLayer 성공 여부만이 아니라 그 대상 인풋이 실제 vMix 인풋 목록에 진짜 존재하는지까지 매번 대조해 로그에 ✓/⚠ 표시(vMix가 존재하지 않는 인풋을 가리켜도 에러 없이 조용히 무시하는 이 프로젝트의 기존 교훈과 동일 이유) + 설정 탭에 독립 "🔍 DVE 레이어 구성 확인" 버튼·결과창 신설(verifyVMixInputMap()과 같은 패턴) — 이름·연결 존재까지만 자동 확인 가능하고, 실제 화면 배치(위치/크기)는 "지금 실제로 적용" 버튼으로 vMix 프리뷰 모니터에서 직접 봐야 한다고 명확히 안내 | Playwright(vmixCall/vmixGetState를 실제 인풋 목록이 있는 것처럼 가로채 목킹) — 온에어 중인 DVE는 프리암이 건드리지 않는 것 확인, 존재하지 않는 인풋(REAL_SRV1 제거)을 기본값으로 등록해뒀을 때 검증이 정확히 그 레이어만 ✗로 잡아내는 것 확인, 아이템 전환 시 실제 SetLayer 호출로 재정의값(SRV2)이 아니라 기본값(SRV1)으로 되돌아가는 것 확인, 설정 탭 확인 버튼의 실제 렌더링(전부 초록 ✓) 스크린샷으로 확인 |

| DVE `SetLayer` 파라미터 실장비 검증 — 처음 구현이 완전히 틀렸던 것 발견·수정 (07-18) | 사용자 요청으로 실제 vMix(192.0.2.2, v29.0.0.48, 방송 중인 라이브 시스템)에 직접 연결해 확인. **07-18 앞선 구현은 실제로는 전혀 동작하지 않는 파라미터였다** — `Input=<합성>&SelectedIndex=<idx>&Value=<대상>`은 vMix가 "Invalid Parameters"로 거부(HTTP 500, 함수 이름 자체는 인식하지만 파라미터가 틀림). vMix 공식 함수 레퍼런스(vmix.com/help29/ShortcutFunctionReference.html)를 직접 조회해 진짜 형식을 확인: `Function=SetLayer&Input=<합성>&Value=<레이어번호>,<새 소스>` — SelectedIndex 없이 Value 하나에 콤마로 합쳐 보낸다. 추가로 실측 전용으로만 드러난 두 가지: ① **레이어 번호가 1부터 시작**(Value="1,X"가 실제로는 XML의 `<overlay index="0">`를 바꿈, 이 프로젝트의 0부터 세는 데이터 모델과 어긋남) — `_vmixSetLayer()` 안에서만 +1 변환해 데이터 모델/UI는 그대로 0부터 유지. ② **존재하지 않는 대상을 가리키면 "성공"이라고 응답하면서 그 레이어를 통째로 지워버림**(XML에서 `<overlay index="N">` 자체가 사라짐 — 단순 무시가 아니라 파괴적 실패, 유효한 SetLayer를 같은 번호로 다시 걸면 복구는 됨을 확인). 이 발견에 따라 `applySourceLayers()`를 "먼저 보내고 나중에 확인"에서 "먼저 실제 인풋 목록과 대조해 존재할 때만 보낸다"로 방어 순서를 바꿈 — 검증 실패한 레이어는 SetLayer 자체를 호출하지 않아 직전 내용이 그대로 유지됨. 부수 검증: 제목(title) 기반 주소 지정도 정상 동작, `PreviewInput`을 레이어 구성 합성 인풋에 걸어도 PGM(active)엔 전혀 영향 없음(프리암 기능의 전제 확인) | **실제 vMix 라이브 시스템 직접 검증** — curl로 SetLayer 호출 전/후 XML 상태를 매번 재조회해 실제 변경 여부 확인(HTTP 응답 텍스트만으로는 신뢰 불가 재확인), 정상 케이스·범위 밖 레이어 번호·존재하지 않는 대상 3가지 모두 테스트, 테스트 중 실수로 다른 소스로 잘못 복원했던 것도 원래 key와 대조해 즉시 발견·재수정, 모든 테스트가 끝난 뒤 PGM(active=18)은 처음부터 끝까지 한 번도 안 바뀐 것과 PVW(preview)가 정확히 원래 값(22)으로 복구된 것까지 최종 확인 — **실제 방송에 영향 없이 완료** |
